# KodaX Space Coding Agent 新手教程：软件与微服务开发最佳实践

适用版本：以实际交付安装包为准。本文基于 KodaX Space 0.1.29 工作树和 KodaX 0.7.63 生态说明整理。

适用对象：第一次使用 Coding Agent 的开发者、测试工程师、技术负责人、代码相关知识工作者，以及需要把 KodaX Space 推广给团队成员的客户。

本文目标：让新手在 30 分钟内完成第一次有效协作，在 1-2 天内形成安全、可复用的软件研发工作习惯。

本文按简体中文界面书写。部分能力名在产品中仍保留英文，例如 `Task Dock`、`Artifact`、`MCP`、`AGENTS.md`、slash 命令等；这类名称会保留原文，并在第一次出现时补充中文解释。

## 1. 先建立正确心智

KodaX Space 不是传统聊天机器人，也不是 IDE 的自动补全插件。它更像一个围绕本机项目工作的 AI 开发协作者：可以阅读项目、理解目录结构、提出修改计划、调用工具、生成 diff、运行命令、产出文档或 Artifact，并在关键操作前通过权限模式和确认弹窗让你把关。

初学者最容易踩的坑，是把 Coding Agent 当成“问答框”或“自动写代码机器”。更好的理解方式是：

| 你要管理的对象 | 在 KodaX Space 中对应什么 | 新手要记住什么                                           |
| -------------- | ------------------------- | -------------------------------------------------------- |
| 工作边界       | 项目                      | 先打开正确的项目根目录，避免让 AI 在错误目录工作。       |
| 连续上下文     | 会话                      | 一个任务一个会话，复杂任务不要混在同一段对话里。         |
| 输入任务       | 底部输入框                | 写清目标、范围、约束和验收标准。                         |
| 安全控制       | 权限模式                  | 日常用 `自动接受编辑`，不确定时用 `仅规划`。             |
| 过程可视化     | Task Dock / 弹出面板      | 看计划、变更、审阅/Diff、任务和 Artifact，不只看对话流。 |
| 可交付成果     | 代码变更或 Artifact       | 代码进仓库，报告/文档/HTML 等可先作为 Artifact。         |
| 项目规则       | `AGENTS.md` / 技能        | 把团队约定写成规则，减少每次重复说明。                   |

一句话原则：

> 让 Agent 先理解项目，再给计划；确认计划后小步修改；每次修改都看 diff、跑测试、写总结。

## 2. 中文界面速查

下面这些名称会在教程里反复出现。

| 中文界面或本文说法 | 说明                                                                    |
| ------------------ | ----------------------------------------------------------------------- |
| `设置` -> `偏好`   | 设置语言、默认工作区、界面行为和工作流行为。                            |
| `设置` -> `服务商` | 配置模型服务商、API Key、默认服务商和自定义端点。                       |
| `项目`             | 本地工作目录，也是 AI 读写文件的边界。                                  |
| `会话`             | 围绕某个项目的一段连续对话和任务上下文。                                |
| `底部输入框`       | 输入任务、粘贴图片、引用文件、输入 `/` 命令或 `@` 路径的位置。          |
| `权限模式`         | `仅规划`、`自动接受编辑`、`全自动` 三种常用模式。                       |
| `Task Dock`        | 右侧任务状态区，显示运行、计划、工作流、变更、来源、Artifact 和上下文。 |
| `审阅/Diff`        | 查看 AI 修改和 git 工作区差异。                                         |
| `预览`             | 预览 PDF、DOCX、XLSX、代码和普通文本等文件。                            |
| `终端`             | 项目目录下的真实终端。                                                  |
| `Artifact`         | AI 生成的独立产物，例如报告、文档、HTML、图表等。                       |

## 3. 5 分钟启动检查

第一次打开 KodaX Space 后，按下面顺序检查。

1. 打开项目目录

如果界面还不是中文，先进入 `设置` -> `偏好` -> `语言`，选择 `简体中文`。然后选择真实项目根目录，例如包含 `package.json`、`pom.xml`、`go.mod`、`pyproject.toml`、`.git` 或主要源码目录的目录。不要选择过大的父目录，例如整个磁盘、Downloads 或多个无关项目的上层目录。

2. 配置模型服务商

进入 `设置` -> `服务商`，配置可用的模型服务，填写 API Key，并点击 `测试` 验证连接。KodaX Space 支持内置服务商，也支持自定义 OpenAI-compatible / Anthropic-compatible 端点。企业客户如果通过内部网关访问模型，建议由管理员先统一提供 Base URL、协议和模型名。

3. 设置权限模式

新手建议默认使用：

- `仅规划`：只想让 AI 分析、解释、给方案。
- `自动接受编辑`：日常推荐。AI 可以提出写文件或运行命令，但关键操作仍由你确认。
- `全自动`：只在团队规则、测试和回滚方式都明确时使用。

如果使用 slash 命令，三种模式分别对应 `/mode plan`、`/mode accept-edits`、`/mode auto`。

4. 认识三个主要观察面板

- 中央对话视图：看对话和工具调用。
- 右侧 `Task Dock`：看当前运行状态、计划、变更、上下文和产物。
- `审阅/Diff`、`预览`、`终端`、`Artifact` 等弹出面板：分别用于审查代码变更、预览文件、运行命令和查看生成成果。

5. 发送第一个只读任务

```text
请先只读分析这个项目。阅读 README、package.json/构建配置和主要目录，给我一份：
1. 项目用途
2. 技术栈
3. 启动方式
4. 测试方式
5. 你建议我下一步查看的 5 个文件
请不要修改任何文件。
```

如果 AI 能正确读到项目文件、说明启动方式，并在 `Task Dock` 中显示上下文或工具调用，就说明基础链路可用。

## 4. 前 30 分钟练习路径

建议新手不要一上来就让 Agent “重构系统”或“实现完整模块”。先做三个练习。

### 练习一：项目导览

目标：让 AI 变成项目讲解员。

```text
请帮我做一次项目导览。只读分析，不修改文件。

请输出：
- 主要模块和目录职责
- 应用入口和启动链路
- 配置文件和环境变量入口
- 测试/构建/打包命令
- 你认为最需要补文档或补测试的位置

如果信息不确定，请标注“不确定”并说明还需要看哪个文件。
```

验收标准：

- AI 能引用具体文件或目录。
- 能区分确定事实和推测。
- 没有开始修改文件。

### 练习二：只读代码审查

目标：学习如何让 AI 找风险，而不是直接改代码。

```text
请审查当前 git working tree 的改动，先不要修改文件。

重点看：
- 行为回归
- 类型/空值/边界条件
- 缺失测试
- 安全或权限风险
- 是否有无关改动

请按严重程度排序，并给出文件路径、原因和建议修复方式。
```

验收标准：

- 结论有文件路径和理由。
- 能区分高风险和低风险。
- 不把格式建议当成主要问题。

### 练习三：小修复闭环

目标：让 AI 完成一次最小可验证改动。

```text
请修复一个你能确认的问题。要求：
1. 先说明你要改哪些文件和为什么。
2. 每次只处理一个问题，不做顺手重构。
3. 修改后运行最小相关测试或类型检查。
4. 最后给出变更摘要、测试结果和仍需人工确认的点。
```

验收标准：

- 修改范围小。
- diff 可以读懂。
- 有测试或解释为什么无法测试。
- 最终回答能说明做了什么、如何验证。

完成这三个练习后，用户通常就能理解 Coding Agent 的基本协作节奏。

## 5. 初学者标准工作流：六步法

新手使用 KodaX Space 时，最稳的方式不是“让 AI 一次改完”，而是固定走六步。

1. 让 Agent 先读项目

```text
请先只读不改。请阅读当前项目，输出：
1. 项目技术栈和启动方式
2. 主要目录职责
3. 服务入口、路由入口、配置入口、测试入口
4. 如果这是微服务项目，请列出每个服务的职责、端口、依赖和数据存储
5. 你不确定的地方请明确标出来，不要猜
```

2. 让 Agent 先出计划

```text
我要新增一个“查询订单详情”的接口。请先不要改代码。
请先分析：
1. 需要改哪些服务、哪些文件？
2. 是否影响 API 契约、数据库、缓存、消息队列？
3. 需要新增哪些测试？
4. 回滚或兼容性风险是什么？
最后给出一个小步修改计划，等我确认后再开始。
```

3. 小步修改，不要大包大揽

```text
按照刚才的计划，先只完成第一步：新增请求/响应 DTO 和路由占位。
要求：
1. 不改数据库
2. 不改其他服务
3. 补充最小单元测试
4. 修改后告诉我具体改了哪些文件
```

4. 审查 diff

重点看这六项：

| 检查项 | 要问自己什么                               |
| ------ | ------------------------------------------ |
| 范围   | 是否只改了约定范围内的文件？               |
| 行为   | 有没有引入额外业务逻辑？                   |
| 契约   | API、事件、数据库字段是否兼容？            |
| 安全   | 有没有泄露 token、打印敏感日志、放宽权限？ |
| 测试   | 是否有覆盖核心路径和异常路径？             |
| 可回滚 | 出问题时能否快速撤回？                     |

5. 让 Agent 跑验证，但命令要看清楚

```text
请先列出你建议运行的验证命令，并说明每条命令验证什么。
不要直接运行，等我确认。
```

确认后再继续：

```text
可以运行单元测试和类型检查。不要启动会连接生产环境或外部写入系统的命令。
```

6. 让 Agent 写交付说明

```text
请根据本次修改生成 PR 描述：
1. 背景
2. 主要改动
3. 测试结果
4. 风险与回滚方案
5. 需要 reviewer 重点看的地方
```

## 6. 高质量任务描述模板

新手使用 Coding Agent，效果差异往往不在模型，而在任务描述。推荐用下面的任务卡模板。

```text
背景：
<项目背景、相关模块、为什么要做>

目标：
<希望完成的具体结果>

范围：
<允许查看/修改哪些目录或文件>

约束：
<不要做什么、兼容性、安全、性能、风格要求>

验收：
<怎么判断完成，例如测试命令、页面行为、API 返回、文档结构>

工作方式：
<先计划/先审查/可以直接改/需要每步确认>

交付：
<代码改动、Artifact、说明文档、测试报告等>
```

示例：

```text
背景：
这个项目是一个 Electron + React 桌面应用。最近用户反馈设置窗口里“服务商测试”失败时提示不够清楚。

目标：
改进服务商测试失败时的错误提示，让用户知道是 API Key、Base URL、协议还是网络问题。

范围：
优先查看 `apps/desktop/renderer/src/features/provider` 和 `apps/desktop/electron/providers` 相关文件。

约束：
不要改服务商配置格式，不要引入新依赖，不要影响成功路径。

验收：
补充或更新相关单元测试；如果没有合适测试，请说明原因并给出人工验证步骤。

工作方式：
先给计划，等我确认后再改。
```

## 7. 日常软件研发场景最佳实践

### 7.1 读懂陌生项目

适用场景：接手新项目、客户交付前熟悉代码、定位架构边界。

推荐流程：

1. 使用 `仅规划` 模式。
2. 让 AI 先读 README、构建配置、入口文件和目录结构。
3. 要求输出“事实 / 推测 / 待确认”三栏。
4. 再让 AI 深挖一个模块，而不是一次读完整个仓库。

提示词：

```text
请只读分析这个仓库，先不要修改文件。请分三层输出：
1. 已确认事实：必须来自文件内容。
2. 合理推测：说明推测依据。
3. 待确认问题：列出还需要我或代码继续确认的点。

最后给出“如果我要修一个 bug，应该先理解哪些模块”的学习路径。
```

### 7.2 修复 Bug

适用场景：有错误日志、复现步骤、测试失败、用户反馈。

推荐流程：

1. 先贴错误日志、复现步骤和期望行为。
2. 要求 AI 定位根因，不要直接改。
3. 确认根因后，再让它做最小修复。
4. 运行相关测试。
5. 审查 diff。

提示词：

```text
下面是一个 bug。请先定位原因，不要修改文件。

现象：
<用户看到什么>

复现步骤：
<1, 2, 3>

期望行为：
<应该发生什么>

实际行为/日志：
<错误日志或截图描述>

请输出：
- 最可能根因
- 需要查看的文件
- 修复方案
- 最小验证方式
```

确认后继续：

```text
请按刚才方案做最小修复。不要做无关重构。修改后运行相关测试，并给出 diff 摘要。
```

### 7.3 实现小功能

适用场景：新增按钮、配置项、API 参数、表单校验、小型业务逻辑。

推荐流程：

1. 让 AI 先查现有实现风格。
2. 让它给出文件级计划。
3. 每次只实现一个用户可感知功能。
4. 要求补测试或人工验证步骤。

提示词：

```text
我想新增一个小功能：<功能描述>。

请先阅读相关代码，找出现有模式，然后给出实现计划。计划中请说明：
- 会改哪些文件
- 每个文件为什么要改
- 是否需要测试
- 有哪些兼容性风险

先不要修改文件。
```

计划确认后：

```text
按计划实现。请保持改动最小，沿用现有组件/状态管理/IPC/测试风格。
```

### 7.4 代码审查

适用场景：PR 自查、客户交付前复核、团队合并前把关。

推荐流程：

1. 使用 `仅规划` 或 `自动接受编辑`。
2. 明确审查范围：当前改动、某个分支 diff、某个模块。
3. 要求“发现优先”，不要先总结。
4. 要求按严重程度排序。

提示词：

```text
请以代码审查方式检查当前改动。先不要修改文件。

请重点找：
- 可能导致运行错误或数据错误的问题
- 边界条件遗漏
- 并发/异步/生命周期问题
- 安全、权限、敏感信息风险
- 缺失测试

输出格式：
- 严重程度
- 文件路径
- 问题原因
- 建议修复

如果没有发现问题，请明确说没有发现，并说明剩余风险。
```

### 7.5 写测试

适用场景：修复后补测试、遗留模块补覆盖、复现回归。

推荐流程：

1. 让 AI 先找同类测试风格。
2. 明确要覆盖的行为，而不是说“多写点测试”。
3. 先写失败用例，再实现或修复。
4. 运行最小测试命令。

提示词：

```text
请为 <模块/函数/行为> 补测试。先阅读现有测试风格，不要引入新测试框架。

必须覆盖：
- <场景 1>
- <场景 2>
- <边界条件>

请修改最少文件，并运行对应测试。
```

### 7.6 重构

适用场景：消除重复、拆分复杂函数、迁移 API。

重构最容易失控。新手必须给边界：

- 不改变外部行为。
- 不跨多个无关模块。
- 先建立测试或至少跑现有测试。
- 每轮只做一个重构目标。

提示词：

```text
请评估 <模块/文件> 是否适合重构。先不要修改文件。

请输出：
- 当前复杂点
- 哪些重复或边界可以安全收敛
- 哪些地方不建议现在动
- 一个最小重构方案
- 需要哪些测试保护
```

确认后：

```text
请只执行最小重构方案，保持外部行为不变。不要顺手改命名、格式或无关逻辑。
```

### 7.7 生成文档、PR 描述和变更说明

适用场景：写 README、接口文档、发布说明、客户交付说明、PR 描述。

推荐流程：

1. 让 AI 读取代码和 diff。
2. 明确文档读者。
3. 要求区分“怎么用”和“内部实现”。
4. 作为 Artifact 先看一版，再决定是否写入仓库。

提示词：

```text
请根据当前改动生成一份 PR 描述。读者是项目维护者。

请包含：
- 背景
- 主要变更
- 测试结果
- 风险与回滚
- 需要 reviewer 重点看的地方

请不要修改文件，先作为 Markdown Artifact 输出。
```

## 8. KodaX Space 中建议优先掌握的功能

### 8.1 权限模式

新手默认使用 `自动接受编辑`。当任务还不清楚时切到 `仅规划`。只有在以下条件都满足时再考虑 `全自动`：

- 当前项目规则明确。
- git 工作区干净或已知道当前改动归属。
- 有可靠测试或回滚方式。
- 任务范围足够小。

遇到权限弹窗时，不要机械点击允许。先看：

- 工具要做什么。
- 作用目录是否正确。
- 是否会写文件、运行命令、联网或访问扩展。
- 命令是否包含删除、移动、发布、部署、上传、密钥相关操作。

### 8.2 Task Dock 和审阅/Diff 面板

不要只看 AI 最后的自然语言回答。日常要养成三步检查：

1. 看计划：它准备怎么做。
2. 看变更/审阅-Diff：它实际改了什么。
3. 看测试结果：改动是否被验证。

如果 diff 里出现无关格式化、跨模块大改、删除大量代码，应该立即要求停止并解释原因。

### 8.3 快速提问

`快速提问` 适合临时提问，例如：

- “这个错误是什么意思？”
- “这段日志可能来自哪里？”
- “帮我解释这个函数。”
- “这个 API 设计有没有明显问题？”

`快速提问` 当前使用临时的仅规划会话。回答有价值时，可以选择 `Continue in Coder`，把它提升到正常 Coder 会话继续做事。

### 8.4 Artifact

Artifact 适合让 AI 先产出独立成果，而不是直接写入项目：

- 调研报告
- 架构图说明
- PR 描述
- 测试计划
- API 文档
- HTML Demo
- 表格或图表

最佳实践：客户交付类内容先做 Artifact，人工确认后再落盘。

### 8.5 工作流

工作流适合多步骤、可复跑的任务，例如：

- 大型代码审查
- 生成并验证发布说明
- 分析多个模块后汇总报告
- 先调查再分阶段实现

新手不需要第一天就使用工作流。等熟悉单会话工作方式后，再把重复任务保存成工作流。

### 8.6 MCP、技能、AGENTS.md

当团队开始多人使用 KodaX Space，建议尽快沉淀三类上下文：

- `AGENTS.md`：项目级规则，例如编码风格、测试命令、禁止事项、目录边界。
- 技能：可复用任务方法，例如“生成发布说明”“安全审查”“接口文档生成”。
- MCP：连接外部工具或数据源，例如 GitHub、内部文档、数据库只读查询等。

新手阶段先用 `AGENTS.md` 就足够。不要一开始就安装大量 MCP 或第三方扩展。

## 9. 团队级使用规范

建议客户团队采用下面的轻量规则。

### 9.1 每个任务开始前

- 确认 git 分支和工作区状态。
- 打开正确项目根目录。
- 选择合适权限模式。
- 给出任务范围和验收标准。
- 如果要改代码，先让 AI 给计划。

### 9.2 每个任务执行中

- 大任务拆小。
- 不同时让 AI 修 bug、重构、升级依赖、改样式。
- 权限弹窗逐项确认。
- 发现方向偏了，立即要求停止并重新计划。
- 重要决策让 AI 写出依据。

### 9.3 每个任务结束前

- 查看 diff。
- 运行测试、类型检查或最小人工验证。
- 要求 AI 给变更摘要。
- 要求 AI 说明未验证风险。
- 由人决定是否 commit、push、发 PR 或交付。

### 9.4 推荐写入项目的 `AGENTS.md`

可以在项目根目录放一份简短的 `AGENTS.md`：

```markdown
# 项目协作规则

- 默认使用中文回答，代码和命令保持原文。
- 修改代码前先阅读相关测试和现有实现风格。
- 不要做无关格式化或顺手重构。
- 不要修改锁文件，除非任务明确要求依赖变更。
- 修改后优先运行最小相关测试。
- 涉及数据库、部署、删除文件、密钥、生产环境的操作必须先询问。
- 最终回答必须包含：变更摘要、验证结果、剩余风险。
```

## 10. 常见反模式

| 反模式                         | 为什么不好                   | 更好的做法                       |
| ------------------------------ | ---------------------------- | -------------------------------- |
| “帮我把这个项目优化一下”       | 范围无限，容易乱改           | 指定模块、目标和验收指标。       |
| “直接改，越快越好”             | 容易跳过理解和测试           | 小任务可直接改，大任务先计划。   |
| 一次要求修 bug + 重构 + 加功能 | 难以审查和回滚               | 拆成多个 session 或多个阶段。    |
| 不看 diff 直接接受             | 人失去最终责任               | 每次改动都看变更/审阅-Diff。     |
| 把密钥、客户数据直接贴给模型   | 数据可能进入第三方模型服务商 | 使用脱敏样例，遵守团队数据策略。 |
| 在全自动模式下跑未知命令       | 风险不可控                   | 新项目先用仅规划/自动接受编辑。  |
| 让 AI 自己决定发布/部署        | 高风险操作                   | 部署、发布、迁移必须人工确认。   |

## 11. 新手提示词速查

### 项目理解

```text
请只读分析这个项目，输出模块结构、启动方式、测试方式和主要风险。不要修改文件。
```

### 当前改动审查

```text
请审查当前 git 改动，按严重程度列出 bug、回归风险和缺失测试。先不要修改文件。
```

### 小修复

```text
请修复 <问题>。先给文件级计划，确认后再改。改完运行最小相关测试。
```

### 测试补齐

```text
请为 <行为> 补测试，沿用现有测试风格，不引入新框架。运行相关测试并汇报结果。
```

### 文档生成

```text
请根据 README、配置文件和源码生成一份面向新开发者的快速上手文档，先作为 Artifact 输出。
```

### 问题定位

```text
根据以下日志和复现步骤定位原因。请先分析，不要修改文件。输出最可能根因、证据和验证方式。
```

### 让 AI 自查

```text
请复查你刚才的改动：有没有无关变更、遗漏测试、边界条件或安全风险？如果有，请先列出来，不要继续修改。
```

## 12. 安全、隐私与责任边界

KodaX Space 的默认理念是本地优先和显式授权，但用户仍然需要理解责任边界。

必须人工把关的操作：

- 删除、移动、覆盖大量文件。
- 数据库迁移、生产数据修复。
- 发布、部署、打 tag、推送远端分支。
- 修改认证、权限、支付、加密、审计相关代码。
- 访问客户敏感数据、密钥、Token、证书。
- 安装或启用第三方 MCP、扩展、脚本。

建议团队规则：

- 提供给模型的日志和数据先脱敏。
- 客户代码按合同和数据分级选择模型服务商。
- 生产环境操作不要交给 Agent 自动执行。
- AI 产出必须经过人类 review。
- 重大改动通过 PR 和 CI 闭环。

## 13. 故障排查

| 问题               | 优先检查                                                    |
| ------------------ | ----------------------------------------------------------- |
| AI 不能读取项目    | 是否已打开正确项目，路径是否在当前工作区内。                |
| 模型服务商无法使用 | API Key、Base URL、协议、模型名、网络代理。                 |
| 修改前一直弹确认   | 当前是安全设计；日常用 `自动接受编辑`，必要时配置允许规则。 |
| AI 输出太泛        | 任务缺少范围、验收标准或上下文文件。                        |
| AI 改动太大        | 要求停止，缩小范围，先仅规划，再分阶段执行。                |
| MCP 工具不可见     | 打开 MCP 面板刷新配置，查看 server 状态和 diagnostics。     |
| 结果不可信         | 要求引用文件证据、运行测试、区分事实和推测。                |

## 14. 给客户团队的 1 天上手安排

建议用于内部培训或客户交付。

上午 30 分钟：基础演示

- 打开项目。
- 配置模型服务商。
- 解释项目、会话、权限模式、Task Dock、审阅/Diff。
- 完成“项目导览”练习。

上午 60 分钟：只读审查

- 让每个成员选一个熟悉项目。
- 运行当前改动审查。
- 比较 AI 发现和人工发现的差异。
- 总结哪些提示词更有效。

下午 90 分钟：小修复闭环

- 每人选择一个低风险 bug 或文档问题。
- 先计划，再改动，再测试。
- 查看 diff，形成 PR 描述。

下午 30 分钟：团队规则沉淀

- 编写或更新 `AGENTS.md`。
- 约定默认权限模式。
- 约定不能交给 AI 自动做的操作。
- 约定 PR 中如何标注 AI 辅助内容。

## 附录 A：微服务等复杂开发场景快速上手

微服务、多仓库、多语言项目比单体项目更容易让 Coding Agent 迷路。核心原则是：先建立服务地图，再限定单次任务的服务边界。

### A.1 微服务项目的第一轮提问

```text
请只读分析这个微服务项目，先建立服务地图，不要修改文件。

请输出：
1. 服务列表：服务名、目录、语言/框架、启动命令。
2. 服务间关系：HTTP/gRPC/消息队列/数据库/缓存/定时任务。
3. 关键配置：端口、环境变量、配置文件、Docker/Kubernetes/Compose 文件。
4. 本地启动路径：最小可运行依赖是什么。
5. 测试策略：单元测试、契约测试、集成测试在哪里。
6. 不确定项：哪些关系需要我补充或进一步确认。
```

这一步的目标不是让 AI 立刻写代码，而是让它把系统边界画清楚。

### A.2 建议建立的服务地图

让 AI 输出类似下面的表：

| 服务          | 路径             | 技术栈      | 依赖                | 对外接口       | 本地启动            | 测试命令         |
| ------------- | ---------------- | ----------- | ------------------- | -------------- | ------------------- | ---------------- |
| user-service  | `services/user`  | Spring Boot | PostgreSQL, Redis   | REST `/users`  | `./gradlew bootRun` | `./gradlew test` |
| order-service | `services/order` | Node.js     | user-service, Kafka | REST `/orders` | `npm run dev`       | `npm test`       |
| gateway       | `apps/gateway`   | Go          | user/order          | HTTP proxy     | `go run ./cmd`      | `go test ./...`  |

如果项目是多仓库，建议一个仓库一个项目；跨仓任务先让 AI 只做分析，真正改代码时一次只打开并修改一个仓库。

### A.3 微服务 Bug 定位模板

```text
这是一个跨服务 bug。请先定位，不要修改文件。

现象：
<用户看到的问题>

请求链路：
<入口服务 -> 中间服务 -> 下游服务>

相关日志：
<按服务分段贴日志，敏感信息已脱敏>

已知环境：
<本地/测试/生产，版本号，配置差异>

请输出：
- 可能故障点排序
- 每个故障点需要查看的服务和文件
- 如何用最小命令复现或验证
- 是否涉及接口契约、数据迁移或异步消息
```

### A.4 改动微服务时的安全顺序

推荐顺序：

1. 只读分析调用链。
2. 确认是单服务问题还是契约问题。
3. 如果是单服务问题，只改该服务。
4. 如果是契约问题，先更新契约或接口文档，再分服务改。
5. 优先补单元测试和契约测试。
6. 最后再跑集成测试或 docker compose。

不要让 AI 一次性修改 3 个以上服务。除非任务已被拆分成明确步骤，并且每一步都有测试。

### A.5 API 契约变更模板

```text
我需要修改一个跨服务 API 契约。请先分析影响，不要修改文件。

变更目标：
<新增字段/删除字段/改变语义/错误码调整>

涉及服务：
<调用方和被调用方>

兼容要求：
<是否需要向后兼容，是否灰度，是否双写/双读>

请输出：
- 现有契约位置
- 调用方列表
- 兼容性风险
- 分阶段改造计划
- 需要补的测试
```

确认后再让 AI 实施第一阶段，例如只新增可选字段，不删除旧字段。

### A.6 数据库迁移场景

数据库迁移必须更保守。建议提示词：

```text
请审查这个数据库迁移需求，先不要修改文件。

目标：
<迁移目的>

约束：
- 必须向后兼容
- 不能阻塞线上读写
- 需要回滚方案
- 需要说明数据修复脚本是否可重复执行

请输出：
- schema 变更方案
- 应用代码兼容方案
- 回滚方案
- 验证 SQL 或测试建议
- 高风险点
```

不要让 AI 在全自动模式下直接执行生产数据库命令。迁移脚本和数据修复脚本必须人工审查。

### A.7 消息队列和异步任务

异步系统常见问题是“代码看起来没错，但消息语义错了”。要求 AI 检查：

- Topic / queue 名称。
- producer 和 consumer 的序列化格式。
- 幂等键。
- 重试和死信队列。
- 消息版本兼容。
- 乱序、重复、延迟的处理。

提示词：

```text
请只读审查这条异步消息链路。重点检查 producer/consumer 契约、幂等、重试、死信、版本兼容和观测日志。不要修改文件。
```

### A.8 Docker、Kubernetes、CI 场景

适合让 AI 做：

- 解释 Dockerfile 和 compose 依赖。
- 找 CI 失败日志的根因。
- 生成本地复现命令。
- 检查 Helm/Kubernetes 配置是否与应用端口、健康检查一致。

不建议让 AI 自动做：

- 推送镜像。
- 修改生产 namespace。
- 执行 `kubectl apply` 到真实集群。
- 改 CI/CD 密钥和部署权限。

提示词：

```text
请根据以下 CI 日志定位失败原因。先不要修改文件。
请区分：
- 代码问题
- 测试环境问题
- 缓存/依赖问题
- Docker/Kubernetes 配置问题

请给出本地最小复现命令。
```

### A.9 微服务场景推荐 `AGENTS.md`

```markdown
# 微服务项目 Agent 规则

- 先建立服务地图，再修改代码。
- 跨服务改动必须先输出影响分析。
- 一次任务最多修改一个主要服务，除非用户明确批准跨服务计划。
- API 契约变更必须保持向后兼容，除非用户明确说明可以破坏兼容。
- 数据库迁移必须包含回滚和验证方案。
- 消息队列改动必须检查幂等、重试、死信和版本兼容。
- 不允许自动执行生产部署、生产数据库、真实集群 `kubectl apply`。
- 修改后优先运行对应服务的最小测试，再建议集成测试。
```

### A.10 微服务任务拆分示例

大任务：

```text
给订单系统增加优惠券功能。
```

不要直接这样交给 AI。应该拆成：

1. 只读分析订单、支付、用户、营销相关服务边界。
2. 输出数据模型和 API 契约方案。
3. 在单个服务中增加领域模型和单元测试。
4. 增加接口或消息契约。
5. 修改调用方。
6. 补契约测试和集成测试。
7. 生成发布和回滚说明。

每一步都可以是一个独立 session 或一个 session 中的独立阶段。

## 附录 B：客户交付时可直接发送的简短说明

如果只想给新用户一段非常短的入门说明，可以发送下面这段：

```text
使用 KodaX Space 时，请先打开项目根目录，配置模型服务商，然后默认用“自动接受编辑”模式。

第一次不要直接让 AI 大改代码。先让它“只读分析项目”，再让它“只读审查当前改动”，最后选择一个小问题让它“先给计划，确认后修改，修改后运行测试”。

```
