# v0.1.11 — Partner 地基（Surface 抽象 + doc-workspace 骨架）

> 2026-06-08 Planning。Partner 批次第一版，对应 [ADR-007](../ADR/ADR-007-partner-surface-model.md) 的「三件套」之**地基层**：把 Surface 从隐式单态变成一等抽象，立起 Partner 的 doc-workspace 空壳与工具/作用域边界。**全部内核无关**——不依赖任何待交付的 SDK 入口，纯 Space 侧工程。

## 版本目标

让 Space 从"只有 Coder 一个面"升级为"可在 Coder / Partner 两个 surface 间切换"，Partner 侧先把布局骨架、工具白名单、文件作用域三件事立住（暂用内置 coding 画像跑，画像替换留 F053）。Definition of Done：用户能点 `[Partner]` tab（或拖一个文档触发隐式入口）进入 doc-workspace 三栏空壳，选一个非 git 目录作用域，发一条消息，agent 在 `non-bash-subset` 工具集下作答。

## 本版本范围

| Feature | 描述 | 关键文件 |
|--|--|--|
| F045 | Surface 抽象落地 + `[Coder][Partner]` tab + 隐式入口判定 | `store/surface.ts`(新) / `shell/SurfaceTabs.tsx`(新) / `App.tsx` |
| F046 | Partner doc-workspace 三栏布局骨架 | `features/partner/`(新目录) |
| F047 | Partner 工具白名单（non-bash-subset）+ 非 git 文件作用域 | `electron/kodax/` 工具裁剪 + `electron/projects/` 作用域 |

依赖关系：F045 是 F046/F047 的前置（先有 surface 概念，才有 Partner 布局与工具裁剪）。F046、F047 可并行。

## FEATURE_045: Surface 抽象落地 + `[Coder][Partner]` tab + 隐式入口判定

> **实现说明（2026-06-13，与本节 6-08 规划有偏差，以落地为准）**：
> - **砍掉隐式入口**（`inferSurface` / `surface-inference.ts` / 拖文件自动切面）——对标 Claude Desktop 应明确手动切换，"猜意图"违背极简哲学（[[minimal_intelligent_philosophy]]）。tab 是唯一入口。
> - **session 分面不走 renderer surface 字段，改走 KodaX SDK session tag**（SDK 0.7.49 交付）：写盘 `session.tag = surface`，list 侧 mapper `tag==='partner'?'partner':'code'` 反推 + main 端 filter（**不把 tag 下推 SDK**，避开 ②B all-fetch 回退坑）。Coder/Partner 会话列表彼此独立。
> - per-surface 当前 session + 持久化"停留面"在 **F046** 落地。

### 需求概述

当前 renderer 是隐式单 surface（处处假设 Coder）。[HLD §9.4](../HLD.md#94-surface-抽象) 定义了 `Surface = 'code' | 'partner'` 与 `SurfaceSpec`，但从未真实落地。本 feature 把 Surface 变成一等运行时状态：

1. 顶部出现 `[Coder] [Partner]` tab 切换器（对齐 [ADR-004](../ADR/ADR-004-panel-model.md) 的常驻锚点；Partner 不再灰态，因为本版起有真实空壳）。
2. **隐式入口**（[极简且智能](../../README.md)）：用户拖一个 PDF/docx 进窗口、或在「新 session」时选了一个非 git 目录 → 自动判定为知识工作场景并切到 Partner surface，无需用户先理解 Coder/Partner 概念分野。tab 仅作锚点与显式覆盖。

### 影响范围

- **renderer store**：新增 `store/surface.ts`——`currentSurface: Surface`、`setSurface()`、`SURFACES` spec 表（搬 HLD §9.4 的形态）。
- **renderer shell**：新增 `shell/SurfaceTabs.tsx`；`App.tsx` 顶部挂载；按 `currentSurface` 在 Coder 布局 vs Partner 布局（F046）间路由。
- **隐式判定**：新增 `store/surface-inference.ts`——纯函数 `inferSurface({ droppedFiles?, projectRoot?, isGitRepo? }): Surface | null`。
- **session 绑定**：session 记录加 `surface` 字段（renderer 侧 store；持久化复用 KodaX session metadata，不新增落盘目录）。

### 技术方案

```ts
// store/surface.ts
export type Surface = 'code' | 'partner';
export interface SurfaceSpec {
  sessionKind: 'code' | 'partner';
  tools: 'all-coding' | 'non-bash-subset';
  layout: 'code-workspace' | 'doc-workspace';
  scope: 'git-root' | 'any-dir';
  artifacts: boolean;
  agentProfile: 'coding-default' | 'partner';
}
export const SURFACES: Record<Surface, SurfaceSpec> = { /* 见 HLD §9.4 */ };
```

```ts
// store/surface-inference.ts —— 隐式入口，纯函数、可单测
const DOC_EXTS = ['.pdf', '.docx', '.pptx', '.xlsx', '.csv', '.md', '.txt'];
export function inferSurface(sig: SurfaceSignal): Surface | null {
  if (sig.droppedFiles?.some(f => DOC_EXTS.includes(ext(f)))) return 'partner';
  if (sig.projectRoot && sig.isGitRepo === false) return 'partner';
  return null; // 无信号 → 不自动切，保持当前
}
```

切换是**渲染层路由**，不重启 runtime；同一 main 进程的 KodaX runtime 复用（与 Quick Ask「同 runtime 不同 client」同构）。

### 接口契约

无新 IPC。surface 是纯 renderer 状态 + session metadata。拖文件用 Electron `webContents` 的 `will-navigate`/HTML5 drop（renderer 内 `onDrop`），不新增 main 通道。

### 实现步骤

1. `store/surface.ts` + `SURFACES` 表 + zustand slice（`currentSurface` 默认 `'code'`）。
2. `store/surface-inference.ts` + 单测（doc 拖入 / 非 git 目录 / 无信号 三类）。
3. `shell/SurfaceTabs.tsx`：两个 tab，active 态、点击 `setSurface`；接 `inferSurface` 的自动切换 + 一行 toast「已切到 Partner（知识工作）」。
4. `App.tsx`：按 `currentSurface` 渲染 Coder 既有布局 or Partner 布局（F046 的 `<PartnerWorkspace/>`）。
5. session 创建（`shell/createSession.ts`）写入 `surface`；切换 session 时恢复其 surface。

### 验收标准

- [ ] 顶部常驻 `[Coder] [Partner]` tab，点击即时切换布局，不重启 runtime、不丢当前 session 列表。
- [ ] 拖入一个 `.pdf`/`.docx` → 自动切 Partner，toast 提示；拖入 `.ts` 源码不触发。
- [ ] 新 session 选非 git 目录 → 自动建议 Partner；选 git repo → 保持 Coder。
- [ ] `inferSurface` 单测覆盖三类信号 + 无信号返回 null。
- [ ] 切到 Partner 后再切回 Coder，Coder 的会话/右栏状态完好（无副作用）。
- [ ] 不破坏现有单 surface 行为：从不碰 Partner 的用户，体验与 v0.1.10 一致。

## FEATURE_046: Partner doc-workspace 三栏布局骨架

### 需求概述

Partner 的布局与 Coder 的 `code-workspace` 不同（[HLD §9.4](../HLD.md#94-surface-抽象)）：三栏 `Sources（左）| 对话 + 任务进度（中）| Artifact 预览（右）`，**去掉** Coder 的 Subagent tree / 内置 Terminal 抽屉（知识工作不需要）。本 feature 只立骨架与占位，Artifact 实体留 F048、skill 留 F049+。

### 影响范围

- 新目录 `apps/desktop/renderer/src/features/partner/`：
  - `PartnerWorkspace.tsx`——三栏容器
  - `SourcesPanel.tsx`——左栏：作用域目录树 + 已加入的知识源列表（文件/URL 占位）
  - `PartnerConversation.tsx`——中栏：复用 `shell/ConversationStreamV2` + 任务进度条占位
  - `ArtifactPanel.tsx`——右栏：空态占位（"产出会显示在这里"），F048 填充
- 复用：中栏对话流直接复用现有 `ConversationStreamV2` / 输入桥 `inputBridge`，不另造对话引擎。

### 技术方案

纯布局组件，无新状态机。三栏用现有栅格/分隔条样式（与右栏 popout 一致的视觉语言）。左栏 Sources 列表项数据先用 F047 的作用域目录；URL 知识源在 F052 接入，本版给空 section 占位。中栏任务进度条对接现有 Work 预算事件，但文案/语义为"多步文档事务"（"读 N 源 → 处理 → 产出"），不复用 H0/H1/H2 编码徽标。

### 接口契约

无新 IPC（复用 `agent.*` / `session.*` 既有通道）。

### 实现步骤

1. 建 `features/partner/` 四个组件，`PartnerWorkspace` 为 F045 路由目标。
2. 中栏挂 `ConversationStreamV2`（传入 Partner session）。
3. 左栏 Sources：渲染 F047 的作用域目录树（只读浏览）+ "添加知识源" 占位按钮。
4. 右栏 Artifact：空态组件 + 预留 `artifacts: ArtifactRef[]` props（F048 注入）。
5. 任务进度条占位组件（接 Work 事件，Partner 文案）。

### 验收标准

- [ ] 切到 Partner 显示完整三栏，无 Subagent/Terminal 抽屉。
- [ ] 中栏可正常对话（发消息、流式回答、tool call 卡片渲染），与 Coder 一致。
- [ ] 左栏显示当前作用域目录树；右栏显示 Artifact 空态文案。
- [ ] 窗口缩放 / 分隔条拖动布局稳定，无溢出。
- [ ] 复用 `ConversationStreamV2`，无对话逻辑复制粘贴。

## FEATURE_047: Partner 工具白名单（non-bash-subset）+ 非 git 文件作用域

### 需求概述

Partner session 必须跑在受限工具集上（[HLD §9.4](../HLD.md#94-surface-抽象)：`non-bash-subset`，默认无 `bash`），且文件作用域可锁到**任意目录（含非 git 的 Documents/Downloads）**，默认排除 `.git`。在 SDK 的"工具能力维度元数据"（[ADR-007](../ADR/ADR-007-partner-surface-model.md) R3）到位前，Space 侧按名单裁剪（参照已有 plan-mode blocklist 先例）。

### 影响范围

- **main / kodax 边界**：在起 Partner session 时，传入工具白名单/黑名单。复用现有 `real-session` 里 `PLAN_MODE_BLOCKED_TOOLS` 的硬编码裁剪机制，新增 `PARTNER_TOOL_ALLOW`（read/grep/glob/web_fetch/web_search + 富格式 IO；显式 **不含** bash/edit-as-code 等）。
- **main / projects**：作用域校验放宽——Partner 的 `assertAllowed` 接受非 git 目录；但仍做 path-traversal 防护，且默认把 `.git` 排除出 Sources 列表与读取范围。

### 技术方案

```ts
// electron/kodax/partner-tools.ts
export const PARTNER_TOOL_ALLOW = new Set([
  'read', 'grep', 'glob', 'web_fetch', 'web_search',
  // 富格式 IO（经 skill/registerTool 暴露的 pdf/docx/xlsx 读写）
]);
// 起 session 时：把不在 allow 集内的工具从注册表过滤（与 plan blocklist 同一个裁剪点）
```

作用域：Partner 的 `ProjectScope` 增加 `kind: 'any-dir'`，`assertAllowed(path, scope)` 仍强制 `isPathInside(scope.root)`，但不再要求 `scope.root` 是 git root。Sources 枚举默认 `ignore: ['.git', 'node_modules', ...]`。

> 注：这是 workaround，待 SDK R3（工具能力维度元数据）到位后切到"按 `network`/`readonly` 维度声明"，移除硬编码名单。design 里标注此切换点，对应 [[kodax_sdk_export_gaps]] Partner 批次 R3。

### 接口契约

- `project.openScope` IPC 增加 `allowNonGit?: boolean`（Partner 传 true）。
- session 创建参数增加 `toolPolicy: 'all-coding' | 'partner-subset'`（map 到工具裁剪）。

### 实现步骤

1. `electron/kodax/partner-tools.ts`：定义 `PARTNER_TOOL_ALLOW` + 裁剪函数（接到现有 blocklist 裁剪点）。
2. session 起点按 `surface`/`toolPolicy` 选裁剪集；Coder 不变。
3. `electron/projects/`：`assertAllowed` 支持 `any-dir` scope + path-traversal 测试。
4. Sources 枚举默认 ignore `.git` 等。
5. 单测：Partner session 内 bash 工具不可见；非 git 目录可作用域且 `../` 越界被拒。

### 验收标准

- [ ] Partner session 的工具清单**不含 bash**，含 read/grep/glob/web_*。
- [ ] Partner 可把作用域设到一个非 git 目录（如 `~/Documents/notes`），正常读取其中文件。
- [ ] 越界路径（`scope.root` 外）读取被拒，envelope 走错误而非 throw。
- [ ] `.git` 目录默认不出现在 Sources 列表、不被读取。
- [ ] Coder session 行为零变化（工具全集、git-root 作用域）。
- [ ] design 内标注 R3 到位后的 workaround 退场点。
