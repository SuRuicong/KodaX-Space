## Batch Intent

This batch turns Space from "KodaX in an Electron shell" into a desktop host that keeps up with KodaX as KodaX evolves. The principle is:

**KodaX owns runtime behavior; Space owns visibility, review, recovery, and user control.**

The batch is organized by release (version assignment current as of the 2026-06-29 F103 slip; [../FEATURE_LIST.md](../FEATURE_LIST.md) is the authoritative matrix):

## External Gap Review Adjustment (2026-06-19)

A cross-product gap review highlighted several useful directions. The planning decision is:

- No new pre-`v0.1.20` feature debt is reopened. `F081-F084` are treated as the current continuity closeout, with SDK-gated gaps recorded in the capability ledger.
- MCP is not a gap. `F039` already covers Space-side lifecycle and tool catalog management; future work is limited to resume-state visibility in `F090`.
- The provider config SSRF/env-injection item was a real `v0.1.19` release note risk, but the current working tree already guards `provider.addCustom`, KodaX `config.json` custom providers, and provider connection tests. It should be tracked as release verification, not as a new planned feature.
- OS-level execution sandboxing is valid strategic work, but it is not inserted into the workflow-v2 sequence. `F094` should produce a feasibility and threat-model checkpoint; implementation belongs with `F099` or a later sandbox-specific feature.
- PR/GitHub collaboration closure is useful, but it should land with connectors and Partner workbench in `v0.2.x`, not in the `v0.1.x` KodaX capability catch-up lane.
- Enterprise SSO/SAML/SCIM remains route-dependent. `v0.2.x` keeps a local-first policy/audit pack; identity federation is deferred until a deliberate enterprise track exists.

## v0.1.20 Execution Plan

### Slice 1: F081 Version and Capability Diagnostics

Scope:

- Add `docs/KODAX_CAPABILITY_LEDGER.md`.
- Align package/version docs or explicitly explain why internal packages are still `0.1.9`.
- Extend version diagnostics with KodaX SDK package/version, supported contract level, and degraded capability flags.
- Link About/Settings diagnostics back to the capability ledger or an exported diagnostics payload.

Acceptance:

- A developer or user can answer "which KodaX capability is Space consuming?" without reading source.
- Version drift is either fixed or intentionally documented.
- Repointel warm, Quick Ask sideQuery, and CLI teleport appear as explicit capability states.

### Slice 2: F082 Repointel Local Doctor and Trace Surface

Scope:

- Keep the existing session `repointel_trace` event path as the source of truth.
- Add a compact trace detail view from the chip and `/repointel` output.
- Add `repointel.status` or equivalent IPC that reports local availability, last trace, current project eligibility, and whether warm is SDK-supported.
- Show disabled warm controls with a precise reason until KodaX exposes a public warm API.

Acceptance:

- A project can show `available`, `idle`, `traced`, `degraded`, or `unavailable` without running a hidden warm action.
- The last trace explains mode, engine, bridge, latency, cache hit, and failure reason when present.
- No Space code reads private Repointel storage.

### Slice 3: F083 Quick Ask Continuity Before True Side Query

Scope:

- Harden temporary session lifecycle: created, completed, cancelled, deleted, and orphaned states should be visible in logs/diagnostics.
- Add "Continue in Coder" from Quick Ask, creating or opening a normal session seeded with the question and answer.
- Add "Continue in Partner" only as a visible planned/degraded action until Partner ingestion semantics are settled.
- Preserve current no-tools/plan-mode behavior until KodaX exposes true `sideQuery`.

Acceptance:

- Closing Quick Ask does not leave ambiguous hidden sessions.
- A useful Quick Ask answer can become a normal Coder session with provenance.
- The UI does not claim no-session semantics until the SDK actually provides them.

### Slice 4: F084 Space-Side Handoff Receiver

Scope:

- Define a handoff JSON schema for `~/.kodax/handoffs/<handoff-id>.json`.
- Add handoff IPC for `list`, `watch`, `accept`, `dismiss`, and stale/error states.
- Accept handoffs by validating workspace access and opening/resuming the referenced session through existing session IPC.
- Use fixture handoff files for manual verification before requiring CLI writer support.

Acceptance:

- A valid handoff file appears in Space and opens the matching session or gives a precise recoverable error.
- Invalid, stale, inaccessible, or mismatched-workspace handoffs are not silently ignored.
- Accepting a handoff never duplicates or rewrites KodaX session truth.

### Slice 5: F104 Display Language MVP

Scope:

- Add a persisted `languageMode: 'system' | 'zh-CN' | 'en-US'` setting and effective-locale resolver.
- Add `Settings > Preferences > Language` with `Follow system`, `简体中文`, and `English`.
- Add menu-bar quick switching under `View > Language`, backed by the same setting.
- Cover high-frequency app chrome: menu labels/items, Settings modal, left sidebar labels/actions, right sidebar tab/headings, and common modal/toast strings.
- Keep model responses, tool output, file contents, provider/model names, paths, command output, and artifacts untranslated.

Acceptance:

- A clean install follows the OS preferred UI language when it is Simplified Chinese, otherwise English.
- Users can switch to Simplified Chinese or English from Settings and from the menu bar without restarting.
- The chosen language persists across restart.
- The covered surfaces do not mix languages except for proper nouns, paths, provider/model names, and runtime/user/generated content.
- The menu-bar language item shows the checked effective selection and does not create a second preference state.

### SDK Dependency Gates

| Gate                             | Blocks                                          | v0.1.20 behavior before gate opens                                                         |
| -------------------------------- | ----------------------------------------------- | ------------------------------------------------------------------------------------------ |
| Public Repointel warm/status API | F112 adapter for real warm start/cancel/progress control | Show status from local trace/project availability and disable warm with explanation.       |
| KodaX `sideQuery` or equivalent  | F112 adapter for true no-session Quick Ask and Partner promotion | Keep temporary plan-mode session and make lifecycle/provenance explicit.                   |
| CLI/REPL handoff writer          | F112 integration for end-to-end teleport from terminal | Ship Space-side watcher/accept flow against fixture files and future CLI payload contract. |

### Recommended Order

1. F081 first, because it creates the capability ledger and diagnostics vocabulary.
2. F082 second, because current trace events already exist and can be made useful without waiting on warm APIs.
3. F083 third, because it improves daily UX while keeping the current SDK workaround honest.
4. F084 fourth, because the receiver can be implemented independently but end-to-end value depends on CLI writer timing.
5. F104 fifth or in parallel with renderer polish, because it is mostly host/settings/UI infrastructure and should not block SDK-gated runtime work.

## FEATURE_081: KodaX Capability Sync Ledger and SDK Contract Watchlist

### Problem

Space docs and implementation can drift from KodaX. A concrete example already happened: workflow host attribution was once documented as an SDK gap, but KodaX later shipped `hostMetadata` and Space started using it while older docs still described the gap.

### Design

Add a living capability ledger that maps KodaX features to Space consumption status.

Recommended location:

```text
docs/KODAX_CAPABILITY_LEDGER.md
```

Ledger fields:

| Field           | Meaning                                                                  |
| --------------- | ------------------------------------------------------------------------ |
| KodaX feature   | Upstream feature ID, version, and design link.                           |
| SDK/API surface | Import path, type, event, or command Space depends on.                   |
| Space status    | `not-needed`, `blocked`, `planned`, `partial`, `consumed`, `deprecated`. |
| Space feature   | F-id that implements the desktop surface.                                |
| Contract risk   | Compatibility, persistence, permission, or UX risk.                      |
| Verification    | Build/test/manual check proving the contract still works.                |

### UX

This feature is mostly internal, but it should surface in Settings or About as a diagnostics link: current KodaX package version, supported SDK contract level, and known degraded capabilities.

### Acceptance

- A capability ledger exists and links KodaX F211/F231/F232/F235/F233/F224/F228/F105/F113/F139 to Space features.
- README, PRD, HLD, and FEATURE_LIST do not disagree on current package/version status.
- Every future KodaX SDK bump includes a short ledger update or an explicit "no Space impact" note.

Implemented 2026-06-18:

- Added `docs/KODAX_CAPABILITY_LEDGER.md`.
- Extended `space.version` with KodaX SDK version, dependency spec, capability contract, and capability states.
- Aligned root, desktop, IPC schema, UI kit, and lockfile versions to the 0.1.20 release candidate baseline.
- Aligned the desktop workspace KodaX dependency range with the installed 0.7.53 SDK baseline.
- Added Space-side consumption of KodaX 0.7.53 sidecar verifier and todo-drift host callbacks through typed session events, renderer notices, and notifications.

## FEATURE_082: Repointel Doctor / Warm / Trace Control Plane

### Problem

Repointel is one of Space's core differentiators, but the current plan still treats status/warm as partial. Users need to know whether repo intelligence is installed, warmed, degraded, or unavailable, without learning daemon internals.

### Design

Create a compact Repointel control plane:

- status chip in Coder chrome,
- project-level warm action,
- doctor panel for install/path/daemon/cache/index health,
- trace viewer for the last context-pack/preturn decision,
- graceful fallback explanation when Repointel is missing.

Space only calls public KodaX/Repointel status and control APIs. It does not read private Repointel storage directly.

### Acceptance

- Opening Repointel status shows project/git/trace availability or a clear pending state.
- Warm support is shown as SDK-gated with a precise reason; Space does not run a hidden warm action.
- Missing project/git/trace states produce actionable diagnostics.
- A trace view explains recent context-pack/preturn decisions when SDK trace events exist.

Implemented 2026-06-18:

- Added `repointel.status` IPC with project/git/trace/warm diagnostics.
- Split `/repointel status` and `/repointel trace`; warm is explicitly reported as SDK-gated.
- Added status readout to the Repointel chip popover.

## FEATURE_083: Quick Ask v2 Continuity and Side Query Semantics

### Problem

Quick Ask should remain a temporary side-query surface, not a hidden chat product. The missing part is continuity: when a temporary answer becomes useful, the user needs a clean path into Coder or Partner without inventing a second session truth.

### Design

Define Quick Ask v2 semantics:

- no tools by default,
- no automatic session persistence,
- optional "Continue in Coder" and "Continue in Partner",
- optional "Attach current file/diff/project summary" when invoked from Space,
- sideQuery-style KodaX call once SDK support is available,
- short-lived transcript that expires unless promoted.

Promotion creates a normal KodaX session and records the original Quick Ask prompt/answer as a seed note.

### Acceptance

- Quick Ask can answer without adding a normal Coder session unless promoted.
- Promoting to Coder creates one normal session with provenance.
- Quick Ask never silently writes to repo files or KB.
- Keyboard invocation and dismiss behavior remain fast and predictable.

Implemented 2026-06-18:

- Quick Ask now captures session events locally instead of relying on the global session store.
- Added `Continue in Coder`, which promotes the persisted temporary session into Coder and lets `session.history` restore the transcript.
- True no-session side-query semantics remain blocked on SDK support.

## FEATURE_084: CLI to Space Teleport v1

### Problem

Space's PRD promises one session across CLI/REPL/desktop surfaces. Teleport is still deferred, so users cannot reliably move an active terminal session into the desktop review surface.

### Design

Implement file-system mediated handoff:

```text
~/.kodax/handoffs/<handoff-id>.json
```

Handoff payload:

- session id,
- workspace path,
- current branch/worktree if available,
- last known permission mode,
- requested target surface,
- optional focus hints such as file, diff, workflow run, or artifact id.

Space watches or scans handoffs, validates workspace access, then opens the matching session. The CLI/REPL remains source of truth for session data.

### Acceptance

- A valid fixture or future CLI handoff appears in Space and opens the matching persisted session history.
- Invalid or stale handoffs fail with clear errors.
- Opening a handoff never duplicates or rewrites session truth.
- Focus hints are parsed later; v0.1.20 receiver preserves session truth first.

Implemented 2026-06-18:

- Added handoff IPC for `list`, `accept`, `dismiss`, and `handoff.changed` watcher push.
- Added Space titlebar handoff inbox UI.
- Accepting a handoff first verifies the matching persisted session through existing `session.list`/history paths, then removes only the handoff descriptor with a session-id guard.

## FEATURE_104: Display Language MVP

### Problem

Users need an obvious way to switch KodaX Space between Simplified Chinese and English. The full F076 plan is still correct, but waiting for every surface, pseudo-locale QA, and CI scanner before shipping any language preference leaves the desktop app feeling unfinished for bilingual users.

### Design

F104 is a narrow, user-visible MVP extracted from F076. It ships the preference plumbing and the most visible UI coverage first, without claiming full localization.

Language model:

```ts
type SupportedLocale = 'zh-CN' | 'en-US';
type LanguageMode = 'system' | SupportedLocale;
```

Resolution rules:

1. `zh-CN` and `en-US` explicit modes always win.
2. `system` maps Electron preferred system languages or `navigator.languages` to `zh-CN` when the first supported Simplified Chinese match is present.
3. Other languages, empty values, `C`, and `POSIX` resolve to `en-US`.

Settings and IPC:

- Extend Space settings with `languageMode`, defaulting to `system`.
- Add `settings.setLanguageMode`.
- `settings.get` returns both stored `languageMode` and resolved `effectiveLocale`.
- Renderer may cache the last language mode for first paint, but main process settings remain source of truth.

UI entry points:

- Canonical entry: `Settings > Preferences > Language`.
- Quick entry: top menu `View > Language`.
- Both entries write the same setting and show the same three options: `Follow system`, `简体中文`, `English`.

MVP coverage:

- menu bar labels and menu item labels,
- Settings modal shell, tabs, and Preferences sections,
- left sidebar project/session chrome and empty states,
- right sidebar tab/headings for environment, live/runtime, changes, tasks/workers/plan panels,
- common modal/toast strings for close, save, cancel, open settings, permission confirmation, and basic error/success feedback.

Explicitly out of scope:

- assistant response language preference (F080),
- zh-Hant support (F079),
- pseudo-locale QA (F077),
- required i18n scanner/CI enforcement (F078),
- translating model output, user content, tool output, file contents, paths, provider/model names, command text, logs, or artifacts.

### Acceptance

- Fresh install defaults to `system` and resolves Chinese OS preferences to Simplified Chinese.
- Users can switch language from Settings and from `View > Language`.
- Switching updates covered renderer chrome without restart.
- The selected mode persists across app restart.
- Menu quick switching and Settings never drift into separate states.
- Covered surfaces have no obvious English/Chinese mixing except proper nouns and runtime/user-generated content.
- Unsupported/uncategorized UI strings remain acceptable only when listed as post-MVP F076 coverage debt.

### Implementation Notes

Recommended implementation order:

1. Add locale primitives and parity tests for the MVP key set.
2. Add settings schema migration and IPC tests for valid/invalid `languageMode`.
3. Add renderer provider/hook with first-paint cache and main-process reconciliation.
4. Add Settings language section.
5. Add top-menu `View > Language` checked submenu.
6. Migrate MVP surfaces and add a focused render test/smoke check for both locales.

F104 should leave the full F076 architecture intact: dictionary keys, resolver behavior, and Settings storage should be the foundation that F076 expands instead of code that F076 has to replace.
