## 背景与结论

Partner 已经在 PRD/HLD 里被定义为「同一 KodaX runtime 上的知识工作 surface」：工作对象是任意知识源，主要工具是 read/search/web/富格式 IO，证据标准是引用与来源核验，产出是 Artifact。现在缺的不是另一个 agent 内核，而是一个能让知识工作复利增长的持久知识工作区。

[llm-wiki.md](../llm-wiki.md) 的核心洞察是：普通 RAG 每次问答都从 raw documents 重新拼答案；LLM Wiki 则把来源读过一次后，持续维护一个结构化、交叉引用、能记录矛盾与综合结论的 markdown wiki。这个 wiki 是中间层：raw sources 是真相源，wiki 是编译后的可读知识，query 先查 wiki，再按需回 raw source。

本 feature 的结论：

**Partner Knowledge Base 不做「文件上传 + 临时 RAG 问答」，而做「LLM 维护的本地 persistent wiki + 可审计索引 + 可沉淀规则/技能/记忆」**。

它是 Partner doc-workspace 的持久工作对象，和 Artifact 一样是一等数据模型；它不替代 session，也不替代 raw source，而是让每次 ingest、query、lint 后的有价值知识都可以沉淀下来。

## 外部调研摘要

| 产品 / 模式 | 可借鉴点 | 对 Space 的取舍 |
|---|---|---|
| ChatGPT Projects | Project 把 chats、files、instructions、project memory 组织成长期工作区；回答可保存回 project source | 借「项目级上下文边界 + save response」；不照搬云端文件堆，Space 以本地 KB 目录为真相源 |
| OpenAI GPT Knowledge | 明确区分 knowledge=参考材料，instructions=行为规则 | 直接采纳：KB 内容不承载 Partner 行为约束，行为放 `PARTNER.md` / rules |
| Claude Code Memory | `CLAUDE.md` 管人为规则，auto memory 管模型发现的经验；`MEMORY.md` 只加载短入口，topic files 按需读取 | 借「短入口 + topic files 按需读」；避免每次把整库塞进 context |
| Devin / Windsurf Memories & Rules | 自动 memories、显式 rules、`AGENTS.md`、skills 分层；durable knowledge 推荐写入 rules/AGENTS，而不是依赖自动 memory | 借分层；Partner KB 主体是 wiki，memory 只放短事实或偏好 |
| Devin Knowledge / DeepWiki | Knowledge 有 trigger、folder、repo pin、suggestions、dedupe/conflict；DeepWiki 自动生成 repo wiki 且可用配置 steer | 借 trigger/folder/pin、维护任务、steer file；但 Space 的 wiki 可被用户本地查看和版本化 |
| Continue codebase/docs awareness | 从旧 `@Docs/@Codebase` 迁到 Agent tools + rules + MCP；旧索引是 embedding + keyword + rerank | 借 hybrid retrieval 作为后续索引层；MVP 先用 markdown index + rg/FTS |

## 核心决策

1. **四层知识架构**：Raw Sources / Compiled Wiki / Knowledge Index / Rules-Skills-Memory。
2. **markdown-first**：KB 是一个本地目录，可被 Obsidian、git、普通编辑器直接打开；Space 不把用户锁进私有数据库。
3. **raw source immutable**：来源进入 KB 后只追加 manifest 和 extract，不覆写原始来源；LLM 只维护 wiki。
4. **compiled wiki 是主价值**：检索是辅助，真正复利来自 LLM 更新 entity/concept/synthesis pages。
5. **可审计引用**：wiki 每条重要 claim 必须能追到 source id；query 回答默认带引用。
6. **显式保存高价值回答**：query 产生的比较表、分析、决策笔记可以保存回 wiki，成为新知识页。
7. **MVP 不上复杂向量库**：小规模用 `index.md` + `log.md` + `rg` / SQLite FTS；后续再加 hybrid BM25/vector/rerank。
8. **不改 KodaX 内核**：本 feature 是 Space 侧 surface/data/tooling；Partner harness 的 faithfulness 目标仍等 F053/SDK R1-R2。

## 本版本范围

F070 交付 Partner KB 的 MVP：

- 一个本地 KB 目录模型与 manifest schema。
- Partner doc-workspace 左栏加入 KB / Sources 入口。
- `create / open / ingest / query / save-answer / lint` 六个工作流的 host + UI 基线。
- wiki 文件规范：frontmatter、source 引用、claim 置信度、contradictions、open questions。
- `index.md` / `log.md` 自动维护。
- 只读检索 + web/source ingest 权限边界。
- `create_artifact` 与 KB 的桥接：wiki query 可以生成 report artifact，artifact 可保存为 wiki page。

不在本版本做：

- 企业协作实时多人编辑。
- 云同步与权限分发。
- 全量 Connector 同步器（Slack/Drive/Notion 等只预留 source type）。
- 重型向量库、知识图谱、自动调度 daemon。
- 让 Partner 自动写用户任意目录文件；KB 写入只发生在用户选定的 KB root。

## FEATURE_070: Partner Knowledge Base（Persistent Wiki）

### 1. 需求概述

让 Partner 能围绕一个可持久化的知识库工作：用户可以把本地文件、网页、Artifact、对话回答加入 KB；Partner 读取 raw sources，维护 wiki 页面和索引；后续提问优先基于这个已编译 wiki，并能把有价值的探索结果继续沉淀回 KB。

一句话：**把 Partner 从「一次性文档问答」升级为「会持续整理资料、维护知识库、积累综合结论的知识工作伙伴」。**

### 2. 用户故事

1. 用户拖入一组 PDF / docx / 网页链接，创建「竞品调研」KB；Partner 逐个 ingest，生成 source summary，并更新 competitors、pricing、positioning 等主题页。
2. 用户问「A/B/C 三家谁更适合中型团队？」Partner 先查 wiki，再回 raw source 核验关键 claims，回答带出处，并提示可保存为 `syntheses/vendor-fit.md`。
3. 用户每周追加新资料；Partner 更新已有页面，标记新旧资料矛盾，不把旧结论静默覆盖。
4. 用户运行 lint；Partner 找出孤儿页、无引用 claim、过期来源、重复页面、待补充资料，并生成维护建议。
5. 用户把一次 query 生成的报告保存为 Artifact；再将 Artifact 正文保存回 wiki，未来 query 可以复用。

### 3. 信息架构

#### 3.1 Raw Sources

Raw Sources 是真相源，LLM 只读不改。来源可以是：

- 本地文件：PDF / docx / pptx / xlsx / csv / md / txt / image。
- Web snapshot：URL、抓取时间、正文 extract、截图或图片资产。
- Connector snapshot：Slack/Drive/Notion/GitHub Issues 等后续类型。
- Artifact snapshot：Space 内已生成的 markdown/html/chart/doc artifact。
- Conversation excerpt：用户显式保存的回答或片段。

每个 source 进入 `sources/manifest.jsonl`，记录稳定 `source_id`、hash、时间、来源类型、标题、权限状态和 extract 路径。

#### 3.2 Compiled Wiki

Compiled Wiki 是 LLM 维护的 markdown 页面。页面类型：

- `sources/<source_id>.md`：单个来源摘要、关键摘录、图片/表格说明、source metadata。
- `entities/*.md`：公司、产品、人物、系统、项目等实体。
- `concepts/*.md`：概念、主题、术语、机制。
- `syntheses/*.md`：跨来源综合、比较、结论、研究报告。
- `decisions/*.md`：决策记录、推荐、取舍。
- `timelines/*.md`：事件、版本、会议时间线。

Wiki 页面必须有 YAML frontmatter：

```yaml
---
kb_page_type: synthesis
title: Vendor fit for mid-market teams
created_at: 2026-06-18T10:00:00+08:00
updated_at: 2026-06-18T10:00:00+08:00
sources:
  - src_20260618_001
  - src_20260618_002
confidence: medium
status: active
tags: [vendor-analysis, mid-market]
---
```

正文建议结构：

```markdown
# Vendor fit for mid-market teams

## Summary

## Key Claims

- Claim text. [src_20260618_001]

## Evidence

## Contradictions / Tensions

## Open Questions

## Related

- [[entities/vendor-a]]
- [[concepts/deployment-model]]
```

#### 3.3 Knowledge Index

MVP 索引由三个部分组成：

- `wiki/index.md`：内容目录。按 page type 分组，列 page link、一句话摘要、source count、updated_at。
- `wiki/log.md`：时间线。append-only 记录 ingest/query/save/lint。
- `indexes/fts.sqlite`（可选，MVP 可后置）：markdown FTS，用于快速全文检索。

后续可加：

- BM25 + vector hybrid。
- LLM rerank。
- qmd / MCP 搜索适配。
- source graph / entity graph。

#### 3.4 Rules / Skills / Memory

KB 不承载行为规则。规则与流程放在独立层：

- `PARTNER.md`：该 KB 的工作协议，例如引用格式、术语偏好、输出风格、哪些来源优先。
- `rules/*.md`：可启用/禁用的短规则，适合 trigger/glob/model decision。
- `skills/*/SKILL.md`：复用流程，例如 `ingest-market-report`、`weekly-kb-lint`、`make-decision-brief`。
- `memory/MEMORY.md`：短入口，只保存 Partner 自己发现、未来有用的操作经验，不保存主要知识。

### 6. 工作流

#### 6.1 Create / Open KB

入口：

- Partner 左栏 `Knowledge Base` 区域。
- 拖文件进入 Partner 时提示「加入当前 KB / 创建新 KB / 仅本次对话使用」。
- Artifact 菜单 `Save to Knowledge Base`。

行为：

1. 用户选择 KB root。
2. Space 初始化目录与 `kb.json`。
3. Partner session context 注入当前 KB 摘要：`kb title + index.md 前 N 行 + PARTNER.md`。
4. KB root 写入 session metadata，切换 session 后恢复。

#### 6.2 Ingest Source

流程：

1. 将 source 追加到 `sources/manifest.jsonl`，状态为 `pending`。
2. 根据类型抽取文本、表格、图片说明，写入 `sources/extracts/<source_id>.md`。
3. Partner 读取 extract，生成 `wiki/sources/<source_id>.md`。
4. Partner 根据 source 内容更新相关 entity/concept/synthesis pages。
5. 更新 `wiki/index.md`。
6. 追加 `wiki/log.md`：

```markdown
## [2026-06-18] ingest | src_20260618_001 | Vendor A Security Whitepaper

- Added source page: [[sources/src_20260618_001]]
- Updated: [[entities/vendor-a]], [[concepts/security-posture]]
- Notes: New source conflicts with src_20260610_004 on SOC2 timeline.
```

UI 要求：

- Ingest 是一个可观察任务，有步骤进度：`extract → summarize → update pages → index → done`。
- 写 wiki 前展示更新摘要；MVP 可以自动写入，但必须有 `View changed pages`。
- 失败 source 不阻断其他 source，manifest 记录失败原因。

#### 6.3 Query KB

流程：

1. 读 `wiki/index.md` 和当前 query。
2. 用 `rg` / FTS 找相关 wiki pages。
3. 读相关 wiki pages。
4. 对关键 claims 按需回读 source extract 或 raw source。
5. 回答带 citations。
6. 若回答形成可复用知识，提示 `Save answer to wiki`。

回答格式：

```markdown
结论...

依据：
- [src_20260618_001] ...
- [src_20260618_004] ...

不确定点：
- ...
```

#### 6.4 Save Answer to Wiki

入口：

- Assistant message 菜单：`Save to KB`。
- Artifact 菜单：`Save as wiki page`。

流程：

1. 用户选择 page type 与目标路径，或让 Partner 建议。
2. Partner 把回答转成 wiki page 格式，保留 citations。
3. 更新 `index.md` / `log.md`。

#### 6.5 Lint / Maintenance

Lint 检查项：

- orphan pages：没有 inbound links。
- uncited claims：关键断言没有 source id。
- stale claims：页面更新时间早于相关 source。
- contradictions：不同页面对同一主题结论冲突。
- duplicate pages：高度相似页面。
- missing pages：index 中频繁出现但没有独立页面的实体/概念。
- broken links：`[[...]]` 指向不存在。

输出：

- 一个 maintenance report artifact。
- 可选自动修复：补 index、补 related links、合并重复页草案。
- 高风险修复（删除/合并/覆盖结论）必须确认。

### 9. Tool / Permission 边界

Partner 当前只允许 read/search/web/create_artifact。F070 需要新增 Space 自有 KB 工具，但它们只写 KB root，不写项目任意文件：

```ts
export const PARTNER_SPACE_TOOL_ALLOW = new Set([
  'create_artifact',
  'partner_kb_read',
  'partner_kb_search',
  'partner_kb_ingest',
  'partner_kb_write_page',
  'partner_kb_update_index',
]);
```

安全约束：

- 所有 KB 写入 path 必须 resolve 后位于 `kb.rootPath` 内。
- `sources/raw` 若复制本地文件，必须由用户确认；默认只记录原路径 + hash + extract。
- web snapshot 需要网络权限提示，沿用 Partner web 研究策略。
- Connector snapshot 后续必须显示 connector 名称、scope、retention。
- 删除 / 合并 wiki page 需要确认。
- 任何 source 中疑似 secret 的内容不进入 `index.md` 摘要；lint 报告提示但不扩散。

### 10. 索引策略

MVP：

- `index.md` 是第一入口，控制在可加载范围内。
- `rg` 搜 wiki 与 extracts。
- `log.md` append-only，便于最近活动恢复。

规模阈值：

- `wiki/index.md` 超过 25KB：生成分区 index，如 `index.entities.md` / `index.syntheses.md`。
- source > 200 或 page > 500：启用 `indexes/fts.sqlite`。
- source > 1000 或 query latency > 2s：评估 BM25/vector/rerank。

### 11. 与现有功能关系

- F045/F046/F047：F070 依赖 Partner surface、doc-workspace、tool whitelist 已存在。
- F056-F059：F070 复用 Artifact 数据层和预览；query/lint report 可落 Artifact。
- F049/F051/F052：这些 skill pack 可以把 F070 作为持久化目标，但不是硬依赖。
- F053：完整 Partner harness 会增强 faithfulness/completeness 判官，但 F070 MVP 不依赖。
- Workflow F060-F066：后续 ingest/lint 可以升级成 workflow run；MVP 可先用单任务进度。

### 12. 实现步骤

1. Schema：新增 `partner-kb` IPC schemas 和 types。
2. Main：实现 KB registry、manifest append、page read/write、index/log helper。
3. Extract：接入已有富格式读取能力，至少支持 md/txt/pdf/docx/xlsx/web markdown。
4. Renderer：Partner 左栏新增 KB panel，右栏新增 wiki/source preview。
5. Agent steering：Partner system prompt 增加 KB 操作协议；新增 Space 自有 KB tools。
6. Ingest workflow：source manifest → extract → source page → index/log。
7. Query workflow：index/search/read → answer with citations → save-answer。
8. Lint workflow：输出 report artifact，支持低风险自动修复。
9. Tests：schema、path sandbox、manifest append、index/log update、renderer interaction。

### 13. 验收标准

- 用户能在任意目录创建 KB；目录结构与 `kb.json` 正确生成。
- 拖入一个 markdown/txt/PDF source 后，manifest、extract、source page、index、log 都正确更新。
- 对 KB 提问时，Partner 会优先读取 `index.md` 和相关 wiki page，回答包含 source id 引用。
- 用户能把一次回答保存为 wiki page；保存后 `index.md` 和 `log.md` 更新。
- Lint 能发现至少三类问题：broken links、uncited claims、orphan pages，并生成 artifact report。
- Path sandbox 单测覆盖 `..`、绝对路径、符号链接/Windows junction 等越界写入，全部 fail closed。
- 关闭/重开 Space 后，当前 Partner session 能恢复绑定的 KB。
- 不创建 KB 时，Partner 现有对话和 artifact 行为不变。

### 14. 开放问题

1. KB registry 放在 `~/.kodax/space/partner-kbs.json`，还是仅靠最近打开目录列表？建议用 registry，便于 sidebar 恢复。
2. 本地文件 source 默认是否复制到 `sources/raw/`？建议默认不复制，用户可开启「portable KB」。
3. query 的 citations 格式是否统一成 `[src_id]`，还是支持脚注？MVP 用 `[src_id]`，导出报告时再转脚注。
4. 是否需要 KB 级 git init？MVP 不自动 init，但 UI 提供「Open folder」让用户自己管理；后续可加可选按钮。
5. web snapshot 的正文抽取由 headless fetch 还是有头浏览器工具完成？MVP 用 `web_fetch`，F050 完成后升级到 browser snapshot。

### 15. 后续阶段

#### F071：Partner KB hybrid search

把 MVP 的 markdown index + `rg` 检索升级为可扩展检索层。第一阶段用 SQLite FTS + BM25；第二阶段允许可选 vector index + rerank。UI 不暴露「选算法」配置，而是显示结果来源、匹配理由、wiki/raw source 命中比例和 fallback 状态。

验收：KB query 能同时命中 wiki page、source summary 和 raw extract；结果带 source id；禁用 vector 时仍能全文检索；索引损坏时可重建。

#### F072：KB maintenance scheduler

为 KB 增加周期维护任务：lint、source freshness check、孤儿页、无引用 claim、重复主题、矛盾 claim、过期来源。MVP 先做手动运行 + 上次运行结果；v0.2.x 再接 Local Automations Scheduler。

验收：用户可以运行维护检查并得到可操作清单；每条建议能跳到对应 page/source；维护结果写入 `log.md`。

#### F073：Connector snapshots

把 Slack、Drive、Notion、GitHub Issues/PR 等 connector 内容作为可审计 snapshot 进入 KB。snapshot 必须记录 connector 名、scope、抓取时间、原始 URL/对象 id、retention policy，不把 connector 当前在线内容当作永远稳定的真相源。

验收：connector source 进入 `sources/manifest.jsonl`；撤销 connector 后已保存 snapshot 仍可读，但不能刷新；query 引用能显示 connector 来源。

#### F074：Steerable wiki config

增加 KB 级配置文件，用来指导 wiki 组织方式、source 优先级和 lint policy。建议文件名：

```text
wiki.config.json
```

配置只影响维护策略，不改变 raw source。可配置项包括 page groups、pinned sources、preferred synthesis pages、ignored paths、claim policy 和 freshness window。

验收：配置 schema 可校验；无配置时有稳定默认值；非法配置不阻塞打开 KB，只显示诊断。

#### F075：Knowledge graph view

把 entity、concept、source、synthesis page 的链接关系可视化，帮助用户发现孤岛、过度依赖单一来源、矛盾区域和高价值主题。它是探索/维护视图，不是花哨图谱数据库。

验收：图谱节点可跳转到 page/source；能按 page type/source/tag 过滤；大型 KB 自动降级为列表/分组视图。
