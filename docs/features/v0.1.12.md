## FEATURE_048: Artifact 一等概念

### 需求概述

Partner 的产出（report / slides / sheet / doc / markdown）不能只是对话流里的一段文本，必须是**一等产物**：可在右栏富预览、带版本可"再改一版"、可导出到文件或外部（PR/Issue/Slack）。本 feature 立 artifact 的数据模型 + 右栏预览 + 迭代 + 导出；富格式渲染复用既有 `features/preview/`（F024）。

### 影响范围

- **renderer store**：新增 `store/artifacts.ts`——`ArtifactRef { id, sessionId, kind, title, path?, versions: ArtifactVersion[] }`，`kind: 'markdown'|'docx'|'pptx'|'xlsx'|'report'`。
- **renderer**：`features/partner/ArtifactPanel.tsx` 填充——列表 + 当前 artifact 富预览（复用 `features/preview/` 渲染器）+ 版本切换器 + 导出按钮。
- **main**：新增 `electron/ipc/artifact.ts`——`artifact.write`（写产物到作用域目录/临时区）、`artifact.export`（另存为 / 复制到剪贴板 / 贴外部占位）。
- **产出捕获**：agent 产出 artifact 的约定——优先复用 SDK 已有的 artifact 概念（`KodaXInputArtifact` / `listArtifacts` / `readArtifact` 在 SDK export 中存在），把 agent 写出的产物登记为 ArtifactRef；无 SDK 支持的 kind 走 main 侧文件落地。

### 技术方案

```ts
// store/artifacts.ts
interface ArtifactVersion { v: number; createdAt: string; path: string; summary?: string; }
interface ArtifactRef {
  id: string; sessionId: string;
  kind: 'markdown' | 'docx' | 'pptx' | 'xlsx' | 'report';
  title: string; current: number; versions: ArtifactVersion[];
}
```

- **登记**：监听 agent 产出事件——若 agent 经 write/skill 落了一个产物文件，main 发 `artifact.registered` 事件，renderer 入 store。优先消费 SDK 的 `listArtifacts`/`readArtifact`（若 managed run 暴露），否则按"产物写进作用域 + 约定路径"捕获。
- **迭代**："再改一版" = 带上当前 artifact 内容作为 context 发新指令，agent 产出新版本 → `versions.push`，`current++`，预览默认跳到最新，可回看旧版（复用 Monaco diff 比对两版）。
- **导出**：`artifact.export` → 另存为（原生 save dialog）/ 复制 markdown 到剪贴板 / "贴到 PR/Issue/Slack"（M2 Connector 前先占位为复制）。

### 接口契约

```ts
// packages/space-ipc-schema/src/channels/artifact.ts
artifact.write   : { sessionId, kind, title, content } → { id, version, path }
artifact.list    : { sessionId } → { artifacts: ArtifactRef[] }
artifact.read    : { id, version? } → { content, kind }
artifact.export  : { id, version?, target: 'file'|'clipboard' } → { ok, path? }
```

安全：写产物受 F047 作用域约束（path inside scope）；导出"file"走原生 dialog（用户选路径，不静默写任意位置）。

### 实现步骤

1. schema `channels/artifact.ts` + index 注册。
2. `store/artifacts.ts` slice。
3. `electron/ipc/artifact.ts`：write/list/read/export handler；先支持 markdown + 透传 F024 可渲染格式。
4. 产出捕获：探测 SDK `listArtifacts`/`readArtifact` 在 managed run 是否可用；可用则消费，否则文件约定捕获（design 标注二选一的实测点）。
5. `ArtifactPanel.tsx`：列表 + 预览（接 `features/preview/`）+ 版本切换 + 导出。
6. 迭代链路：把"再改一版"接到对话输入（注入当前版本内容为 context）。
7. 测试：write→list→read→export 闭环；版本递增 + 旧版可读；越界写被拒。

### 验收标准

- [ ] agent 产出一份 markdown，右栏 ArtifactPanel 出现条目并富预览。
- [ ] "再改一版"产生 v2，预览跳最新，可切回 v1，两版可 diff。
- [ ] 导出"另存为"弹原生 dialog 写到用户选定路径；"复制"把 markdown 进剪贴板。
- [ ] docx/xlsx/pdf 类 artifact 复用 F024 渲染器预览（不另造渲染）。
- [ ] 产物写入受 F047 作用域约束，越界被拒。
- [ ] design 标注"SDK artifact API vs 文件约定捕获"的实测选择点。

## FEATURE_055: 打包 renderer 迁 `app://` 自定义标准协议

> 2026-06-14 起。Artifact 路径 D（嵌 LiveCanvas sandbox，见记忆 `livecanvas_artifact_plan` / `livecanvas_gap_sandbox_bundle`）的**打包正式版安全地基**。源于与 LC 的集成确认：sandbox iframe 靠 `postMessage` 锁定一个**真实可比对的父窗口 origin**；打包后 renderer 走 `file://`（origin 为 `null`/不透明）无法锁定，只能 `"*"`（任何本机进程可冒充父窗口、隔离作废）——LC 明确此路无安全解。解法是把打包 renderer 迁到注册的 `app://` 标准安全协议，给它一个真实 origin `app://space`。**LC 侧据此放宽 `parent-origin.ts`/`child.ts` 的协议检查以接受受信列表中的 `app://`，并保留禁 `"*"`/null。**

### 需求概述

把**打包正式版**的 renderer 从 `file://`（`loadFile`）迁到 `protocol.registerSchemesAsPrivileged` 注册的 `app://` 标准安全协议，window 经 `loadURL('app://space/index.html')` 加载，主进程 `protocol.handle('app', …)` 从 `RENDERER_DIST` 提供文件。**dev 不变**（仍 `http://localhost:5173`，Vite HMR 需 http；LC localhost bypass 已覆盖）。这是 app 级（整窗口 origin）改动，非 artifact 局部，单列一个 feature。

### 影响范围（集中在 `electron/main.ts` 的"大门口"）

- **scheme 注册**：`app.whenReady()` 之前 `protocol.registerSchemesAsPrivileged([{ scheme:'app', privileges:{ standard:true, secure:true, supportFetchAPI:true, codeCache:true } }])`。
- **protocol handler**：`protocol.handle('app', …)` 把 `app://space/<path>` 映射到 `RENDERER_DIST` 内文件；path-traversal 安全（只 serve 目录内，拒越界），MIME 推断。
- **窗口加载**：`createMainWindow()` 的 prod 分支 `loadFile` → `loadURL('app://space/index.html')`；dev 分支不动。
- **CSP**（`applyCsp`）：prod `default-src 'self'` 对 `app://space` 自然生效；保留 `THEME_BOOTSTRAP_INLINE_HASH`、`worker-src 'self' blob:`（Monaco）、`style-src 'unsafe-inline'`；按需把 artifact sandbox 的 `frame-src`/`connect-src` 加进来（F048 路径 D 消费时）。
- **导航/外链**：`will-navigate` 的 `ALLOWED_FILE_PREFIX` 判定 → `app://space/` 前缀；`setWindowOpenHandler`、`will-navigate` 的 https 外链放行逻辑保持。
- **不受影响**：preload（仍按 `PRELOAD_PATH` 注入）、所有 IPC、terminal/mcp/notification/updater 等（它们不依赖 renderer 的 scheme）。

### 技术方案

```ts
// 1) app ready 之前（模块顶层或 whenReady 之前同步调）
protocol.registerSchemesAsPrivileged([
  { scheme: 'app', privileges: { standard: true, secure: true, supportFetchAPI: true, codeCache: true } },
]);

// 2) whenReady 内、createMainWindow 之前
const RENDERER_ORIGIN = 'app://space';
protocol.handle('app', (request) => {
  const { pathname } = new URL(request.url);          // app://space/index.html → /index.html
  const rel = decodeURIComponent(pathname).replace(/^\/+/, '') || 'index.html';
  const abs = safeJoinWithinRoot(RENDERER_DIST, rel); // 越界→null
  if (!abs) return new Response('forbidden', { status: 403 });
  // SPA/静态：缺失文件回退 index.html 视需要；MIME 由扩展名推断
  return netFileResponse(abs);                        // 复用 net.fetch(pathToFileURL) 或自读
});

// 3) prod 加载
mainWindow.loadURL(`${RENDERER_ORIGIN}/index.html`);
```

注意点：① Windows + asar 下 `RENDERER_DIST` 路径与现状一致（`__dirname/../apps/desktop/dist`）；handler 读 asar 内文件用 `net.fetch(pathToFileURL(abs))` 最稳（Electron 原生支持读 asar）。② `app://space` host 段必须合法且固定；renderer 内所有资源引用须 root-relative（Vite build base='./' 或 '/' 需核对，保证 `app://space/assets/...` 解析正确）。③ secure scheme 才是 secure context（crypto.subtle / SW 等可用），与 file:// 行为差异需回归。

### 接口契约

无新 IPC。导出常量 `RENDERER_ORIGIN = 'app://space'`（prod）作为 **artifact sandbox 注入的受信父 origin**（F048 路径 D：Space 自托管 sandbox 时把 `window.__LC_TRUSTED_PARENT_ORIGINS__ = [RENDERER_ORIGIN]` 注入 sandbox 的 index.html，见 LC 集成回复 Q4）。dev 下父 origin 取 `window.location.origin`（`http://localhost:5173`）走 LC bypass。

### 实现步骤

1. `registerSchemesAsPrivileged`（app ready 前）。
2. `protocol.handle('app', …)` + path-safety helper + asar 友好读取，单测覆盖正常/越界/MIME。
3. prod `loadURL('app://space/index.html')`；核对 Vite build 的 asset base，保证资源 root-relative 解析。
4. 改 `applyCsp` / `will-navigate` / `ALLOWED_FILE_PREFIX` / windowOpenHandler 的 prod 分支到 `app://` 前缀。
5. **全链路回归**（关键）：启动可见、session/对话、内置终端、Monaco（worker blob:）、MCP、通知、自动更新、外链放行、剪贴板粘贴图片——逐项在打包 smoke 下验不回归。
6. e2e：打包构建 boot smoke 断言 `location.origin==='app://space'` + 现有 e2e 全过。

### 验收标准

- [ ] 打包版 renderer 从 `app://space/index.html` 加载，`window.location.origin === 'app://space'`。
- [ ] dev 模式仍 `http://localhost:5173`，HMR/DevTools 正常。
- [ ] `app://` handler 拒绝路径穿越（`../` 越界 → 403），MIME 正确。
- [ ] 现有全部功能在 `app://` 下无回归（session/terminal/mcp/monaco/通知/更新/外链/剪贴板）。
- [ ] CSP 不引入任何 `'unsafe-*'` 新放宽（除既有 Monaco blob: / theme hash / style inline）。
- [ ] 导出 `RENDERER_ORIGIN` 供 F048 路径 D 作受信父 origin；与 LC 集成回复 Q1/Q4 一致。
- [ ] secure-context 行为差异（vs file://）已逐项回归，无功能因 origin 变更而坏。

## FEATURE_056: 静态 Artifact 基线（LC-free）+ tier 门控

> 2026-06-15 起。**发布解耦**：LiveCanvas（LC）是半成品，Space 接近可发布。把 artifact 能力拆成两层 tier —— **静态 tier（LC-free，发布基线）** + **交互 React tier（依赖 LC 的 sandbox，路径 D）**。静态 tier 用 Space 现成能力拼装（md/code/F024 富预览 + 新增 html/chart），**发布永不依赖 LC**；react tier 门控关闭，待 LC GA（发 npm + F055 app://）再开。决策见与用户 2026-06-15 讨论。

### 需求概述

绝大多数真实 artifact 场景（报告 / 文档 / 表格 / 图表 / 排版）是**静态可渲染**的，不需要 eval 沙箱。唯有"AI 写任意交互式 React 当场跑"才需 LC sandbox。故：静态 kinds 全部 LC-free，作发布基线；`react` kind 收进 tier 门控（`isReactArtifactEnabled()`，发布 = false）。

### 影响范围

- **renderer 新增** `features/artifact/`：
  - `artifactKind.ts` —— kind 枚举 + 门控 `isReactArtifactEnabled()`。
  - `ArtifactView.tsx` —— 按 kind 派发的渲染器注册表。
  - `renderers/`：`HtmlArtifact`（iframe `srcdoc` + `sandbox=""` 禁脚本，新）、`ChartArtifact`（recharts from JSON spec，新）、`SvgArtifact`/`ImageArtifact`（直渲）；markdown→复用 `messages/Markdown`，code→Monaco，pdf/docx/xlsx→复用 F024 `RichPreview`，react→门控 `SandboxFrame`（P1c）。
- **renderer 依赖**：新增 `recharts`（仅 renderer；chart spec 直渲，无 eval）。
- **现有 P1c（LC tier）**：`SandboxFrame` / `useSandboxInfo` 原样保留，被门控收口。

### 技术方案

```ts
type ArtifactKind = 'markdown'|'code'|'html'|'svg'|'image'|'pdf'|'docx'|'xlsx'|'chart'|'react';
// 静态 kinds（LC-free）发布即可用；'react' 仅 isReactArtifactEnabled() 为真时启用。
```

- **html**：`<iframe srcdoc={html} sandbox="">`（空 sandbox = 禁脚本/表单/弹窗 → 纯静态安全；无新依赖、无需 dompurify，因 sandbox 已禁 JS）。
- **chart**：zod 校验 `{ type:'line'|'bar'|'area', data:[], series:[{key,color?}], xKey }` spec → 映射成 recharts 元素（renderer 内直渲，无 iframe、无 eval）。spec→config 抽纯函数便于单测。
- **门控**：`isReactArtifactEnabled()` 默认 false（发布）；dev / 显式 flag 开。`react` kind 命中门控关 → 显示"交互预览暂不可用"或不向 agent 开放该 kind。

### 实现步骤

1. `artifactKind.ts`（kind 枚举 + 门控）。
2. 新渲染器 `HtmlArtifact` + `ChartArtifact`（+ chart spec zod + 纯 map 函数）。
3. `ArtifactView` 注册表 + 复用包装（markdown/code/doc）。
4. recharts 依赖；ArtifactPanel 接 `ArtifactView`。
5. 验证：HTML 沙箱禁脚本（Playwright 独立 harness 真验）+ chart spec map（node:test 纯函数）+ typecheck/lint。
6. 后续层：artifact store + IPC + agent 生成（让基线真正产出/展示）。

### 验收标准

- [ ] 静态 kinds（markdown/code/html/svg/image/pdf/docx/xlsx/chart）全部 LC-free 渲染，无 `@livecanvas/*` / `@kodax-ai/livecanvas-sandbox-shell` import。
- [ ] HTML artifact 的 `sandbox=""` iframe 确认禁脚本（注入 `<script>` 不执行）。
- [ ] chart spec → recharts 渲染出 svg；非法 spec 安全降级（不崩）。
- [ ] `react` tier 默认门控关闭：发布版不暴露交互 React artifact。
- [ ] 复用 F024 RichPreview 渲染 pdf/docx/xlsx artifact，不另造。
- [ ] 发布构建不因 LC 缺失而坏（artifact 静态能力完整可用）。

## FEATURE_058: Artifact 生成

### 需求概述
agent 如何产出 artifact —— **in-process `create_artifact` 工具**(走 SDK `registerTool`,同 F050 路子),agent 对"成形产物"自动 emit;`artifactId` 省=新建、带=新版本(迭代)。steering 优先静态 kind(react 仅门控开时开放)。

### 技术方案

### 验收标准
- [ ] agent 在对话中产出 markdown/chart artifact,自动登记进 store + 右栏出现。
- [ ] 迭代:带 artifactId 再 emit → 新版本。
