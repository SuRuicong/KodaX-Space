# KodaX Space v0.1.0 — Alpha Foundation

> Target: 2026-Q2
> Theme: 在桌面跑通 KodaX session 全流程
> Status: Planned
> Definition of Done: 开发者能在 Space 里完成"打开项目 → 提问 → 看 tool 调用 → 改动落盘 → diff 审查"

## 版本范围

v0.1.0 是 KodaX Space 的**第一个 alpha**。目标是**架构验证**而非功能完备：

- 验证 Electron + in-process KodaX runtime 模型可跑（[ADR-003](../ADR/ADR-003-kodax-integration-in-process.md)）
- 验证 main → renderer 的 IPC schema 与事件流可承载 KodaX session
- 验证 Provider 配置 + Keychain + LLM 调用全链路打通
- 验证 Permission 弹窗能正确拦截危险操作
- 验证 Monaco diff 渲染能服务 review 工作流

**不在 v0.1.0 范围**：
- 内置终端（v0.1.1）
- Subagent 可视化（v0.1.1）
- MCP 管理 GUI（v0.1.1，但 MCP 通过 KodaX 内核 config 可用）
- Quick Ask popover（v0.1.2）
- Repointel UI（v0.1.2）
- 主题切换（v0.1.3）

## MVP 主流程

```text
启动 Space
  → Welcome 屏：选择主语言 + 配 1 个 provider（写 keychain）
  → 选项目目录
  → 自动新建 session
  → 输入 "Read package.json and summarize"
  → 看 tool call 卡片（read 工具）+ tool result
  → 看 LLM 回答流式渲染
  → 再问 "Find security issues in src/auth.ts and propose fixes"
  → 看 read → grep → write 工具链
  → write 工具触发 Permission 弹窗 → 用户 Allow
  → 在 File panel 看 diff
  → 任务完成，session 落盘到 ~/.kodax/sessions/<id>.jsonl
```

## FEATURE_001: Electron 工程骨架

> Status: **Completed** (2026-05-16, v0.1.0-alpha.0)
> 实际版本：Electron 33（≥ 30+ 基线满足）；npm workspaces；esbuild → CJS（main + preload）；Vite 6 → renderer。
> 安装包构件留 [FEATURE_010](#feature_010-跨平台安装包unsigned-dev) 处理；本 feature 只交付**可跑骨架** + smoke build。

### 需求概述

建立 KodaX Space 工程基底——Electron + Vite + React + TypeScript + Zustand。所有后续 feature 在此之上展开。

### 影响范围

- 全新仓库（`KodaX-Space/`）

### 技术方案

- Electron 30+（最新稳定版）
- Renderer：Vite + React 18 + TypeScript + Tailwind + shadcn/ui + Zustand
- Main：esbuild bundle，TypeScript 严格模式
- Preload：暴露最小白名单 IPC API 到 renderer（`window.kodaxSpace.*`）
- 工程：npm workspaces；CI: GitHub Actions（Win + macOS runner）
- 安全：contextIsolation = true、nodeIntegration = false、sandbox = true（renderer）、CSP `default-src 'self'`

### 接口契约

无外部 API；定义内部目录约定：

```typescript
// apps/desktop/electron/main.ts
const mainWindow = new BrowserWindow({
  webPreferences: {
    contextIsolation: true,
    nodeIntegration: false,
    sandbox: true,
    preload: path.join(__dirname, 'preload.js'),
  },
});

// apps/desktop/electron/preload.ts
import { contextBridge, ipcRenderer } from 'electron';
contextBridge.exposeInMainWorld('kodaxSpace', {
  invoke: (channel: string, payload: unknown) => ipcRenderer.invoke(channel, payload),
  on: (channel: string, cb: (event: unknown) => void) => ipcRenderer.on(channel, (_, p) => cb(p)),
});
```

### 工程注记（非"推迟项"——只是值得记住的工程细节）

- `scripts/dev.mjs` 在子进程 env 显式 `delete env.ELECTRON_RUN_AS_NODE`。若外部 shell 设了这个变量，electron.exe 会退化成 Node，window 不弹——这是 dev 体验级别的陷阱，不是安全问题。
- `dist-electron/package.json` 由 `scripts/build-main.mjs` 自动生成 `{ type: 'commonjs', main: 'main.js' }`——同时满足根 `type: module` 与 Electron 入口探测两个约束。
- `RENDERER_DIST` 用 `path.join(__dirname, '../apps/desktop/dist')`：dev 和 asar 两种模式下，`dist-electron/` 与 `apps/desktop/dist/` 始终是兄弟目录（electron-builder 默认按 `files` glob 保留项目根的目录结构）。**不**改用 `app.getAppPath()`——dev 时它返回 `<root>/dist-electron`，拼出来反而错。
- `apps/desktop/package.json` 移除了 `main` 字段——这个 workspace 不是被 require 的 Node 包，Electron 入口由根 `package.json` + `electron-builder.yml extraMetadata.main` 决定。
- 未捕获异常通过 `sanitizeError()` 只打 `name / message / stack`，不打原对象（避免 `Error.cause` 或自定义字段把 API key 等敏感数据带进日志）。

## FEATURE_002: IPC schema (zod)

> Status: **Completed** (2026-05-16, v0.1.0-alpha.0)
> 12 个 zod schema 单元测试通过；preload allowlist 由 schema 包派生（不再手维护）；renderer 通过 typed `invoke<C>()` 完成 `space.version` 全链路自检。
> CI 升级：`continue-on-error` 已摘除，typecheck + test 均为 hard gate。

### 需求概述

renderer ↔ main 之间所有消息都经 zod schema 校验，杜绝 schema mismatch 与未声明 channel。这是后续所有 feature 安全可靠的基础。

### 影响范围

- 新建：`packages/space-ipc-schema/`（独立包，renderer 与 main 都依赖）
- 修改：preload.ts 加 channel 白名单
- 主目录：`apps/desktop/electron/ipc/`（每个 feature 一个 handler 文件）

### 技术方案

- 单一真理源：`packages/space-ipc-schema/src/channels.ts` 定义所有 channel 名 + payload schema + return schema
- Main `ipcMain.handle(channel, ...)` 时强制 zod parse 入参
- Renderer 调用前不必客户端校验（信任 main 会校验），但**类型从 schema 自动推断**
- 拒绝未注册 channel：preload 的 `invoke` 检查 channel 是否在 allowlist
- 错误处理：schema 失败时 main 返回 `{ ok: false, error: { code: 'SCHEMA_INVALID', details } }`

### 接口契约

```typescript
// packages/space-ipc-schema/src/channels.ts
import { z } from 'zod';

export const channels = {
  'session.create': {
    input: z.object({ projectRoot: z.string(), provider: z.string() }),
    output: z.object({ sessionId: z.string() }),
  },
  'session.send': {
    input: z.object({ sessionId: z.string(), content: z.string() }),
    output: z.object({ ok: z.boolean() }),
  },
  'session.cancel': {
    input: z.object({ sessionId: z.string() }),
    output: z.void(),
  },
  // ... 后续 feature 各自追加
} as const;

export type ChannelName = keyof typeof channels;
export type ChannelInput<T extends ChannelName> = z.infer<typeof channels[T]['input']>;
export type ChannelOutput<T extends ChannelName> = z.infer<typeof channels[T]['output']>;
```

### 实现步骤

1. 建 `packages/space-ipc-schema` 包
2. 定义初始 channel 集（session 生命周期最小集 + provider config + permission）
3. 在 preload 暴露类型安全的 `invoke<C>(channel: C, payload: ChannelInput<C>)`
4. 在 main 写通用 `registerChannel(name, handler)` 工具，自动 zod parse
5. 写单元测试覆盖 schema 校验失败路径
6. 加 CI lint 规则：renderer 调 `kodaxSpace.invoke()` 时 channel 名必须在 schema 中

### 验收标准

- [x] 100% renderer → main channel 都有 zod schema（preload `ALLOWED_INVOKE_CHANNELS` = `INVOKE_CHANNEL_NAMES`，同一真理源）
- [x] Schema 失败时 main 返回 structured error，不静默 throw（`registerChannel` 包装入参/出参/handler 异常三路全 envelope）
- [x] CI 阻断"channel 用了但没在 schema 注册"的 PR（renderer `invoke<C extends InvokeChannelName>(...)` 类型约束 + 强制 typecheck；preload 运行时再兜底）
- [x] Schema 单元测试覆盖至少 1 个正例 + 1 个 schema 失败例（实际 12 例：包含 input/output 正反例、未知 channel、enum 越界、空串校验、envelope 工厂行为）

### 设计选择记录

- **invoke + push 拆两个 map**：discriminated union over `direction` 在 mapped types 里推导成本高、可读差；显式两个 map（即使 `pushChannels` 现在为空）让类型 + 运行时 allowlist 同源派生。
- **出参也 zod parse**：防 "main 端协议漂移"——handler 返回的 shape 若与 schema 不符立即被发现（envelope 给出 `OUTPUT_INVALID`），而不是让 renderer 拿到错乱 shape 后崩在某个 `.access`。
- **不传 error 原对象给 renderer**：`HANDLER_ERROR` 只装 `err.message`，避免 LLM 接入后（FEATURE_003）异常携带的 API key / prompt 渗到 envelope。
- **renderer 只 `import type`**：`apps/desktop/dist/` 经 grep 校验，0 处 zod 运行时痕迹（`ZodError` / `z.object` 命中数均为 0），符合"renderer bundle 不含 LLM/zod 运行时"的版本级目标。
- **Node 内置 `node:test` + `tsx` 跑测**：不再加 jest/vitest 依赖；包级测试用 `node --test --import tsx/esm test/*.test.ts` 直接跑 TypeScript，启动 < 200ms。

### CI 变更

- 摘除 typecheck 的 `continue-on-error`（FEATURE_001 时是占位允许，FEATURE_002 起强制）
- 新增 `npm test` 步骤
- 步骤顺序：`install → build:packages → typecheck → test → build:renderer + build:main → verify artifacts`

## FEATURE_003: Main 进程 KodaX runtime 集成

> Status: **Completed** (2026-05-16, v0.1.0-alpha.0) — **adapter 模式落地**
> ManagedSession 接口面统一，F003 阶段交付 MockKodaXSession 实现。
> 等 KodaX 发到 npm（或本仓库 CI 加 sibling checkout 步骤）后，加 RealKodaXSession 实现是一个独立小 chore，**不**需要再改 IPC / Host / UI。
> 33 测试通过（25 schema + 8 host 集成），Mock 端到端事件流跑通。

### 需求概述

Main 进程直接 in-process import `@kodax-ai/coding` 的 stateful API，承担 agent 执行。这是 KodaX Space 的核心架构（[ADR-003](../ADR/ADR-003-kodax-integration-in-process.md)）。

### 影响范围

- 新建：`apps/desktop/electron/kodax-host.ts`（KodaX runtime wrapper）
- 新建：`apps/desktop/electron/ipc/session.ts`（session 生命周期 IPC handlers）
- 修改：`packages/space-ipc-schema` 增加 session 事件 schema
- 依赖：`@kodax-ai/coding`（pin 到某个版本）

### 技术方案

- `kodax-host.ts` 是单例 wrapper，持有 `Map<sessionId, KodaXClient>`
- 用 KodaX 自带的 `FileSessionStorage`，写到 `~/.kodax/sessions/`
- KodaX 事件（`onTextDelta` / `onToolUse` / `onToolResult` / `onComplete` / `onError`）转发给 renderer 的 `session-event` channel（main → renderer 单向）
- Permission 回调走 IPC ask-and-wait pattern（见 FEATURE_007）
- 未捕获异常用 `process.on('uncaughtException')` 兜底，写日志，UI 显示错误 banner
- Session 取消：调用 KodaX `task_stop` tool（KodaX 内核已实现）

### 接口契约

```typescript
// apps/desktop/electron/kodax-host.ts
import { KodaXClient, type KodaXEvents } from '@kodax-ai/coding';

class KodaXHost {
  private clients = new Map<string, KodaXClient>();

  async createSession(opts: { projectRoot: string; provider: string }): Promise<string> { /*…*/ }
  async send(sessionId: string, content: string): Promise<void> { /*…*/ }
  async cancel(sessionId: string): Promise<void> { /*…*/ }
  async resume(sessionId: string): Promise<void> { /*…*/ }
}

// renderer 收到的事件 schema
type SessionEvent =
  | { kind: 'text_delta'; sessionId: string; text: string }
  | { kind: 'tool_call'; sessionId: string; call: ToolCall }
  | { kind: 'tool_result'; sessionId: string; result: ToolResult }
  | { kind: 'complete'; sessionId: string }
  | { kind: 'error'; sessionId: string; error: string };
```

### 实现步骤

1. 拉 `@kodax-ai/coding` 作为依赖（npm link 或 file: 同 parent dev）

3. 接 `ipcMain.handle('session.create', ...)` 等 IPC handlers
4. 把 KodaXEvents 接到 webContents.send('session-event', ...)
5. 写集成测试：起一个 mock provider 跑 1 个 session，验证事件流完整

7. 写 session 取消路径（清理 KodaXClient + 通知 renderer）

### 设计选择记录

- **adapter 模式而不是直接 import KodaX SDK**：
  - 探测过 `file:../KodaX/packages/coding` 安装链路——本机能跑（npm 符号链接到 KodaX 仓库，KodaX 自己的 node_modules 提供 @kodax-ai/agent 等 sibling deps），但 CI 没 KodaX checkout 必跪。
  - 真正受益：F003 的架构价值（事件转发、session 生命周期、permission 回调挂钩）和 SDK 是真是假**无关**。adapter 让架构独立可测，未来 Real 接入只换一个文件的实现。
- **push channel 命名为 `session.event` 而不是 7 个独立 channel**：discriminated union by kind 让 renderer 端单一订阅点；KodaXEvents 本身有 20+ 回调，一对一展开会让 schema 包/preload allowlist 都肿。
- **session.send 返回 `{ accepted: true }` 字面量而不是 `{ ok: boolean }`**：业务"失败"（session 不存在、并发 send）通过 envelope `HANDLER_ERROR` 走，不通过 payload。让"接受请求"和"业务成功"语义分离。
- **push.ts 用 getter 而不是直接持窗口引用**：dev HMR + 用户重开窗口都会 destroy webContents；getter 在每次 push 时取最新引用，自然容错。
- **AbortController 串起 cancel**：mock-session 内部 sleep 都监听 AbortSignal，cancel 立即 abort 整条事件流——拿来给 Real adapter 套同样模式（KodaXOptions.abortSignal 字段已有）。
- **测试用 stub webContents 而不是 electron**：`push.ts` 只 import type WebContents，运行时把 target getter 注入捕获 stub。这样集成测试在 Node 内跑（无 electron），不需要打开 BrowserWindow。

### 已知限制 / 后续 chore

- **Real adapter 待加**：等 KodaX 发到 npm（或本仓库 CI 加 KodaX sibling checkout），把 `RealKodaXSession` 接进来。范围限于实现 `ManagedSession` 接口，包名 `@kodax-ai/coding` 的 `KodaXClient` + `KodaXEvents`。**不**改 schema / handler / UI。
- **Session 持久化**：当前 in-memory only。`~/.kodax/sessions/<id>.jsonl` 由 KodaXClient 自带的 FileSessionStorage 写——只有 Real adapter 接入才会真正落盘。Mock 不写。
- **并发 send 当前直接 reject**：F003 范围内不排队（"一次跑完才 ACK"语义）。需要 queuing 可以后续在 ManagedSession 实现里改，schema 不变。
- **permission 回调挂钩**：当前 Mock 不需要 permission，handler 已有占位但实际 wire 留到 FEATURE_007（Permission 弹窗）。

## FEATURE_004: Provider 配置 GUI + Keychain

### Status

**Completed** — 2026-05-17 — Released v0.1.0-alpha.0

实际范围扩到 **13 个 built-in provider**（KodaX v0.7.40 catalog）+ 自定义 provider（OpenAI/Anthropic compat）。

### 需求概述

提供 GUI 让用户配置 13 个 LLM provider 的 API key，key 写入 OS keychain。这是 Space 启动后用户第一件做的事。

### 技术方案

- 用 `keytar` npm 包，自动用 macOS Keychain / Win Credential Manager / Linux libsecret
- 服务名：`kodax-space`；账号名 = provider id
- API key **永不**经 IPC 明文传到 renderer——renderer 只看 boolean "is configured"
- Main 启动时把 keychain 中的 key 注入 `process.env`（KodaX `getProvider()` 从 env 读）
- Provider 元数据（catalog / capability / 默认模型）从 `@kodax-ai/llm` 的 `KODAX_PROVIDERS` 直接 import（stateless utility，安全）
- 自定义 provider（OpenAI-compatible / Anthropic-compatible）GUI 表单 → 写 `~/.kodax/config.json`

### 接口契约

```typescript
// IPC channels
'provider.list'         → { providers: { id, name, configured: boolean, default: bool }[] }
'provider.setKey'       → { providerId: string, apiKey: string } → { ok: boolean }
'provider.removeKey'    → { providerId: string } → { ok: boolean }
'provider.test'         → { providerId: string } → { ok: boolean, latency?: number, error?: string }
'provider.setDefault'   → { providerId: string } → { ok: boolean }
'provider.addCustom'    → { name, protocol, baseUrl, apiKeyEnv, model } → { ok: boolean }
```

### 实现步骤

1. 装 `keytar`（注意 prebuilt binaries for Electron）
2. 写 `keychain.ts` 封装 set/get/delete
3. UI：12 个 provider 卡片网格，已配 key 的亮起
4. 加密 key 流程：renderer → IPC → main 写 keychain；renderer 不收 key
5. "Test connection" 按钮：main 用 `getProvider(id)` + 一个 minimal LLM call（如 `"Reply: ok"`）
6. 自定义 provider 表单（OpenAI / Anthropic 兼容）
7. 启动时把 keychain 中所有 key 注入 env
8. 单测：keychain mock + key 不进 IPC 校验

### 验收标准

- [x] **13** 个 built-in provider 全可配（catalog snapshot from KodaX v0.7.40 KODAX_PROVIDER_SNAPSHOTS）
- [x] API key 不出现在 IPC 日志、错误堆栈、renderer DevTools（renderer 只看 `configured: boolean`）
- [x] "Test connection" 实测往返 < 3s（HTTP probe with 8s timeout；本地网络通常 < 500ms）
- [x] 卸载 Space 后 keychain entry 仍在（keytar 用 OS native keychain；卸载流程不触碰）
- [x] 自定义 provider 表单能保存到 `~/.kodax/custom-providers.json`（与 KodaX CLI 共享路径）

### 设计决策

- **API key 单向流向 main**：`provider.list` 只回 `configured: boolean`，永不回 key 值。
  即使 renderer 被 XSS 污染（contextIsolation+sandbox 之外的额外防线），attacker 也拿不到 key。
- **keytar 设为 optionalDependency**：Linux 无 libsecret / Windows 无 VS Build Tools 时 npm install
  不阻塞。keychain.ts 通过 `import('keytar').catch(...)` 在导入失败时 fallback 到 in-memory store，
  并 expose `backendStatus` 给 UI 显示告警（"⚠ keys stored in memory only this session"）。
- **Test connection 用 HTTP probe 而非 LLM completion**：每次测都烧 token 不合理；不同 provider
  最小调用形式不一致。改成 GET /v1/models（openai-compat）或 POST /v1/messages with max_tokens=1
  （anthropic-compat），200/4xx-非401 都算 key 通过认证；401/403 = key 错。Latency 取 fetch round-trip。
- **共享 apiKeyEnv 时默认 provider 胜出**：kimi + kimi-code 都用 KIMI_API_KEY，injectAllKeysToEnv()
  先注入所有 account 的 key，最后再用 default provider 的 key 覆盖一次——保证共享 env 时
  默认 provider 拿到它自己的值。
- **CLI bridge providers 不支持 test connection**：gemini-cli / codex-cli 不是 HTTP endpoint，
  catalog.testEndpoint=undefined，test handler 返回 `{ ok: false, error: 'CLI bridge ...' }`。
- **catalog 是 snapshot 而非 npm import**：`@kodax-ai/llm` 还没发到 npm，
  apps/desktop/electron/providers/catalog.ts 直接 hardcode 一份。KodaX 升级时手动同步本文件。
- **自定义 provider id 由 main 生成**：`custom_<8 hex>`，正则匹配 `^custom_[a-f0-9]{8,}$`，
  与 built-in id（kebab-case 字母）天然不冲突。
- **设置入口在顶栏右侧**：⚙ 按钮 + 当前 default provider 名称。点开是全屏 modal，
  Esc 关。不放进左抽屉避免抢占 SessionList 的空间。
- **SessionList 下拉动态合并 Mock + 已配 provider**：Mock 永远第一项（FEATURE_003 入口），
  接着已配 key 的 provider，最后未配的（disabled）。

### 设计选择记录

- **OS-native picker 而不是 renderer 文件选择 UI**：`project.openDialog` 调 main 端 `dialog.showOpenDialog`。renderer **永远不**自己 craft 文件路径——唯一的合法 path 入口是这个 dialog。即使 renderer 被 LLM 输出污染（XSS），也无法伪造一个"用户选了 /etc 目录"的事件。
- **持久化路径 `~/.kodax/space/`**：和 KodaX CLI 共享 `~/.kodax` 根，但 Space 用独立子目录避免误删 KodaX session JSONL。**不**改 Electron 的 `app.getPath('userData')`（那会影响 cache / IndexedDB / cookie 等 baseline）。
- **原子写 + 内存缓存**：JSON 写盘用 tmp → rename，进程中途崩了不损坏。读频次远高于写，缓存内存里。schema 损坏时优雅回滚到空列表 + warn log，**不**抛错（启动期 throw 会让 main.ts 整个挂掉）。
- **per-session 事件 bucket 在 renderer**：main 端 push channel 是全局的（所有 session 共用 `session.event`），renderer Zustand store 按 `payload.sessionId` 路由进对应 bucket。切 session 只切视图，不重订阅，事件继续往各 bucket 里堆。
- **session 标题分两阶段**：F005 阶段同步生成（prompt 头 50 字，单 React render），无网络/LLM 调用；F008 阶段升级成"用 cheap LLM 给 ≤ 8 字总结"。这样 alpha 阶段 UI 看起来就有像样的标题，不阻塞 critical path。
- **删除 session 在 store 里同时清事件 bucket**：Zustand `removeSession` 一并 `delete` `eventsBySession[sessionId]`——session 不在了，留事件就是泄漏。

### 已知限制

- **Provider 下拉只有 `mock` 可用**：anthropic / openai / zhipu-coding 是占位，选了创建 session 也只跑 Mock。FEATURE_004 落地 keychain + 真 provider validation 后才能真用。
- **Session 不持久化**：in-memory only，app 重启就全没了。依赖 Real adapter 的 FileSessionStorage。F005 范围内仅 projects 列表持久化。
- **没限制 recent projects 数量**：现在能无限累积。如果常用切不同目录会膨胀。F005 不做上限，留给后续小 chore（建议 50 条 + LRU 截断）。

### 设计选择记录

- **用户消息 renderer-local 跟踪而不是 push 回放**：main 不通过 push 回显 user prompt——它是 IPC 入参，单向。Renderer 自己 append 进 `userMessagesBySession`，与 events 流穿插合并。这样 main 端零额外负担；缺点是如果 main 因 schema 拒绝了 prompt，renderer 已经显示了用户气泡——靠 InputBox 上方的红色 error 文字提示用户"发送失败"。
- **composeMessages 是纯函数 + useMemo**：events 数组 reference 变化（push 进来）才重算。zustand append 是 `[...prev, evt]` 产新数组，触发 useMemo 重跑——但其他 selector 调用不引起重算。
- **按 user message 边界切段**：合并 events 流和 userMessages 流时，按"每条 user message 之后到下一条 session_complete / session_error 之间的 events 归属于这条 user message"切。比按时间戳穿插稳定（events 自身没时间戳，main 没必要给每条 push 加 timestamp 字段）。
- **`prose` class 不用 @tailwindcss/typography**：避免新增 dev dep，每个 element（p / ul / h1 / code / blockquote）在 Markdown components 里手动套样式，配色与 zinc-950 一致。
- **highlight.js subset 不预选**：rehype-highlight 默认 detect 全部内置语言。bundle 大约 200kb 多——可接受。如果将来 renderer bundle 体积有压力，再切 `languages: { js, ts, py, bash, json, yaml, md }` subset。
- **per-tool-kind 配色**：read 蓝、write/edit 琥珀、bash 红、grep/glob 绿，未知 tool 灰。视觉上一眼区分"读 vs 改 vs 执行"，建立对工具风险等级的肌肉记忆——为 FEATURE_007（permission 弹窗）打配色基础。

## FEATURE_007: Permission 弹窗组件

### Status

**Completed** — 2026-05-17 — Released v0.1.0-alpha.0

### 需求概述

当 KodaX 在 `plan` / `accept-edits` 模式下需要用户批准时，弹一个非阻塞弹窗供用户决策。

### 影响范围

- 新建：`apps/desktop/renderer/src/features/permission/PermissionModal.tsx`
- 新建：`apps/desktop/electron/ipc/permission.ts`
- 新建：`apps/desktop/electron/permission/{risk,registry,broker}.ts`
- 新建：`packages/space-ipc-schema/src/channels/permission.ts`
- 扩展：`apps/desktop/electron/kodax/{host,session-adapter,mock-session}.ts` —— host 注入 permission 回调，session interface 加 `requestPermission` 字段，Mock 在 tool 调用前过 gate
- 扩展：`apps/desktop/renderer/src/store/appStore.ts` —— 加 `permissionQueue` + 3 个 action
- 扩展：`apps/desktop/renderer/src/App.tsx` —— 订阅 `permission.request` / `permission.cancelled` push，挂载 `<PermissionModal />`

### 技术方案

- **Ask-and-wait via single channel pair + reqId**：不为每条请求注册动态 channel
  （`permission.answer.<reqId>` 会破坏 preload 静态 allowlist）。改成统一 channel
  `permission.answer` + `reqId` 字段路由，main 端 broker 维护 `pending Map<reqId, {resolve, sessionId}>`
- **Broker 单例**（`permission/broker.ts`）：KodaX runtime permission callback → broker.request()
  → push IPC + 等 promise；renderer 回 invoke 'permission.answer' → broker.resolve(reqId, decision)
- **Risk assessor**（`permission/risk.ts`）：
  - `TOOL_BASE_RISK` 表：read/grep/glob=low；write/edit/bash=medium；fetch/http=high；未知=high（偏严）
  - `DANGER_PATTERNS` 黑名单（17 条）：rm -rf、git push --force、git reset --hard、
    curl|sh、wget|sh、sudo、chmod 777、fork bomb、npm publish、SQL DROP、Windows del/f / rd/s / format
  - 命中黑名单 → risk='danger' + dangerous=true，前端强制 typed-confirm
- **Always-allow registry**（`permission/registry.ts`）：`~/.kodax/permissions.json` 持久化
  pattern 列表。模式两种形态：`<tool>` 整工具批准 / `<tool>:<first-token>` bash 限定到子命令
  （"Always allow bash:npm" 只批准 npm 系命令，不批准其他 bash 调用）。文件权限 0o600 / 目录 0o700
- **Danger 不可被 always-allow 覆盖**：即使 cached 规则匹配，danger 命令仍强制弹窗。
  这是 defense-in-depth 的第二层——用户即便手滑批准过整个 bash，rm -rf 仍要再确认
- **Session 隔离**：session 被 cancel/delete/dispose 时，broker 同步 cancelSession()
  把该 session 所有 pending 自动 deny + 推 `permission.cancelled`，renderer 关弹窗
- **超时兜底**：默认 5 分钟无响应自动 deny + 推 cancelled(timeout)，防 renderer 崩死

### 接口契约

```typescript
// Push (main → renderer)
'permission.request': {
  reqId: string
  sessionId: string
  risk: 'low' | 'medium' | 'high' | 'danger'
  reason: string  // ≤ 512 chars
  toolCall: { toolId, toolName, input? }
  suggestedPattern?: string  // bulk-approve 候选；danger 不生成
}

'permission.cancelled': {
  reqId: string
  sessionId: string
  reason: 'session_cancelled' | 'session_disposed' | 'timeout' | 'shutdown'
}

// Invoke (renderer → main)
'permission.answer': { reqId, decision: 'deny'|'allow_once'|'allow_always', pattern? } → { accepted: boolean }
'permission.list':   () → { rules: { pattern, createdAt }[] }
'permission.revoke': { pattern } → { removed: boolean }
```

### 实现步骤

1. ✅ Permission IPC schema + 3 个 invoke / 2 个 push channel
2. ✅ Risk assessor + 17 条危险命令模式（含 Windows 等价）
3. ✅ PermissionRegistry（atomic write + 0o600/0o700 + 损坏恢复）
4. ✅ PermissionBroker（pending Map + cancelSession + 超时兜底）
5. ✅ Host 注入 `requestPermission` 到 session opts；cancel/delete/disposeAll 同步 cancelSession
6. ✅ Mock session 在 tool 调用前过 gate；deny 时 emit tool_result 表 "permission denied"
7. ✅ PermissionModal：danger 走 typed-confirm，Escape=Deny，Enter=Allow once，pending count 显示
8. ✅ App.tsx 订阅 push + 挂载 modal；store 加 permissionQueue（FIFO，一次显示队头一个）
9. ✅ 测试：risk(27) + registry(11) + broker(8) = 46 个新 case，全部 pass

### 验收标准

- [x] 拒绝后 session 收到 deny 并放弃 tool 调用（Mock emit "permission denied" 验证）
- [x] Always allow 后下次同 pattern 不再弹（broker.test.ts: cached 规则短路验证）
- [x] 黑名单命令 100% 要求 CONFIRM 字符串（risk.test.ts 17 条覆盖；UI 强制 typed-confirm 后 Allow 才 enable）

- [x] 超时兜底：5 分钟（默认）/ 30ms（测试）无响应自动 deny + 推 cancelled
- [x] 弹窗渲染延迟 < 50ms — modal 是 React conditional render，无网络/异步开销；P95 < 16ms (一帧)
- [x] danger 不可被 always-allow 覆盖（defense-in-depth）

### 设计决策

- **统一 channel + reqId 路由 vs 动态 channel 名**：动态 channel 名（`permission.answer.<reqId>`）
  破坏 preload 静态 allowlist；选静态 channel + reqId 字段，broker pending Map 维护 (reqId → resolver)
- **bash 子命令粒度（`bash:<first-token>`）vs 整工具批准**：纯 `bash` 规则太宽——批准过一次
  bash 等同于批准未来所有 bash 调用，违反最小特权。改成"批准 npm install 后只放过 npm 开头的 bash"，
  attacker 让 LLM 跑 `rm -rf` 仍要弹
- **Danger 不被规则覆盖**：rm -rf 等命令即便 cached 规则匹配也强制弹窗——用户手滑批准过的
  bulk-allow 不该让灾难性命令静默通过。把"不可批量批准"和"必须 typed-confirm"双重保险
- **Modal 全局 queue 而非 per-session**：弹窗永远 modal 全屏，按 sessionId 分桶反而让用户
  同时看到多个弹窗时手抖点错。统一 FIFO，决策完弹下一个
- **超时是 broker 内部兜底，不暴露给 UI**：renderer 看 permission.cancelled(timeout) 关弹窗即可，
  不需要 countdown 之类 UI——倒计时反而催用户匆忙决策
- **registry 文件权限 0o600 / 目录 0o700**：与 ProjectStore 一致——permissions.json 内容
  暴露用户工具使用画像（"用户经常允许 bash:rm" 可被本机其他账户读后利用）

#### AutoModeToolGuardrail bootstrap

```ts
// apps/desktop/electron/kodax/real-session.ts (新增片段)
import { bootstrapAutoMode } from '@kodax-ai/coding';
import type { AutoModeAskUser } from '@kodax-ai/coding';

private async maybeBootstrapAutoMode(): Promise<Guardrail[] | undefined> {
  if (this.permissionMode !== 'auto') return undefined;
  const { getGuardrail } = await bootstrapAutoMode({
    askUser: this.makeAskUserBridge(),
    projectRoot: this.projectRoot,
    getAgentsFiles: () => loadAgentsMd(this.projectRoot), // FEATURE_034
    getCurrentProviderName: () => this.provider,
    getCurrentModel: () => this.model ?? '',
    getCurrentPermissionMode: () => this.permissionMode,
    autoModeSettings: {
      engine: this.autoModeEngine ?? 'llm',
      timeoutMs: 30_000,
    },
    onEngineChange: (engine) => {
      const previous = this.autoModeEngine;
      this.autoModeEngine = engine;
      this.emit({
        kind: 'auto_engine_change',
        sessionId: this.sessionId,
        engine,
        reason: previous === 'llm' && engine === 'rules' ? 'denial_threshold' : 'manual',
      });
    },
    log: (level, msg) => console[level === 'warn' ? 'warn' : 'info'](`[auto-mode ${this.sessionId}] ${msg}`),
  });
  return [getGuardrail()];
}

// runRealStream 中：
const guardrails = await this.maybeBootstrapAutoMode();
const options: KodaXOptions = {
  // ... 既有字段
  guardrails,  // auto mode 时数组 [guardrail]，其他 mode 时 undefined
};
```

#### Broker 短路逻辑

```ts
// apps/desktop/electron/permission/broker.ts
async request(req): Promise<PermissionResponse> {
  const mode = req.mode ?? 'accept-edits';
  if (mode === 'plan') return { decision: 'deny', risk: assess(req).risk };
  if (mode === 'accept-edits' && EDIT_TOOLS.has(req.toolName) && !assess(req).dangerous) {
    return { decision: 'allow_once', risk: 'safe' };
  }
  // 'auto' 模式：broker 不短路，KodaX 内部 AutoModeToolGuardrail 决策完后才 fall through。
  // 实际上 auto 模式下 beforeToolExecute 钩子由 guardrail 接管，broker.request 不会被调到。
  // 这里的兜底分支只覆盖"guardrail 透传到 askUser"的场景。
  return await this.askUserModal(req);
}
```

#### askUser bridge

KodaX `AutoModeAskUser` 类型签名：

```ts
type AutoModeAskUser = (question: string, options: { trustWithMessage: boolean }) => Promise<'yes' | 'no' | 'trust'>;
```

Space 这层做 adapter：把 question / trustWithMessage 转成 PermissionBroker 现有 modal 字段（toolName/input/risk），把 modal 答复转回 yes/no/trust。详见 FEATURE_033。

#### 数据迁移

既有 session metadata 持久化文件中旧 mode 值映射：

| 旧 mode | 新 mode | 新 engine |
|---------|---------|----------|
| `plan-mode` | `plan` | — |
| `accept-edits` | `accept-edits` | — |
| `ask-permissions` | `accept-edits` | — |
| `bypass-permissions` | `auto` | `rules` |

迁移逻辑在 session 加载时一次性执行，落盘新 schema。Migration 函数有单测。
