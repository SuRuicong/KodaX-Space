# v0.2.x - Desktop-Native Expansion Plan

> Status: Planned
> Source of truth: this file + [../FEATURE_LIST.md](../FEATURE_LIST.md)

## Batch Intent

v0.2.x starts after Space has caught up with the KodaX runtime surface. The goal is to build capabilities that are natural for a desktop host:

- long-lived Partner workspaces,
- Space-owned office deliverables that do not require Partner project writes,
- Partner workbench flows that consume the earlier F113 controlled workspace file-write path,
- governed connectors,
- local automations,
- refreshable live artifacts,
- audit and policy controls,
- optional remote runners,
- notebook/data review,
- guarded desktop automation feasibility,
- legacy continuity remainder hooks for old SDK/CLI-gated gaps,
- distribution/platform maturity,
- extension points for future surfaces.

The boundary remains the same: **Space does not become a cloud service, IDE, or independent runtime.**

Carry-over from shipped emergency versions:

- v0.2.0 takes F049, F070-F074, F095, and F109 as the Partner re-enable foundation.
- F113 lands earlier in `v0.1.30` and is consumed here as a prerequisite write path.
- v0.2.1-v0.2.2 are reserved patch version numbers after the Partner foundation lane.
- v0.2.3 takes F050-F052 plus F073/F080/F096/F097 for browser-backed research, document skills, connectors, automations, and assistant response language preference.
- v0.2.4-v0.2.5 are reserved patch version numbers after the browser/connectors/automations lane.
- v0.2.6 takes F053/F075/F079/F098/F110 for Partner trust, knowledge graph, admin policy, localization follow-up, and refreshable live artifacts.
- v0.2.7-v0.2.8 are reserved patch version numbers after the trust/live-artifact lane.
- v0.2.9 takes F042/F055/F067/F099-F102/F111/F112 for native helper parking lot, app protocol hardening, LiveCanvas re-integration gate, remote/data/platform work, slot registry, guarded desktop automation feasibility, and legacy continuity remainder hooks. F105 lands earlier in v0.1.24; any full legal/distribution follow-up can feed into F101.
- v0.2.10-v0.2.11 are reserved patch version numbers after the platform/runner/data lane.

## External Gap Review Adjustment (2026-06-19)

The post-`v0.1.19` competitive review adds four planning constraints for `v0.2.x`:

- **PR and GitHub collaboration closure belongs here.** It should be implemented through the connector and Partner workbench path: issue/PR snapshots, review briefs, PR descriptions, branch/commit evidence, and optional create/open PR actions. It should not become a separate cloud-agent product.
- **Enterprise governance is local-first first.** `F098` covers policy files, allowlists, redaction, and local audit export. SSO/SAML/SCIM are intentionally out of scope until there is an explicit enterprise distribution track.
- **Remote runner is opt-in infrastructure, not a default cloud.** `F099` may cover SSH/Docker/self-hosted runners and long-task monitoring, with clear data-boundary warnings. It should not weaken the local-first default.
- **Commercial licensing must not punish community use.** `F105` lands earlier in `v0.1.24`; v0.2.x should treat it as a distribution constraint, not as a reason to add startup nagging or split the source tree.

Semantic codebase indexing is also not pulled into `v0.2.x` as a standalone Cursor-style feature. Partner KB search can evolve from markdown/FTS to hybrid search (`F071`), while code intelligence remains KodaX/Repointel-owned unless a future SDK contract requires a Space surface.

## Cowork Gap Review Adjustment (2026-07-05)

The Partner surface is intentionally still disabled because the current capability chain cannot produce finished office deliverables or user-visible files safely: Partner can call read/search/web/source/KB/artifact tools, but its policy blocks project `write`/`edit`/`bash`, while the current office artifact path only references already-existing files for `pdf`/`docx`/`xlsx` and has no `pptx` writer. The v0.2.x plan treats this as the first parity blocker, not as a UI polish item.

Planning updates:

- **Partner re-enable gate moves to `v0.2.0`.** `F109` adds Space-owned `docx`/`xlsx`/`pdf`/`pptx` writers and export UX; `F095` wires those outputs plus the earlier `v0.1.30` F113 reviewed workspace file proposals into a real workbench flow. `PARTNER_ENABLED` should stay false until these features pass end-to-end acceptance.
- **Raw write/edit/bash remains outside Partner.** `F113` is not a rollback of `F047`: Partner can propose and apply files through Space-owned review UX, but it still does not receive unrestricted project `write`, `edit`, or shell tools.
- **Unimplemented Cowork-style capabilities move behind patch gaps.** Feature-bearing lanes are now `v0.2.0`, `v0.2.3`, `v0.2.6`, and `v0.2.9`; every lane is followed by two patch-reserve version numbers.
- **Connectors and automations follow the foundation.** Browser-backed research, connector snapshots, OAuth/token visibility, GitHub issue/PR loops, and local schedules move to `v0.2.3`, after Partner has a reliable deliverable path.
- **Live artifacts are a later Partner surface, not the unblocker.** `F110` targets refreshable dashboards/reports using existing artifact sandboxing and source snapshots in `v0.2.6`; it does not wait on the LiveCanvas React tier.
- **Computer-use parity is guarded feasibility.** `F111` goes to `v0.2.9` and must prefer connector APIs and browser tools before any visible screen-control fallback. It is default-off and policy/audit gated.
- **Historical partials get a real parking feature.** `F112` goes to `v0.2.9` for the remaining F015/F017/F018 hooks: public Repointel warm API, true no-session Quick Ask sideQuery/Partner promotion, and CLI/REPL handoff writer integration. It is not a Partner re-enable prerequisite.
- **Local-first remains the product boundary.** Hosted control planes, broad cloud-agent execution, unrestricted screen control, SSO/SAML/SCIM, and silent local-file mutation remain non-goals for this v0.2.x batch.

## FEATURE_109: Partner Deliverable Chain and Office Artifact Writers

### Problem

Partner cannot be re-enabled as a credible Cowork-style surface while it cannot produce finished deliverables. Today the safe Partner tool policy blocks project writes and shell execution, while the artifact tool only creates office artifacts by referencing an existing workspace path for `pdf`/`docx`/`xlsx` and has no `pptx` kind. That leaves Partner able to talk about reports/decks/sheets, but not reliably hand the user a finished file.

### Design

Add Space-owned deliverable writers that create files inside the artifact store first, then export only by explicit user action:

- extend artifact content schemas and renderer support for generated `docx`, `xlsx`, `pdf`, and `pptx`,
- support structured writer payloads: document outline/styles/assets, workbook sheets/formulas/charts, PDF render source, slide deck layout/speaker notes,
- implement the writers in the desktop host using bundled document/spreadsheet/slide/PDF libraries rather than Partner `write` or `bash`,
- keep generated files under `~/.kodax/space/artifacts` with manifest metadata, source refs, citations, content hash, preview status, and export history,
- expose a Partner-safe tool path, either by extending `create_artifact` with writer payloads or adding a narrow `create_document_artifact` family,
- add export UX for Save As / reveal / attach back to conversation, with overwrite and workspace mutation always explicit,
- add previews in ArtifactPanel for the new generated office kinds, including graceful fallback when native preview support is missing,
- update Partner steering so task packs prefer artifact writers for deliverables instead of asking for project write permissions.

Initial supported deliverables:

- research/report memo as `docx` and `pdf`,
- table extraction or budget-style analysis as `xlsx`,
- summary/briefing deck as `pptx`.

### Acceptance

- Partner can generate `docx`, `xlsx`, `pdf`, and `pptx` artifacts without project `write`, `edit`, or `bash`.
- Artifacts persist across app restart and can be previewed or exported from ArtifactPanel.
- Exporting to the workspace or an arbitrary filesystem path is a separate explicit user action.
- `xlsx` formulas/sheet names, `docx` headings/styles, `pptx` slides/notes, and `pdf` rendering survive reopen/export smoke tests.
- Source/citation metadata is attached to generated deliverables where relevant.
- `PARTNER_ENABLED` remains false until this feature, `F113`, and `F095` pass at least one full Partner workbench end-to-end flow.

## FEATURE_095: Partner Workbench and Skill Pack Delivery Flow

### Problem

Partner has a surface, artifacts, and KB planning. It still needs a complete workbench flow that lets users start real knowledge-work tasks without hand-assembling skills, sources, and artifact outputs. After the Cowork gap review and write-path correction, this feature is explicitly paired with `F109` and `F113`: the workbench is only useful if it can produce finished deliverables and save ordinary files safely.

### Design

Add a Partner workbench:

- task launcher for common work: PRD, RFC, report, meeting summary, changelog, PR description, review brief, research memo,
- source tray connected to KB and local files,
- skill pack selector with minimal presets,
- artifact target selector backed by `F109` generated office artifacts,
- workspace file output target backed by `F113` reviewed file proposals,
- source sufficiency and output-readiness preflight before run,
- citation/faithfulness checklist,
- run progress tied to workflow or single-session execution depending on KodaX support,
- visible "cannot run yet" state when a requested output kind is not implemented or not policy-allowed.

Initial skill packs should focus on code-adjacent knowledge work:

- architecture doc,
- PR description,
- changelog/release notes,
- review summary,
- product requirement breakdown.

### Acceptance

- A user can create a Partner task from sources and receive an artifact without touching raw skill paths.
- Generated artifacts carry source/citation metadata where relevant.
- Workbench runs use existing KodaX sessions/workflows instead of a Partner-specific engine.
- Coder and Partner remain separate surfaces with clear source/object boundaries.
- The first release includes at least one complete end-to-end flow from source attachment to `docx`/`pdf` export through `F109`.
- The first release also includes at least one complete end-to-end flow from source attachment to reviewed `md` or `txt` workspace file apply through `F113`.
- Partner is not re-enabled in the surface switcher until the workbench can complete artifact and workspace-file output flows without asking for blocked tools.

## FEATURE_096: Connector Catalog and OAuth Permission Center

### Problem

The PRD lists connectors, but the plan does not break down how Space should expose connector setup, permissions, source import, and GitHub collaboration closure.

### Design

Add a connector catalog:

- connector cards for GitHub, GitLab, Slack, Notion, Google Drive, local folders, generic web, and locally installed MCP/plugin-backed connectors,
- install/auth status,
- OAuth or token setup flow where applicable,
- secure token storage and revocation path,
- scopes, retention, and data-boundary display,
- per-connector permission prompts,
- connector source snapshots into Partner KB where supported,
- snapshot lifecycle controls: sync now, scheduled sync handoff to `F097`, stale markers, retention limits, and delete-local-snapshot actions,
- origin metadata on every imported source: connector id, external object id, captured timestamp, scope used, and content hash,
- GitHub issue/PR snapshot import for Partner KB and review tasks,
- optional branch/PR actions gated by explicit user confirmation: open PR page, prepare PR description, or create PR when credentials and repo policy allow it.

The catalog should treat MCP/connectors as governed capabilities, not as a hidden config file.
GitHub support should start as inspectable snapshots and review artifacts before adding write actions.
Connector writes should be separate from snapshot reads: read-only snapshots can ship first, while issue comments, PR creation, branch writes, or external edits require a second visible confirmation and audit event.

### Acceptance

- Users can see which connectors are installed, authenticated, degraded, or disabled.
- OAuth scopes or token permissions are visible before use.
- Connector snapshots are tagged with connector/source metadata.
- Revoking a connector disables future access without corrupting existing snapshots.
- GitHub issue/PR content can be captured as an auditable source for Partner workbench tasks.
- Any PR creation or branch-write action has a visible permission boundary and produces evidence in the session or audit trail.
- Connector snapshots can feed `F095` Partner tasks and `F110` refreshable live artifacts without re-auth prompts when the stored permission is still valid.
- Disabling a connector stops scheduled refreshes and makes stale live artifacts visibly stale rather than silently refreshing.

## FEATURE_097: Local Automations Scheduler

### Problem

Automations are a desktop-native strength, but Space should not depend on cloud routines. The first version should run locally and remain inspectable.

### Design

Add local automations:

- scheduled run definitions,
- manual trigger,
- pause/resume/delete controls,
- last/next run status,
- run history with skipped/failed/completed states,
- input prompt/template,
- source refresh settings for KB/connector snapshots,
- artifact target and export policy,
- target surface/session/workspace,
- permission mode,
- capability snapshot at creation time so a schedule does not silently gain newly installed tools,
- failure notification,
- OS scheduler integration later: Task Scheduler, launchd, systemd timer.

Automations start KodaX sessions or workflows; they do not define a new execution engine.
MVP runs only when Space or its approved background helper is available. If the machine is asleep or the app is not allowed to wake, the next launch shows a skipped-run record with a run-now action rather than pretending the schedule succeeded.

### Acceptance

- Users can create, pause, resume, delete, and manually run local automations.
- Every automation run has a session or workflow run record.
- Dangerous tools still require permission policy review.
- Failed runs create diagnostics and optional desktop notification.
- Scheduled runs cannot perform connector writes, filesystem exports, remote-runner starts, or desktop automation unless the stored policy explicitly allows that class of action.
- Automations can later refresh `F110` live artifacts, but the first release must make refresh history and stale data visible.

## FEATURE_098: Policy, Audit, and Admin Pack

### Problem

Teams need visibility and policy controls before they can trust Space for broader use. The PRD mentions enterprise policy and SIEM, but the first implementation should be local-first and should not imply SSO/SAML/SCIM support.

### Design

Add an admin pack:

- policy JSON import/export,
- provider allow/deny list,
- MCP/connector allow list,
- artifact writer/export policy for `F109`,
- workspace file proposal/apply policy for `F113`,
- automation policy for `F097`,
- remote runner and desktop automation gates for `F099`/`F111`,
- permission defaults,
- audit event export,
- redaction rules,
- local audit viewer,
- structured local audit stream compatible with file, JSONL, and optional OpenTelemetry-style exporters,
- SIEM/syslog export as optional follow-up.

This is local-first. Team distribution can come later via MDM/config file deployment.
SSO, SAML, SCIM, hosted admin dashboards, and centralized tenant management are non-goals for this feature. They require a separate enterprise track and distribution decision.

### Acceptance

- Policies can be inspected before applying.
- Audit events capture high-risk actions without leaking secrets.
- Exported audit data is structured and documented.
- User override rules are explicit and visible.
- The UI and docs do not claim enterprise identity federation.
- Managed configuration can be piloted with local files or MDM-style deployment before any hosted control plane exists.
- Audit coverage includes source attachment, connector snapshot, artifact generation/export, scheduled run, remote runner start, and desktop automation attempt events.
- Policy can keep Partner enabled for read-only work while blocking artifact export, connector writes, remote execution, or screen-control classes independently.

## FEATURE_099: Remote Runner and Long Task Monitor

### Problem

Some coding or research tasks are too long or environment-specific for the local desktop process. Space needs an optional self-hosted runner model without becoming a cloud sandbox product.

### Design

Add remote runner support:

- self-hosted runner registration,
- SSH/Docker runner profiles,
- local worktree sandbox or copy-on-write workspace mode where supported,
- resolved absolute path validation before any recursive copy, delete, or merge-back operation,
- capability and tool availability display,
- long task monitor,
- reconnect/resume state,
- logs/artifacts retrieval,
- explicit data boundary warnings.

Remote runners should be opt-in per task and must disclose what files/context leave the local machine.
The default desktop path remains local execution with permission governance; runner profiles are explicit infrastructure choices.
This feature may consume the sandbox feasibility output from `F094`, but it must not expose "safe autonomy" until copy/worktree merge-back and discard paths are understandable to a non-expert user.

### Acceptance

- Users can register and test a runner.
- Starting a remote run shows data boundary and permission implications.
- Long-running remote sessions can be monitored and resumed.
- Local and remote runs are clearly distinguished in session history.
- Worktree/copy sandbox experiments have a clear merge-back or discard path before they are exposed as safe autonomy.
- Remote runs cannot be started by `F097` automations unless the policy pack explicitly allows that runner profile.
- Runner logs and artifact retrieval preserve source/path redaction rules from `F098`.

## FEATURE_102: Surface Extension Points and Slot Registry

### Problem

Space will accumulate many surfaces: Coder, Partner, Workflow, Artifact, Memory, Todos, Connectors, Automations. Without extension points, the app risks hard-coded UI sprawl.

### Design

Add a conservative slot registry:

- static slot declarations for known app regions,
- typed contribution contracts,
- feature-gated panels/actions,
- no arbitrary remote UI execution,
- no marketplace in MVP,
- internal use first for Connector/Partner/Automation panels.

This is not a plugin marketplace. It is a maintainability layer for Space-owned surfaces.

### Acceptance

- New panels/actions can register through typed local contracts.
- Disabled features do not leak empty UI.
- Contributions are inspectable in dev diagnostics.
- Renderer remains free of direct KodaX runtime imports.

## FEATURE_110: Partner Live Artifacts and Refreshable Dashboards

### Problem

Static artifacts are enough for first deliverables, but Cowork-style recurring reports and dashboards need an artifact that can be refreshed from known sources without restarting the whole task. Space already has interactive HTML artifact sandboxing and source metadata; it lacks a persistent live-artifact manifest, refresh history, and a safe way to re-bind source snapshots.

### Design

Add refreshable live artifacts on top of the existing artifact store:

- live artifact manifest with artifact id, source refs, connector/local source bindings, refresh policy, last refresh, stale state, and version list,
- manual refresh MVP; scheduled refresh can attach later through `F097`,
- refresh inputs come from KB pages, local source snapshots, connector snapshots from `F096`, or explicit user-provided data files,
- every refresh creates a new artifact version with diffable metadata; old versions remain available,
- interactive HTML/dashboard rendering uses the existing static sandbox and host-injected data payloads, not arbitrary network access from inside the iframe,
- offline behavior uses the latest captured snapshot and marks data as stale,
- source/citation metadata stays visible in ArtifactPanel and in exported reports.

This feature is intentionally independent from `F067` LiveCanvas React-tier re-integration. `F067` may later provide richer component rendering, but `F110` should be useful with the current sandbox.

### Acceptance

- A Partner task can create a refreshable dashboard or report from KB/local/connector sources.
- Manual refresh produces a new version, preserves the prior version, and records source hashes/timestamps.
- Failed refreshes keep the previous artifact usable and display the failure reason.
- Live artifacts do not perform network requests or connector calls from sandboxed user-generated HTML.
- A disabled connector or revoked source permission makes the artifact stale instead of silently refreshing.

## FEATURE_111: Controlled Desktop App Automation Feasibility and Guardrails

### Problem

Claude Cowork can operate local apps and the user's computer. Space currently has safer primitives: connectors, browser tooling, file/source snapshots, KodaX sessions, and optional runner planning. General desktop automation is powerful but high-risk, so it should enter the roadmap as guarded feasibility and a narrow opt-in prototype, not as a default Partner capability.

### Design

Define a controlled desktop automation lane:

- capability order is connector API first, browser tool second, visible screen-control fallback last,
- per-app allowlist with OS accessibility/screen-recording consent checks,
- visible automation session with stop control, current target app, and action log,
- no password entry, payment flow, 2FA handling, destructive external action, or hidden background control in MVP,
- screenshots and OCR/text extraction pass through `F098` redaction rules before model use or audit export,
- every click/type/screenshot/navigation/action is audit-recorded with timestamp, target app/window, and permission class,
- manual confirmation before irreversible external writes, sends, uploads, purchases, deletes, or submissions,
- feasibility output before implementation: supported OS/app matrix, library/API choice, threat model, test harness, and rollback plan.

The first prototype, if included in this feature, should target one low-risk visible app flow and must remain disabled by default. Partner should never gain broad "use my computer" power from this feature alone.

### Acceptance

- Feasibility and threat model document the supported OS/app matrix, risks, non-goals, and required permissions.
- Desktop automation is default-off and policy-gated; enabling it requires an explicit user/admin action.
- Any prototype runs visibly, can be stopped, and records an audit trail.
- Redaction rules apply before screenshots or extracted text are persisted or exported.
- Partner tool policy can expose only named, narrow automation actions; it cannot expose unrestricted screen control.
- The product docs clearly position this as guarded local automation, not cloud-agent delegation.

## FEATURE_112: Legacy Continuity Remainder Hooks

### Problem

The v0.1.2-era F015/F017/F018 designs promised more than Space could safely deliver at the time. v0.1.20 closed the Space-owned slices through F082/F083/F084, but three remaining capabilities are still upstream- or SDK-gated: real Repointel warm controls, true no-session Quick Ask side queries with Partner promotion, and an end-to-end CLI/REPL handoff writer. Keeping those only as historical Partial notes makes the roadmap look fuzzier than it should.

### Design

Create a single v0.2.9 parking feature for these legacy continuity hooks:

- public Repointel warm/status adapter once KodaX exposes start/cancel/progress semantics,
- Quick Ask `sideQuery` or equivalent no-session call path, with an explicit promotion bridge into Coder and Partner when supported,
- CLI/REPL handoff writer and Space-to-CLI pickup protocol integration, or a fixture-backed compatibility harness if the CLI surface is not yet owned by Space,
- capability ledger entries that distinguish implemented Space UI, degraded fallback, unavailable SDK contract, and ready upstream contract,
- user-visible fallbacks that never imply hidden warm actions, silent session persistence, or bidirectional teleport when the contract is unavailable,
- tests around degraded states, capability transitions, and handoff payload validation.

This feature should not block Partner re-enable. It is scheduled with platform work because it depends on mature runtime contracts and cross-surface protocol stability rather than Partner deliverable UX.

### Acceptance

- F015/F017/F018 historical Partial rows each point to F082/F084/F083 for shipped slices and F112 for remaining hooks.
- When the SDK/CLI contracts are absent, Space shows precise degraded states and keeps the current safe fallback behavior.
- When contracts are present, warm controls, true side query, and handoff writer integration can be enabled without changing old v0.1.x design docs again.
- Handoff payloads are validated against workspace boundaries before they can open or attach a desktop session.
- Promotion into Partner respects `F095`/`F109` readiness and cannot bypass Partner tool policy.
