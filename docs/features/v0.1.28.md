> Design batch: React 19 migration + bash-output-compression rendering + packaged-Terminal fix
> Status: Completed

> Runtime baseline: React `19.2.7` / react-dom `19.2.7`; Electron pinned to `42.5.0` (exact); esbuild main-process target `node24`; `better-sqlite3@12.11.1` (exact).

## 1. Goal

`v0.1.28` is a runtime + SDK catch-up release with no new user-facing surface of its own. It (a) upgrades the renderer to **React 19**, (b) catches up to **KodaX SDK 0.7.61**, (c) renders the SDK's new **bash-output compression** state in tool cards, (d) fixes the **built-in Terminal in packaged installers**, and (e) refreshes documentation (per-version feature docs, `FEATURE_LIST`, and a new Chinese user manual).

## 2. React 19 Upgrade

| Item | Change |
| --- | --- |
| `react` / `react-dom` | `^18.3.1` → `19.2.7` (exact) in the desktop app and `@kodax-space/space-ui-kit`. |
| `@types/react` / `@types/react-dom` | `^18.3.x` → `19.2.x`. |
| UI-kit peer range | `react` peer widened to `^18.0.0 || ^19.0.0` (kit stays compatible with both). |
| Ambient JSX types | New `apps/desktop/renderer/src/types/react-jsx-compat.d.ts` re-exports React 19's `JSX.*` into the global `JSX` namespace, preserving the pre-19 ambient `JSX.Element` types the renderer uses so no component signatures change. |

Decision: keep the upgrade **type-transparent** — no component rewrites, no `React.FC` churn, no new React 19 APIs adopted in this release. The shim is the entire migration surface on the type side; runtime behavior is validated by the full unit + e2e suites and the renderer-boot smoke.

## 3. KodaX 0.7.61 SDK Catch-up

- Root and desktop workspaces resolve `@kodax-ai/kodax@0.7.61`; lockfile regenerated in sync.
- **Bash output compression rendering.** KodaX `0.7.61` can compress a bash tool's stdout/stderr and (optionally) save the full raw output to disk, leaving markers in the tool result:
  - `[Bash output compressed by <filter>.]`
  - `[Bash output compressed; full raw output saved to: <path>. Use read on that path if details are needed.]`
  - Space parses these (`renderer/src/features/session/messages/bashOutputCompression.ts`) and renders a `BashCompressedResult` tool card that shows a `compressed` marker with the filter name(s) and, when a raw-output path is present, a clickable link that opens it via the existing file-open path. The recovery-hint boilerplate is stripped from the visible result so the card stays readable.

## 4. Packaged-Terminal Fix (node-pty)

The built-in Terminal (F011) uses `node-pty`, a **workspace-only** native dependency. `electron-builder install-app-deps` rebuilds top-level native deps but skipped this one, so packaged installers shipped without a usable `node-pty` and the Terminal could not spawn a shell.

Fix: ship `node-pty` as an **extra resource** on Node's runtime module path instead of relying on the native rebuild:

- `electron-builder.yml` copies `node_modules/node-pty` → `resources/node_modules/node-pty`, filtered to `package.json`, `lib/**`, `prebuilds/**`, and the `build/Release` conpty/winpty/spawn-helper binaries (excluding `.pdb`/`.map`/`.ts`). `require('node-pty')` from `app.asar` then resolves against `resources/node_modules`.
- `.pdb` native debug symbols are excluded from the packaged bundle generally.
- `scripts/smoke-pack.mjs` now asserts the platform-specific `node-pty` prebuilds (`conpty.node` / `pty.node` / `spawn-helper` / conpty helpers) are present in the built installer, so a packaging regression fails the release smoke instead of shipping a broken Terminal.
