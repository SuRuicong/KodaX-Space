## FEATURE_052: Partner skill pack：研究（deep-research 内化 + web 知识源）

### 需求概述

第三组 Partner skill——研究：把"多源搜集 → 抽取 → 对抗核验 → 带引用综合"做成 Partner 的一等能力（deep-research 思路内化）。知识源含 web（F050 浏览器 + web_search）、本地文档、Connector（M2 接入，先占位）。产出 = 带出处的研究报告 artifact。

### 影响范围

- 新 `electron/skill/partner-packs/research/`：
  - `deep-research`——多源搜索 + 抓取 + 引用核验 + 综合报告（fan-out 搜索 → 深读 → 综合）
  - `competitive-scan`——竞品/方案调研
  - `literature-review`——文献/文档综述
- 知识源汇聚：把 web（F050）、本地文件（F047 作用域）、未来 Connector 聚成一个带出处的"研究上下文"。
- 复用：已有 deep-research / web-search / web-reader 能力思路；产出经 F048。

### 技术方案

`deep-research` skill 编排：① 多角度 web 搜索（web_search + F050 浏览器深读）；② 抽取关键来源；③ 对每条关键声明做"找证据/反驳"核验（轻量对抗）；④ 综合成报告，每段带 URL/文件出处。每个来源登记到 Sources（F046 左栏），报告登记 artifact（F048）。Connector 源 M2 接入，本版给本地 + web 两类。

> 完整的对抗核验判官质量依赖 F053 的 harness 升级；本版先用 skill 内编排实现"够用"的核验，强判官留 F053。

### 接口契约

无新 IPC；skill 经现有通道 + F050 浏览器工具 + F048 artifact。

### 实现步骤

1. 建 `research/` + 3 个 skill 定义。
2. `deep-research` 编排：多源搜索 → 深读（F050）→ 抽取 → 轻量核验 → 带引用综合。
3. 来源 → Sources 左栏登记；报告 → `artifact.write`。
4. 引用渲染：报告 artifact 内每段可点回源（URL/文件）。
5. 测试：一个研究问题产出报告 artifact，含 ≥N 个真实可点击来源；声明与来源对应。

### 验收标准

- [ ] 给研究问题，`deep-research` 产出带引用的报告 artifact，来源真实可点回。
- [ ] 知识源混合 web（F050）+ 本地文档，均登记到 Sources 左栏。
- [ ] 报告每个关键声明有出处；无出处的断言被标注/降权。
- [ ] 产出登记 F048，可迭代/导出。
- [ ] Connector 源留占位接口，不阻塞本版。
- [ ] 核验质量的已知上限记 known-issue，指向 F053 harness 升级。

## FEATURE_053: Partner 自定义画像 + H1-Partner 完整 harness（依赖 SDK R1/R2）

> **状态：Blocked on KodaX SDK。** 依赖 [ADR-007](../ADR/ADR-007-partner-surface-model.md) 依赖段的 R1（自定义画像 + embedder 工具走完整 harness）+ R2（`instructions` 可覆盖）。需求已转交 KodaX（[[kodax_sdk_export_gaps]] Partner 批次）。SDK 入口到位前**不动手**；F045–F052 全部不依赖它。

### 需求概述

把 Partner 从"借用内置 coding 画像 + skill 编排凑合"升级为**真正的非编码画像 + 完整 harness**：
- **画像**：Space 用 `createAgent({ instructions: <知识工作 system prompt>, tools: <non-bash-subset>, reasoning: <faithfulness 取向> })` 构造 `partner` 画像。
- **harness**：该画像经 SDK 的"自定义画像走完整 managed-task 管线"入口运行——拿到与内置 Coder **完全相同**的 verifier/AMA/managed-task，且 **completeness + faithfulness** 为判官目标（H1-Partner，区别于 Coder 的 H1）。
- **embedder 工具**：F050 的浏览器工具（`registerTool`）在该全 harness 下可用、受权限门控（R1 确认点）。

### 影响范围

- 新 `electron/kodax/partner-profile.ts`：构造 `partner` 画像（instructions + 工具子集 + reasoning）。
- session 起点：Partner surface 走"自定义画像 + 全 harness"入口（替换 F045–F052 期间的"内置画像 + 白名单裁剪"workaround）。
- harness：faithfulness 判官生效——产出的引用/出处被校验，杜撰被拦（直接提升 F049/F052 的产出可信度）。

### 技术方案（SDK 就绪后）

```ts
// electron/kodax/partner-profile.ts （伪码，待 SDK R1/R2 入口）
const partnerAgent = createAgent({
  name: 'partner',
  instructions: PARTNER_SYSTEM_PROMPT,        // R2：instructions 必须可覆盖
  tools: partnerToolSubset,                    // non-bash + 富格式 IO + browser
  reasoning: { profile: 'faithfulness' },
});
// R1：经 runManagedTask/Runner 收自定义 Agent，走与 Coder 同一 substrate 全管线
await runManagedTask({ provider, model, agent: partnerAgent, extensionRuntime }, prompt);
```

> 具体入口形态以 KodaX 交付为准（见 [ADR-007](../ADR/ADR-007-partner-surface-model.md) 建议接口 方案 A/B）。本 design 在 SDK 入口确定后细化。

### 实现步骤（解阻塞后）

1. 确认 SDK 交付的入口（R1/R2 + R1 的 registerTool 确认点）。
2. 写 `PARTNER_SYSTEM_PROMPT`（知识工作画像；与 KodaX REPL 行为做一致性校验）。
3. `partner-profile.ts` 构造画像；Partner session 切到该入口，移除 F047 的硬编码裁剪 workaround（保留作 fallback）。
4. faithfulness 判官接入，校验 artifact 引用/出处。
5. 回归：F049/F052 产出在新 harness 下杜撰率下降、引用校验生效。

### 验收标准（解阻塞后）

- [ ] Partner session 跑 Space 自定义 `partner` 画像（system prompt 与工具子集生效，非内置 Coder 人格）。
- [ ] 自定义画像任务触发 AMA 升级 + 产出 verifier verdict，与内置 Coder **行为对等**（非残血 Runner）。
- [ ] F050 浏览器工具（registerTool）在该全 harness 下可用、受权限门控。
- [ ] faithfulness 判官生效：产出引用被校验，无出处断言被拦/降权。
- [ ] F047 硬编码工具裁剪 workaround 退场（切到画像声明 + R3 维度白名单），保留 fallback。
- [ ] 与 KodaX REPL 的 Partner 行为一致性校验通过（参 [[feedback_mock_fidelity]]：用真 SDK 验，不只过 mock）。

### 当前阻塞处置

- 状态保持 **Blocked**，直到 KodaX 交付 R1/R2 入口。
- 在此之前，Partner 用 F045–F052 的 workaround（内置画像 + 白名单裁剪 + skill 编排）提供"够用"体验；本 feature 是"可信度跃迁"，不是 Partner 上线的前置。
- SDK 每次升版按 [[kodax_sdk_export_gaps]] Partner 批次对账，R1/R2 落地即解阻塞。
