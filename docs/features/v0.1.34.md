## Batch Intent

`KodaX owns runtime behavior; Space owns visibility, review, recovery, and user control.`

v0.1.34 makes the durable-but-currently-invisible parts of a session reviewable: memory proposals (distinct from Partner KB), durable todo/plan timelines with a session-end report, and the MCP/extension state of resumed sessions. Each surface renders KodaX-owned data and adds user review/annotation without inventing a parallel truth.

## FEATURE_088: Memory Governance Surface

> Status: Completed on 2026-07-07 against KodaX SDK 0.7.62 MemoryControlPlane.

### 需求概述

Memory and Partner KB are different things. KodaX memory governs reusable agent context and preferences; Partner KB stores source-backed knowledge. Space needs a UI that makes this distinction explicit and keeps memory writes user-reviewable.

### 影响范围

- New Memory panel in the renderer.
- Must keep Coder memory and Partner KB clearly separated (no list mixing).

### 0.7.62 SDK alignment analysis

Local dependency baseline is `@kodax-ai/kodax@0.7.62`. The release adds FEATURE_228, which turns F224 `memdir_handoff` proposals into a unified Memory Control Plane rather than creating a second memory store.

Available controller API:

- `createMemoryControlPlane({ cwd, learningStorePath?, memoryRoot?, projectDocs?, extraRefs?, sessionId?, sessionLineage?, artifactLedger?, discoverSkills?, memoryReviewer? })`
- `listInbox()` and `showProposal(id)` for pending memory action proposals.
- `approveProposal(id, expectedFingerprints)` and `rejectProposal(id, reason?)` for governed review.
- `listRefs(filter?)` and `readRef(ref)` for active/trusted memory refs and readable snapshots.
- `runCurator(input?)` and `maybeRunAutoCurator(input?)` for non-mutating governance reports.
- `buildMemoryPack(input)` for deterministic task-aware memory hints.
- `reviewMemoryFeedback(input)` for semantic review drafts when a host-provided `memoryReviewer` exists.

Validated behavior from a 2026-07-07 temp-project probe:

- A pending `memdir_handoff` is projected as `memory:<proposalId>` with action `write_memdir`, risk `medium`, target refs for the new topic file plus `MEMORY.md`, a diff preview, and `expectedFingerprints`.
- `approveProposal(id, {})` does not throw. It returns `applied:false` with `skippedReason:"approval fingerprints do not cover proposal preview"`.
- Wrong fingerprints return `applied:false` with `skippedReason:"target memory changed after preview"`.
- Editing `MEMORY.md` after preview returns `applied:false` with `skippedReason:"MEMORY.md changed after preview"`.
- Re-showing the proposal refreshes fingerprints and approval then writes both the topic file and `MEMORY.md`.
- Approved proposals appear as `learning_proposal` refs with lifecycle `trusted`; written memory files appear as `memdir` refs with lifecycle `active`.
- `buildMemoryPack()` returns selected memory refs as hints plus trace metadata; it does not dump full topic bodies into prompt context.
- `runCurator()` is non-mutating and may return a `no_op` report when no deterministic finding is available.
- `reviewMemoryFeedback()` returns no actions and warning `memory reviewer unavailable; semantic memory review was not run` unless Space supplies a reviewer.
- `rejectProposal()` removes the proposal from inbox and may also return reviewer warnings if no reviewer is configured.

SDK gaps / host obligations:

- Approval failure is result-based, not exception-based. Space must treat `applied:false` as blocked even when IPC succeeds.
- `readRef()` requires a full `MemoryItemRef`; there is no public read-by-id API. Space must resolve ids by listing refs first.
- `discoverSkills` defaults to true. Space Memory Governance should pass `discoverSkills:false` so the panel remains Coder memory, not skill inventory.
- Edit, merge, and direct archive are not first-class public controller methods in 0.7.62. Space should expose approve/reject/curate/read now, and only render archive/quarantine/conflict actions when the SDK returns those proposal actions.
- Curator findings are deterministic and structural. The SDK type supports duplicate/conflict/stale/quarantined/orphaned/no-op, but Space must not promise semantic duplicate/conflict detection without a stronger SDK reviewer or host heuristic.
- Feedback review needs a `memoryReviewer`; without it, the review surface is useful only as an unavailable-state explanation.
- Automatic curator reports are written by `maybeRunAutoCurator()`, but there is no public report-history listing API. Space can show the latest run result; browsing retained reports would require file-level host code.

### Product design

Add a Coder-only Memory Governance popout. It is not the Partner KB browser and it does not replace AGENTS.md editing.

Views:

- Inbox: pending `MemoryActionProposal` cards, sorted newest first. Each row shows action, risk, created time, rationale, target refs, source refs, changed paths, preview summary, warnings, and a bounded diff preview.
- Approved: `MemoryItemRef` inventory filtered to memory-related refs, primarily `memdir` active/trusted refs plus trusted learning proposals. Selecting a ref reads a bounded snapshot with fingerprint and warnings.
- Governance: run curator on demand and show report id, generated time, findings, severity, suggested action, ref ids, and warnings.
- Task Hints: optional diagnostic view for `buildMemoryPack()`, letting the user enter a task string and inspect which governed memory refs would be selected.

Actions:

- Approve: re-fetch or use the displayed proposal `expectedFingerprints`, call approve, then check `applied`. If false, keep the proposal visible and show `skippedReason`.
- Reject: call reject with optional reason, refresh inbox and refs, show reviewer warnings when present.
- Refresh: reload inbox, refs, and current selection.
- Read ref: resolve a ref from the listed inventory, call `readRef(ref)`, and render body snapshot.
- Curate: call `runCurator({ includePrivate:false, includeSensitive:false })` by default.

Boundary labels:

- Header copy: "Coder Memory Governance" and "Partner KB is separate."
- Empty states must distinguish "no pending memory proposals" from "no approved memory refs."
- Partner surface `/memory` should keep refusing or redirecting to Partner KB/AGENTS.md wording, not open this panel.

### Technical design

Schema package:

- Add `packages/space-ipc-schema/src/channels/memory.ts`.
- Re-export and register invoke channels in `channels/index.ts` and package index.
- Define zod schemas that mirror the public SDK contracts with payload caps for preview/body strings.
- Channels:
  - `memory.list`: `{ sessionId, includePrivate?, includeSensitive?, query? }` -> `{ inbox, refs, warnings }`
  - `memory.proposal`: `{ sessionId, proposalId }` -> `{ proposal: MemoryActionProposalT | null }`
  - `memory.approve`: `{ sessionId, proposalId, expectedFingerprints }` -> `{ result }`
  - `memory.reject`: `{ sessionId, proposalId, reason? }` -> `{ result }`
  - `memory.readRef`: `{ sessionId, ref }` -> `{ snapshot }`
  - `memory.curate`: `{ sessionId, includePrivate?, includeSensitive? }` -> `{ report }`
  - `memory.pack`: `{ sessionId, task, maxHints?, includeSnippets? }` -> `{ pack }`

Electron main:

- Add a small memory service that creates a fresh controller per call:
  - `cwd = session.projectRoot`
  - `sessionId = session.id`
  - `learningStorePath = sdk.resolveLearningProposalStore(session.projectRoot)`
  - `memoryRoot = sdk.resolveMemoryRoot(session.projectRoot)`
  - `discoverSkills:false`
- Register memory IPC handlers through `registerChannel`.
- Keep all file mutation inside SDK `approveProposal`; renderer never sends paths to mutate.
- Refresh by reconstructing the controller instead of caching stale previews.

Slash command:

- Replace old `/memory pending` learning-store shortcut with the 0.7.62 controller surface:
  - `/memory inbox|pending`
  - `/memory list`
  - `/memory show <proposal-or-ref-id>`
  - `/memory approve <proposal-id>`
  - `/memory reject <proposal-id> [reason]`
  - `/memory curate`
  - `/memory open`
  - `/memory help`
- `/learn memory` may remain as a raw learning-proposal filter for compatibility, but `/memory` becomes the governed memory API.

Renderer:

- Add `memory` to `PopoutKind`, floating popout policy, popout controls, toolbar menu, slash suggestions, and `/memory` action handling.
- Add `MemoryPanel` under `apps/desktop/renderer/src/shell/popouts/`.
- Use current `activeSessionId` and Coder surface gating.
- Keep layout dense and review-oriented: list left, detail right, compact tabs or segmented controls, bounded diff/body panes, and inline result banners.

### 技术方案

0.7.62 scoped implementation creates a Memory panel with:

- proposed memories inbox,
- approved memory refs and readable snapshots,
- non-mutating curator report output for stale/conflicting/quarantined/orphaned/no-op findings returned by the SDK,
- source/session provenance,
- approve and reject actions,
- task-hint diagnostics from `buildMemoryPack()`,
- Coder memory versus Partner KB boundary labels.

Memory writes must be user-reviewable unless KodaX explicitly marks them as safe auto-memory under a configured policy. Edit, merge, and direct archive stay out of the Space UI until the SDK exposes public mutation proposals or controller methods for those operations.

### 接口契约

- Consumes the KodaX memory proposal/approval store and provenance metadata.

### 实现步骤

1. Add schema contracts and schema tests for memory channels.
2. Add main-process memory service plus handler registration.
3. Upgrade `/memory` slash behavior to the SDK controller commands.
4. Add renderer Memory popout and entry points.
5. Add targeted tests for approval fingerprint failure, stale `MEMORY.md`, reject refresh, curator no-op, and pack selection.
6. Run targeted node tests and `npm run typecheck`.

### 验收标准

- Proposed memory changes are visible before they become durable.
- Approve requires preview fingerprints and blocked results are displayed as blocked.
- Users can reject pending memory proposals and see them leave inbox.
- Users can trace each memory to source session or artifact.
- Approved `memdir` refs can be listed and read from the Memory panel.
- Partner KB pages are not mixed into memory lists.
- Conflicts and stale entries are surfaced when returned by SDK governance reports.

## FEATURE_089: Todo Timeline and Session Report

### 需求概述

Space receives todo updates, but todos are still mostly transient UI. KodaX plans a TodoList JSON/CLI surface; Space should turn that into a durable desktop plan view with a session-end report.

### 影响范围

- Renderer Todo panel and Plan popout must share one store.
- Session-end report exportable as artifact.

### 技术方案

Add a Todo timeline:

- current plan,
- completed items,
- blocked/skipped items,
- timestamped changes,
- source event/session,
- filters by active/completed/blocked,
- session-end report summarizing what changed.

Todos remain KodaX-owned data. Space renders and annotates, but does not invent a parallel task system.

### 接口契约

- Consumes `todo_update` events today; durable persistence depends on the KodaX TodoList JSON/CLI surface.
- Builds on the existing `todo_drift_warning` host callback for nudges.

### 实现步骤

_待 Plan 阶段细化（durable timeline depends on KodaX exposing durable todo data）._

### 验收标准

- Todo state survives session reopen when KodaX exposes durable data.
- Plan popout and Todo panel share one store.
- Session-end report can be copied or saved as artifact.

## FEATURE_090: MCP / Extension Resume State Inspector

### 需求概述

When a resumed session depends on MCP servers or extension state, users need to know whether those capabilities were restored, missing, or stale.

### 影响范围

- Per-session extension/MCP state view in the renderer.
- Reconnect/retry actions wired to existing MCP lifecycle management (F039).

### 技术方案

Add per-session extension state:

- restored MCP servers,
- missing MCP servers,
- stale credentials,
- rehydration pending,
- reconnect/retry actions,
- tools that were available in the original run but unavailable now.

This consumes KodaX F211 (`initialExtensionState` / `initialExtensionRecords`) once available.

### 接口契约

- Consumes KodaX `initialExtensionState` / `initialExtensionRecords` resume metadata (KodaX F211).

### 实现步骤

_待 Plan 阶段细化（depends on KodaX F211 resume-state surface）._

### 验收标准

- Resumed sessions show extension/MCP health.
- Missing state does not fail silently.
- Users can reconnect or continue with degraded capabilities.
