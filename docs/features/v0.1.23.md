# v0.1.23 - Runtime Defaults and Mode Persistence

> Design batch: F106
> Status: Planned
> Source of truth: this file + [../FEATURE_LIST.md](../FEATURE_LIST.md)
> Release shape: patch feature, scoped to persistence, fallback resolution, and regression coverage.

---

## Intent

`v0.1.23` turns the Plan / Accept edits / Auto selector from a renderer-local preference into a durable Space runtime preference.

The desired behavior is simple:

- if a session has its own persisted runtime mode, resume with that mode,
- otherwise, if KodaX Space has its own persisted runtime default, use it,
- otherwise, inherit compatible defaults from `~/.kodax/config.json`,
- otherwise, fall back to safe built-in defaults.

KodaX Space should read KodaX CLI config as an ecosystem fallback, but Space should not write back to `~/.kodax/config.json`.

---

## FEATURE_106: Runtime Defaults and Mode Persistence

### Problem

The current mode selector has the right immediate runtime effect, but its persistence boundary is too weak:

- `ModeSelector` writes `pendingPermissionMode` into renderer `localStorage`, which is invisible to main-process resume paths.
- `session.create` and `tryResume` resolve defaults in different places, so new sessions and resumed sessions can drift.
- Historical sessions do not have a Space-owned runtime sidecar, so a resumed session falls back to current defaults rather than the mode the user chose for that session.
- KodaX `~/.kodax/config.json` is already loaded through `loadKodaxUserDefaults()`, but Space has no first-class setting that cleanly overrides it for permission mode.
- `autoModeEngine` is always effectively `llm` on historical fallback, even if the user chose Auto + rules.

The result is confusing: selecting Plan / Accept edits / Auto feels like a user preference, but after restart or resume the app can behave as if that selection never happened.

### Terminology

- **Permission mode**: canonical Space/KodaX mode: `plan`, `accept-edits`, or `auto`.
- **Edit mode**: UI shorthand for `accept-edits`.
- **Auto engine**: Auto sub-engine: `llm` or `rules`.
- **Space runtime defaults**: Space-owned defaults for future sessions.
- **Session runtime sidecar**: Space-owned per-session runtime fields used when resuming a historical session.
- **KodaX user defaults**: compatible defaults projected from `~/.kodax/config.json`.

### Design Principles

1. **Main process owns effective runtime resolution.** Renderer can request explicit values, but fallback and resume decisions must be resolved in main.
2. **Space settings override KodaX settings.** KodaX config remains a fallback, not the source Space mutates.
3. **Per-session state beats global state.** A resumed session should keep the runtime choices it had when it was created or last changed.
4. **No secrets in Space runtime persistence.** Store provider IDs, model IDs, modes, and booleans only. API keys stay in keychain/env.
5. **Failure falls back safely.** Invalid JSON, missing files, unknown enum values, or unsupported KodaX config values must degrade to `accept-edits` + `llm`, not crash startup.
6. **Sources are inspectable.** Tests and diagnostics should be able to tell whether a value came from session, Space, KodaX config, explicit IPC input, or built-in fallback.

### Precedence

For a new session:

| Priority | Source | Notes |
| --- | --- | --- |
| 1 | Explicit `session.create` input | Used when a picker or command intentionally passes a value. |
| 2 | Space runtime defaults | Stored in `~/.kodax/space/settings.json`. |
| 3 | KodaX user defaults | Read from `~/.kodax/config.json` through `loadKodaxUserDefaults()`. |
| 4 | Built-in defaults | `permissionMode: "accept-edits"`, `autoModeEngine: "llm"`, `reasoningMode: "auto"`, `agentMode: "ama"`. |

For resumed historical sessions:

| Priority | Source | Notes |
| --- | --- | --- |
| 1 | Session runtime sidecar | `~/.kodax/space/session-runtime/<sessionId>.json`. |
| 2 | Space runtime defaults | Applies to old sessions that predate sidecars. |
| 3 | KodaX user defaults | Compatibility fallback. |
| 4 | Built-in defaults | Safe final fallback. |

For an active in-memory session:

| Action | Effect |
| --- | --- |
| User selects Plan / Accept edits / Auto | Update active `ManagedSession.permissionMode`, persist session sidecar, and remember it as Space global default for future sessions. |
| User selects Auto engine | Update active `ManagedSession.autoModeEngine`, persist session sidecar, and remember it as Space global default. |
| User selects mode while no session is active | Update Space global default only. |

This preserves the current user expectation that the bottom mode selector also controls the next session.

### Space Global Settings

Extend the Space settings file from:

```json
{
  "version": 1,
  "defaultWorkspace": "C:/Users/example/kodax_workspace",
  "languageMode": "system"
}
```

to:

```json
{
  "version": 2,
  "defaultWorkspace": "C:/Users/example/kodax_workspace",
  "languageMode": "system",
  "runtimeDefaults": {
    "permissionMode": "auto",
    "autoModeEngine": "llm",
    "reasoningMode": "auto",
    "agentMode": "ama"
  }
}
```

`runtimeDefaults` fields are optional. Absence means "inherit from KodaX config or built-in fallback."

Recommended schema:

```ts
export interface SpaceRuntimeDefaults {
  readonly permissionMode?: PermissionMode;
  readonly autoModeEngine?: AutoModeEngine;
  readonly reasoningMode?: ReasoningMode;
  readonly agentMode?: AgentMode;
}
```

`v0.1.23` must implement `permissionMode` and `autoModeEngine`. `reasoningMode` and `agentMode` can be wired at the same resolver boundary if low risk, but they are not required for the acceptance path.

Migration:

- Read both version 1 and version 2 settings.
- When writing after this feature, write version 2.
- If old renderer `localStorage` key `kodax-space.pendingPermissionMode` exists and Space `runtimeDefaults.permissionMode` is absent, import it once into Space settings.
- Do not import invalid localStorage values.
- Keep the localStorage read path for one version as a compatibility bridge, then remove it in a later cleanup.

### Session Runtime Sidecar

Add a Space-owned per-session sidecar:

```text
~/.kodax/space/session-runtime/<sessionId>.json
```

Example:

```json
{
  "version": 1,
  "sessionId": "s_...",
  "permissionMode": "plan",
  "autoModeEngine": "rules",
  "reasoningMode": "auto",
  "agentMode": "ama",
  "updatedAt": "2026-06-23T12:34:56.000Z"
}
```

Rules:

- Write the sidecar when a session is created, using the resolved effective values.
- Update the sidecar when `session.setPermissionMode`, `session.setAutoModeEngine`, `session.setReasoningMode`, or `session.setAgentMode` succeeds.
- Delete the sidecar best-effort when `session.delete` succeeds.
- Sidecar read failures must not block session list or resume.
- Sidecar values must be schema-validated and normalized before use.

Provider and model are deliberately out of MVP sidecar scope. The provider/model path already has separate validation and recent fixes; expanding it here would increase the patch blast radius. A later feature can extend the same sidecar if exact provider/model restoration becomes a priority.

### Resolver Module

Add a main-process resolver module:

```text
apps/desktop/electron/kodax/runtime-defaults.ts
```

Recommended API:

```ts
export type RuntimeDefaultSource =
  | 'explicit'
  | 'session'
  | 'space'
  | 'kodax'
  | 'builtin';

export interface ResolvedRuntimeDefaults {
  readonly permissionMode: PermissionMode;
  readonly autoModeEngine: AutoModeEngine;
  readonly reasoningMode: ReasoningMode;
  readonly agentMode: AgentMode;
  readonly sources: {
    readonly permissionMode: RuntimeDefaultSource;
    readonly autoModeEngine: RuntimeDefaultSource;
    readonly reasoningMode: RuntimeDefaultSource;
    readonly agentMode: RuntimeDefaultSource;
  };
}

export async function resolveRuntimeDefaults(input: {
  readonly explicit?: Partial<ResolvedRuntimeDefaults>;
  readonly sessionId?: string;
  readonly includeSessionSidecar?: boolean;
}): Promise<ResolvedRuntimeDefaults>;
```

The resolver should:

- load session sidecar only when `includeSessionSidecar` is true,
- load Space settings,
- load KodaX defaults through `loadKodaxUserDefaults()`,
- normalize unsupported KodaX permission values:
  - `plan` -> `plan`,
  - `accept-edits` -> `accept-edits`,
  - `default` -> no value,
  - `bypass-permissions` -> no value in MVP,
- always return concrete values.

The resolver should be pure in behavior but not pure in implementation: it can read stores, but it must not write settings or sidecars.

### IPC and Main Flow

#### Settings IPC

Extend `settings.get` output with:

```ts
runtimeDefaults: SpaceRuntimeDefaults;
```

Add:

```ts
settings.setRuntimeDefaults({
  runtimeDefaults: Partial<SpaceRuntimeDefaults>
})
```

Semantics:

- Partial update merges with existing `runtimeDefaults`.
- Passing `null` for a field can clear that field if the schema wants an explicit "inherit from KodaX" action.
- The first implementation can expose only set/merge and leave field clearing to a later Settings UI if not needed.

#### `session.create`

Before `kodaxHost.createSession()`:

1. Validate provider as today.
2. Resolve runtime defaults using explicit input values plus Space/KodaX fallback.
3. Pass the resolved values to `kodaxHost.createSession()`.
4. Write a sidecar for the new session after successful creation.

This removes the hard-coded host fallback from being the first meaningful default.

#### `session.list`

For in-flight sessions, keep using runtime values from memory.

For persisted sessions:

1. Resolve runtime defaults with `sessionId` and `includeSessionSidecar: true`.
2. Use sidecar values where present.
3. Otherwise use Space/KodaX/built-in fallback.
4. Keep surface from SDK summary tag as today.

This prevents the sidebar from displaying `accept-edits` when a persisted session is actually going to resume in `plan`.

#### `tryResume`

Use the same resolver as `session.list`:

```ts
const runtime = await resolveRuntimeDefaults({
  sessionId,
  includeSessionSidecar: true,
});
```

Then create the runtime session with:

- `permissionMode: runtime.permissionMode`,
- `autoModeEngine: runtime.autoModeEngine`,
- `reasoningMode: runtime.reasoningMode`,
- `agentMode: runtime.agentMode`.

Provider/model resolution stays on the existing path in v0.1.23.

#### Runtime Setters

After a successful setter:

- `session.setPermissionMode` writes sidecar field `permissionMode`,
- `session.setAutoModeEngine` writes sidecar field `autoModeEngine`,
- `session.setReasoningMode` writes sidecar field `reasoningMode`,
- `session.setAgentMode` writes sidecar field `agentMode`.

The setter should return `{ ok: true }` even if sidecar persistence fails, but it should log a sanitized warning and optionally return a non-fatal `persisted: false` in a later schema revision.

### Renderer Flow

#### Boot

Renderer boot should treat `settings.get().runtimeDefaults` as the durable "next session" source.

`pendingPermissionMode` in Zustand should become either:

- derived from `settings.runtimeDefaults.permissionMode`, or
- kept as an in-memory mirror populated from settings.

Renderer should not treat localStorage as authoritative after migration.

#### Mode Selector

When a session is active:

1. Optimistically update the session mode in the store as today.
2. Invoke `session.setPermissionMode`.
3. Invoke `settings.setRuntimeDefaults({ permissionMode: mode })`.
4. If session update fails, revert the active session mode.
5. If settings write fails, keep the session update but show a small non-blocking toast that the default was not saved.

When no session is active:

1. Invoke `settings.setRuntimeDefaults({ permissionMode: mode })`.
2. Update local store mirror from the returned settings.
3. Show the selector as `Plan (next)`, `Accept edits (next)`, or `Auto (next)`.

Auto engine follows the same pattern:

- active session: update session engine and global default,
- no active session: update global default only,
- when permission mode is not `auto`, still allow engine selection if the UI already exposes it, because it will apply next time Auto is selected.

#### Settings UI

MVP does not need a full settings panel redesign.

Recommended small addition:

- Preferences section shows "Default permission mode" with the same three options.
- Auto engine control appears when default permission mode is Auto, or as a compact secondary control.
- A small source label may show "Space default" or "Inherited from KodaX config" if the resolver source is exposed.

If time is tight, the bottom `ModeSelector` is sufficient as the primary write surface, and Settings can wait. The durable backend should still be in place.

### KodaX Config Compatibility

Keep using `loadKodaxUserDefaults()` for `~/.kodax/config.json`.

Current mapping is correct for MVP:

- KodaX `permissionMode: "plan"` maps to Space `plan`.
- KodaX `permissionMode: "accept-edits"` maps to Space `accept-edits`.
- KodaX `permissionMode: "default"` maps to no Space value.
- KodaX `permissionMode: "bypass-permissions"` maps to no Space value.

Do not map `bypass-permissions` to Space `auto` automatically. Space Auto is guarded by `AutoModeToolGuardrail`; treating KodaX bypass as Auto would be semantically misleading.

### Failure and Safety

- Corrupt Space settings: warn, fall back to defaults, keep app booting.
- Corrupt sidecar: warn, ignore sidecar for that session.
- Unsupported enum: ignore the field, do not reject the whole settings file if other fields are valid.
- KodaX config load failure: keep existing safe fallback.
- Sidecar directory write failure: active runtime still changes, persistence warning only.
- No secrets are logged or persisted.
- Session IDs used in sidecar paths must be filename-safe and must not allow path traversal.

### Implementation Plan

1. Add schema and store support.
   - Extend settings schema to version 2.
   - Add `SpaceRuntimeDefaults`.
   - Add `settings.setRuntimeDefaults`.
   - Add migration from old localStorage pending permission mode through renderer boot.

2. Add session runtime sidecar store.
   - New module under `apps/desktop/electron/kodax/`.
   - Atomic JSON write.
   - Safe read with zod validation.
   - Best-effort delete.

3. Add resolver.
   - Centralize Space/KodaX/built-in precedence.
   - Return concrete defaults and source metadata.
   - Add unit tests for every precedence branch.

4. Wire main session lifecycle.
   - `session.create` uses resolver and writes initial sidecar.
   - `session.list` uses resolver for persisted session placeholders.
   - `tryResume` uses resolver before creating the runtime session.
   - Runtime setter IPC handlers update sidecars.

5. Wire renderer.
   - Replace localStorage as authoritative source for `pendingPermissionMode`.
   - Mode selector writes `settings.setRuntimeDefaults`.
   - Keep compatibility migration for old localStorage values.

6. Add regression tests.
   - Settings migration and runtime defaults persistence.
   - Resolver precedence.
   - Sidecar write/read/delete.
   - Session create writes sidecar.
   - Setter updates sidecar.
   - `session.list` and `tryResume` use sidecar before Space/KodaX fallback.
   - Renderer mode selector persists default when no session exists.

### Acceptance

- Selecting Plan, quitting, and opening Space again shows Plan as the next-session mode.
- Selecting Auto + rules, quitting, and opening Space again preserves Auto + rules as the next-session default.
- Creating a session in Plan writes a session sidecar.
- Resuming that session later uses Plan even if the global default has changed to Accept edits.
- If no Space setting exists, Space reads compatible `~/.kodax/config.json` permission defaults.
- Once a Space runtime default exists, it wins over `~/.kodax/config.json`.
- Invalid Space settings, invalid sidecars, and invalid KodaX config values do not break startup.
- `session.list` display and `tryResume` runtime use the same effective permission mode and auto engine.
- Existing provider/model default behavior remains unchanged.

  - resumed historical session uses session sidecar.

### Non-Goals

- Do not write to `~/.kodax/config.json`.
- Do not change provider key storage.
- Do not move provider/model default ownership in this patch.
- Do not redefine `bypass-permissions` as Space Auto.
- Do not add a large Settings redesign.
- Do not require SDK changes.
- Do not migrate every old session to a sidecar at startup; create sidecars lazily on create, resume, or setter.

### Later Hooks

- Extend sidecars to provider/model once the product wants exact historical provider/model restoration.
- Expose runtime default sources in Diagnostics.
- Add a Settings action to clear Space runtime defaults and return to KodaX config inheritance.
- Remove legacy localStorage migration after one or two releases.
