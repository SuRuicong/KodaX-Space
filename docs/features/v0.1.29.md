The research says the strongest pattern is **separation of concerns**:

- Codex's **task sidebar** follows what the agent is doing and can surface plan, sources, artifacts, and task summary.
- Codex's **review pane** owns file-level review: changed-file navigation, diff expand/collapse, inline comments, staging, unstaging, and reverting.
- Codex's **worktree/local environment controls** distinguish local work, worktrees, branches, handoff, setup scripts, and project actions.
- The screenshot review shows a distinct **environment information popover**: Changes, Local/Worktree/Cloud mode, Branch, Commit/Push, and Sources are compact rows that open the corresponding deeper surface.
- Claude Code's subagent model reinforces that subagents should return summaries and keep verbose output isolated from the main conversation; UI should show subagent role/status/result first, not raw event streams.
- GitHub Copilot cloud agent reinforces the end-to-end coding-agent loop: research, plan, make changes on a branch, review the diff, iterate, then PR.
- Devin/Cascade reinforces long-task affordances: plans/todos, queued follow-up messages, checkpoints/reverts, tool calling, and real-time awareness.
- Apple, Material, Microsoft, Fluent, and WAI-ARIA all point in the same direction: use anchored popovers/flyouts for lightweight contextual choices; use side sheets or docked panels for secondary content; reserve dialogs/modals for blocking decisions; and make real modal focus behavior strict and accessible.

Design implication for Space:

**Environment Hub answers "where am I, what can this agent touch, what changed, and what environment/action should I switch to?"**

**Task Dock/right sidebar answers "what is the agent doing, what evidence exists, what changed, and what needs review?"**

**Floating surfaces answer "is this an anchored choice, a persistent secondary panel, a review workspace, a global command/search overlay, or a blocking decision?"**

## FEATURE_103: Workspace Environment Hub, Task Dock, and Floating Surface Redesign

### Problem

KodaX-Space has mixed three different jobs into one vague right-side idea:

- environment controls: project, branch, local/worktree/cloud mode, source roots, commit/push;
- task observability: plan, workflow, subagents, current action, waiting/error/approval states;
- review and output: changed files, diffs, generated artifacts, terminal output, approvals.

Codex's screenshot pattern shows why this feels wrong. The compact "environment information" panel is not a runtime summary. It is a workspace/environment hub with action rows. For example, the Changes row is not the diff itself; clicking it should open the full changes list/review surface. Local opens an execution-location menu. Branch opens a branch picker. Sources opens source context.

The current Space right sidebar also exposes worker implementation details. `managed_task_status.events` are grouped by `workerId`, so subagents appear as event buckets or run turns instead of as semantic helpers with roles, state, and latest useful result.

The current floating-layer model has the same problem at a higher level. `PopoutOverlay` currently hosts very different things under one shell: task lists, plan, agents, MCP, preview, terminal, diff, artifacts, and workflow. Some are narrow right slide-ins; some are full-cover workspaces. Separately, Space mounts permission, ask-user, confirm, settings, help, command palette, quick-ask, slash-command, and toast surfaces with separate behavior. Users experience this as "the app popped something up" instead of understanding whether they are making a choice, inspecting evidence, reviewing code, or answering a blocking question.

### Product Principle

F103 should redesign the right side and overlay behavior as a **three-part shell system**:

- **Environment Hub:** compact, always reachable workspace control center.
- **Task Dock:** right sidebar for task progress, evidence, sources, changes, artifacts, and review routes.
- **Floating Surface System:** shared rules for anchored popovers, side/focus panels, review workspaces, command overlays, blocking modals, and toasts.

Simple mental model:

- Environment Hub: "Where am I working, under what mode, and what workspace actions are available?"
- Task Dock: "What is the agent doing, what evidence exists, and what needs review?"
- Floating surface: "Am I choosing, inspecting, reviewing, searching, or deciding?"
- Blocking modal: "The agent cannot safely continue until I answer."

The important correction: **the Codex-inspired top/right panel is an action router, not a mini right sidebar.**

The second correction: **the middle floating overlay is not a generic place for every detail view. It should be reserved for global command/search/help and truly blocking decisions; review and artifact work need a dedicated focus workspace.**

### Surface Responsibilities

| Surface | Owns | Primary Interaction | Does Not Own |
| --- | --- | --- | --- |
| Environment Hub | Project/session scope, changes count, execution location, branch, commit/push, sources, quick project actions | Row click opens the corresponding menu, right-sidebar tab, or review surface | Plan body, worker event history, full diff, artifact preview, terminal logs |
| Task Dock / Right Sidebar | Current run summary, plan/todos, workflow, semantic subagents, changes list, sources/context, artifacts, review status | Tabs/sections, focused drill-down, compact evidence lists | Environment switching menus, raw terminal output, modal approvals |
| Review Workspace | Full file diff, staged/unstaged scopes, inline comments, stage/revert/apply decisions | File tree/list, hunk actions, comments | Global environment status |
| Artifact Workspace | Generated files, previews, export/open actions | Preview, inspect, export | Runtime orchestration |
| Terminal Workspace | Commands and raw command output scoped to project/worktree | Command execution and output inspection | Human-readable task summary |
| Anchored Popover/Menu/Flyout | Compact contextual choices tied to one trigger | Click/keyboard selection, outside/Esc dismissal | Full review flows, long logs, nested popovers, blocking decisions |
| Command/Quick Overlay | Command palette, quick ask, help, global search/switching | Keyboard-first search and action execution | File diff review, worker history, approvals |
| Approval Modal | Blocking permission, ask-user, budget, destructive action decisions | Explicit allow/deny/answer | Background status history |
| Composer/Bottom Strips | Queueing, typing-adjacent feedback, lightweight current activity | Keep the user oriented while typing | Canonical workspace status or full task state |

### Design

#### L0: Environment Hub

Add a compact top-right Environment Hub, similar in spirit to the Codex screenshot.

It should be available whether the right sidebar is open or closed. It should appear as a small toolbar button or header control, then open a compact popover with action rows.

Recommended rows:

| Row | Shows | Click Behavior |
| --- | --- | --- |
| Changes | `+added -deleted`, dirty/staged count, warning if conflicted | Open Task Dock to Changes tab/section with the full changed-file list. If a review pane is already open, focus it. |
| Location | Local / Worktree / Cloud / Partner workspace where applicable | Open execution-location menu: continue locally, move/handoff to worktree/cloud when supported, show disabled reasons. |
| Branch | Current branch or detached/worktree state | Open branch picker with search, current branch marker, dirty-file hint, create-and-checkout action. |
| Commit/Push | Commit readiness, ahead/behind, push/PR state | Open commit/push flow or explain why disabled. |
| Sources | Attached files/dirs, web/connector/KB/source count | Open Task Dock Sources/Context tab. |
| Mode/Permission | Optional compact mode indicator only when it affects action safety | Open permission/mode menu; do not expose private env/API values. |

The Environment Hub is not a card wall. Rows are stable, icon-first, and action-oriented. It should preserve the user's spatial sense: row hover shows the target, row click opens that target, and returning from the target keeps the hub state unchanged.

Codex-specific behavior we should copy conceptually:

- Clicking **Changes** opens the full changes list/review path, not a tiny inline diff.
- Clicking **Local** opens a continuation/location menu.
- Clicking **Branch** opens branch search/selection and create branch action.
- Sources are represented as compact tokens, with detail in the sidebar or source panel.

#### L1: Task Dock / Right Sidebar

Redesign the right sidebar into a task dock. It should not duplicate Environment Hub rows. It should be the structured evidence and review surface for the selected session.

Preferred top-level tabs or grouped sections:

1. **Run**: current run summary, current action, attention/error state, queued follow-ups.
2. **Plan**: plan/todos, drift warnings, session report hook later.
3. **Agents**: semantic subagent status, role, current responsibility, latest meaningful result.
4. **Workflow**: workflow graph/timeline, replay/draft/review status.
5. **Changes**: full changed-file list, staged/unstaged/conflict state, route to diff/review pane.
6. **Sources**: attached files/dirs, source roots, web/connector/KB inputs, source health.
7. **Artifacts**: generated files/previews and export/open routes.
8. **Context**: token budget, compaction, provider/model/mode, runtime health.

Section questions:

| Section | User Question | Primary Content |
| --- | --- | --- |
| Run | What is happening now? | One headline, current action, attention/error state, up to two metrics |
| Plan | What path did the agent commit to? | Current plan/todos and drift warnings |
| Agents | Who is working on what? | Subagent role, current responsibility, phase, latest useful status/result |
| Workflow | What pipeline/run is active? | Run status, stage, replay/draft/review state |
| Changes | What changed in the workspace? | Full changed-file list, diff route, staged/unstaged state |
| Sources | What inputs is the agent using? | Attached sources, source roots, connector/web/KB inputs |
| Artifacts | What did the agent produce? | Generated files, previews, export/open routes |
| Context | What constraints affect this run? | Token budget, compaction, provider/model/mode, runtime health |

The right sidebar may have a sticky Run header, but that header is only the task state. Environment rows such as branch/location/commit belong in the Environment Hub.

#### L2: Floating Surface System

Deep work and transient choices should move out of the compact hub and sidebar, but not all into the same generic middle overlay.

Current Space surface inventory:

- `PopoutOverlay`: `preview`, `diff`, `terminal`, `tasks`, `plan`, `agents`, `mcp`, `artifact`, `workflow`.
- Blocking decisions: `PermissionModal`, `AskUserModal`, `ConfirmDialog`.
- App-level overlays: `SettingsModal`, `HelpOverlayController`, `CommandPaletteController`.
- Contextual popovers: `QuickAskPopover`, `SlashCommandPopover`, Environment Hub row menus after F103.
- Non-blocking notifications: `ToastContainer`, persistent notification surfaces.
- Auto-promotion logic: Smart Popout Director currently promotes `plan` and `diff` when it sees plan/diff signals and no active popout.

Target taxonomy:

| Surface Type | Examples | Use When | Placement / Focus |
| --- | --- | --- | --- |
| Anchored popover/menu/flyout | Environment Hub Location menu, Branch picker, Sources menu, Slash command picker | The choice is tied to one visible trigger and can be ignored or dismissed | Anchored to trigger, no dark backdrop, outside/Esc closes, keyboard navigation, no nesting |
| Dock section / side sheet | Task Dock sections, expanded Agents/Workflow/Plan details | The content is secondary but useful alongside the conversation | Right side or docked sheet, persistent if pinned, no focus trap unless modal |
| Review workspace | Diff/review, changed-file review, large artifact inspection | The user needs width, navigation, and repeated inspection/actions | Full-height review region or full-cover workbench, explicit close/back, no arbitrary center card |
| Terminal workspace | Terminal manager and command output | Raw command output or interactive shell is needed | Dedicated terminal area/popout, preserve scroll and session identity |
| Command/quick overlay | Command Palette, Quick Ask, Help | Global search, navigation, action execution, lightweight ask | Center overlay is allowed, keyboard-first, dismissible, no destructive action without confirmation |
| Blocking modal/dialog | Permission approval, ask-user, budget, destructive confirm | The agent cannot safely continue or the user must answer a blocking question | Center modal, inert background, focus trap, labelled title, Escape/close policy, focus restore |
| Toast/notification | Run complete, background warning, non-blocking error hint | The user should know but does not need to decide now | Edge stack, time/dismiss behavior, never contains required choices |

Routing rules:

- Changes row -> Task Dock Changes; file row -> Review Workspace.
- Artifact row -> Artifact Workspace.
- Terminal action -> Terminal Workspace.
- Approval/ask-user/budget/destructive action -> Blocking Modal.
- Branch/location row -> Environment Hub anchored menus.
- Plan/Agents/Workflow section expand -> docked detail or side sheet first; center overlay only if the user explicitly asks for a temporary focus view.

Central overlay policy:

- The middle floating layer is for global command/search/help and blocking decisions.
- It is not the default destination for worker history, routine plan details, changed-file lists, or environment menus.
- Review/diff/artifact content can occupy the main conversation area or a full-height workbench when it needs space, but it should be presented as a workspace with a title bar, navigation, and explicit close/back, not as a generic modal.
- Smart Popout Director should prefer focusing the Task Dock section. It may open a Review Workspace only when a direct user action implies review, or when an existing user preference explicitly allows first-signal auto-review.
- Do not stack popovers inside popovers. If a popover needs another layer of detail, route to a dock section, side sheet, or workspace.

Accessibility and dismissal rules:

- Blocking modals must implement the WAI-ARIA modal dialog contract: focus moves inside, `Tab`/`Shift+Tab` stay inside, Escape behavior is defined, `aria-modal` is truthful, the dialog has an accessible label, and focus returns to the invoking control or a logical next target.
- Anchored popovers and flyouts should be quickly dismissible with outside click and Escape, keep their trigger relationship visually clear, and avoid covering information needed to use the popover.
- Non-modal side/focus surfaces should not mark the rest of the app inert and should not trap focus unless they temporarily become modal on compact/mobile layouts.
- Destructive or irreversible actions should focus the safest action by default and use explicit labels.

Visual and layout rules:

- Backdrops are for blocking modals and high-focus review workspaces only. Anchored menus and routine side sheets should not dim the whole app.
- Stable dimensions are required: frequent worker/status updates must not resize the surface.
- Use one overlay host / z-index scale so `PermissionModal`, `AskUserModal`, command palette, popovers, and review workspaces do not compete by DOM accident.
- On compact widths, anchored menus may become full-width sheets; review/artifact workspaces become route-like full-screen surfaces; blocking modals remain modal.

This preserves a clear interaction stack:

```text
Hub row -> Dock section -> Focus surface
```

### Agents/Subagents Display Correction

The right sidebar Agents section should show **semantic subagent status**, not run-turn history.

Primary row content should be:

- subagent title or role,
- active/idle/waiting/error state,
- current responsibility,
- current phase,
- latest human-meaningful result,
- optional evidence count or changed-files count.

Event count, raw event list, repeated progress messages, and per-turn summaries are secondary debug/history data.

Recommended right-sidebar row:

```text
Research worker
Finding Codex review behavior · official docs checked
```

Recommended expanded row:

```text
Research worker
Role: source gathering
Now: reading official docs
Latest: confirmed Changes belongs to review/diff path
Evidence: 4 sources
Trace: 5 events
```

Non-recommended primary display:

```text
Worker (7) · worker
- progress: turn 1
- progress: turn 2
- completed: turn 3
```

Rules:

- The compact right sidebar shows at most the active few subagents plus aggregate waiting/fan-out/budget state.
- The full Tasks/Agents popout can expose event history, but semantic status comes first and raw trace is behind expansion.
- A worker with only an id or generic title should not be presented as a meaningful subagent; avoid UUID-like ids in primary UI.
- "All workers idle" appears only when no workflow, worker, approval, fan-out, or waiting state is active.

### State Taxonomy

Task Dock run mode:

- `no_project`: no project is selected, or the selected project cannot be resolved.
- `idle`: project/session is ready and no active run requires attention.
- `running`: a send, tool, workflow, worker, or background task is active.
- `attention`: the user must answer, approve, unblock, or decide.
- `error`: the latest meaningful state is failed or recovering.
- `completed`: a run just completed and the completion signal can linger briefly.

Severity:

- `neutral`: empty or ordinary idle state.
- `info`: useful context, no action needed.
- `running`: active work.
- `warning`: degraded or attention soon needed.
- `danger`: error or blocked user action.
- `success`: recent completion.

### Event Priority

The Task Dock run priority is display-only and separate from Smart Popout Director.

Priority order:

1. User action required: permission, ask-user, budget approval.
2. Error / recovery: `session_error`, retry, provider recovery.
3. Workflow live: running, paused, replaying, future draft/review states.
4. Managed task active worker / verifier.
5. Current tool activity, especially file mutation tools.
6. Context compaction / token pressure.
7. Thinking / writing / sending.
8. Plan / todo update / todo drift warning.
9. Idle environment summary.

### UX Rules

- The Environment Hub should never steal focus; it opens only when clicked or via an explicit shortcut.
- The Task Dock should never auto-open full popouts.
- The middle overlay should never be the default destination for ordinary detail. Use it for Command Palette, Quick Ask, Help, and blocking decisions.
- Anchored popovers must be tied to a visible trigger and must close predictably on outside click/Escape.
- Review/diff/artifact surfaces should behave like workspaces: titled, navigable, wide enough, explicitly closeable, and able to preserve review position.
- Permission, ask-user, budget, and destructive confirmations are the only MVP surfaces that should make the rest of the app inert.
- Running status may open the Task Dock only when existing right-sidebar auto-open behavior already applies.
- Completed state can linger briefly, then fall back to idle environment.
- Attention and error states persist until the next user action, dismissal, or newer run state.
- Use stable dimensions so frequent event updates do not resize the layout.
- Use icon-first compact controls and concise text; avoid a card wall.
- Avoid nested floating layers. A popover that needs more room routes to a dock section, side sheet, or workspace.
- Bottom live strips should be reduced after parity so Space does not say the same thing in two places.
- Partner can share the Environment Hub basics, but Partner-specific Task Dock content waits for the Partner workbench/right-side model after re-enable.

### Non-Goals

- Do not build a dashboard card wall.
- Do not duplicate `TasksPanel`, `PlanPanel`, `WorkflowPanel`, Diff, Artifact, or Terminal.
- Do not make a universal center modal that hosts every advanced surface.
- Do not make routine status/detail views blocking.
- Do not rely on DOM order accidents for overlay priority once `FloatingSurfaceHost` exists.
- Do not add a settings page for summary customization in MVP.
- Do not add new git IPC from the Environment Hub until git snapshot logic is shared.
- Do not expose API key env names, raw env values, or private runtime internals.
- Do not implement full cloud/worktree handoff if the underlying host support is not ready; show disabled reasons instead.
- Do not add Partner-specific Task Dock behavior in this release.
