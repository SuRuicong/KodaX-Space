## FEATURE_113: Partner Controlled Workspace File Writes

### Problem

`F109` lets Partner generate finished office artifacts, but it does not let Partner create or update ordinary workspace files. That leaves a real usability hole: Partner can draft a PRD, changelog, meeting note, research memo, JSON/CSV output, or README update, but cannot save it into the user's selected project/folder without falling back to raw `write`/`edit` tools. Raw project mutation is intentionally blocked by `F047`, so the missing feature is a Space-owned reviewed file-write path, not a blanket permission unlock.

### Design

Add controlled workspace file writes as a Partner-safe Space tool and UI flow:

- introduce a narrow `create_file_proposal` / `update_file_proposal` tool family for Partner,
- proposals contain target path, operation kind, content or patch, source refs, rationale, content hash, and safety classification,
- target paths must resolve inside the current Partner source/output scope and pass traversal, symlink, hidden-dir, binary, size, and overwrite checks,
- proposals land in a review queue with diff preview, source/citation metadata, and apply / reject / save-as actions,
- applying a proposal is performed by Space after explicit user confirmation or a future policy rule, not by exposing raw Partner `write`/`edit`,
- support new text-like files first: `md`, `txt`, `json`, `yaml`, `csv`, `html`, `css`, and other safe text formats; existing-file updates use diff review,
- code file edits are allowed only through the same review/apply path and should offer "handoff to Coder" when the task becomes coding-heavy,
- successful apply creates an audit/provenance entry and refreshes file/diff/artifact surfaces where relevant.

This feature complements `F109`: `F109` owns generated office deliverables in the artifact store; `F113` owns ordinary workspace file proposals and explicit apply/export. Both keep Partner off raw shell and unrestricted filesystem tools.

### Acceptance

- Partner can propose a new markdown/text/json/csv/html file and the user can apply it to a selected workspace/output folder.
- Partner can propose an edit to an existing text file and Space shows a diff before apply.
- Invalid paths, path traversal, hidden-system targets, symlink escapes, oversized files, and binary edits are blocked with clear reasons.
- Applying a proposal updates the workspace through Space, records audit/provenance metadata, and refreshes the relevant file/diff view.
- Rejecting a proposal leaves the workspace unchanged and keeps the proposal visible in session history.
- Partner still cannot call raw `write`, `edit`, `multi_edit`, `bash`, or shell-backed tools in the Partner surface.

### Verification

- `npm run typecheck`
- `npm run build:smoke`
- targeted tests for Partner file proposal validation, diff review, apply/reject, and blocked raw write/edit/bash behavior
