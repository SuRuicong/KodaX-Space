## Batch Intent

KodaX-Space already has rich runtime signals, but they are scattered across the bottom live strips, the right sidebar, popouts, and diagnostics. v0.1.21 turns those signals into a single pinned summary layer:

**Idle = environment radar. Running = live runtime summary. Expanded = full detail panel.**

This borrows the useful part of Codex's pinned summary panel without copying its exact information model. KodaX remains conversation-first; the summary is a quiet, glanceable status surface.

## FEATURE_103: Pinned Runtime Summary and Live Detail Host

### Problem

The current live/status model has three issues:

- Runtime information is split between `BottomBar` live strips, `RightSidebar` sections, popouts, and diagnostics.
- When no task is running, the right side does not provide a compact "environment is ready / degraded / dirty" readout.
- When a task is running, users need a stable one-glance answer to: "What is happening? Do I need to intervene? Where do I open details?"

The existing implementation already has the data. The missing feature is a projection layer and a consistent place to render it.

### Design

Add a pinned runtime summary with three layers.

#### L0: Pinned Summary

Always available in the Coder surface.

When the right sidebar is open, it appears as the sticky top section of `RightSidebar`.

When the right sidebar is closed, it should degrade to a compact right-edge or header-adjacent summary affordance, not a large floating dashboard.

Idle mode shows environment information:

- current project / session,
- provider, model, effort, permission mode, agent mode,
- runtime health summary,
- Repointel / MCP summary when available,
- git dirty summary when already available from a shared snapshot.

Running mode shows live information:

- current state: sending, thinking, tool, workflow, worker, waiting, retrying, error,
- the current focused activity,
- one or two metrics such as workers, todo progress, token budget, changed files,
- a clear click target for details.

#### L1: Right Sidebar Overview

The existing right sidebar stays the main detail surface. Add the summary as a sticky top section above:

1. Pinned Runtime Summary
2. Plan
3. Workflow
4. Workers
5. Changes
6. Working Folder
7. Context

The summary should not duplicate full section content. It should expose only the strongest signal and route the user to the right section or full view.

#### L2: Expanded Detail

Clicking the summary opens the right sidebar if needed and brings the detail surface into focus.

MVP should prefer the existing right-sidebar width mechanism over a new popout kind:

- closed sidebar -> open sidebar,
- default sidebar -> optionally switch to wide width,
- scroll / focus summary at the top,
- do not auto-open `TasksPanel`, `PlanPanel`, or `WorkflowPanel`.

If a later implementation needs a dedicated `session-info` popout, it should reuse the same projection model and keep the right-sidebar path primary.

### Projection Model

Create a renderer-only projection layer. It must not introduce a second runtime truth.

Recommended module:

```text
apps/desktop/renderer/src/shell/sessionLiveProjection.ts
```

Recommended view model:

```ts
export interface PinnedRuntimeSummaryViewModel {
  mode: 'no_project' | 'idle' | 'running' | 'attention' | 'error' | 'completed';
  severity: 'neutral' | 'info' | 'running' | 'warning' | 'danger' | 'success';
  headline: string;
  detail?: string;
  rows: readonly PinnedRuntimeSummaryRow[];
  metrics: readonly PinnedRuntimeMetric[];
  primaryTarget?: 'environment' | 'plan' | 'workflow' | 'tasks' | 'changes' | 'context';
}
```

Inputs should be existing renderer state only:

- `currentProjectPath`
- `currentSessionId`
- `sessions`
- `eventsBySession`
- `pendingSendBySession`
- `todoListBySession`
- `managedTaskStatusBySession`
- `workBudgetBySession`
- `workflowRuns`
- `tokensBySession`
- provider/model/mode pending and session metadata

Do not read API keys, raw env values, private SDK storage, or shell environment variables.

### Event Priority

The summary priority is display-only and separate from Smart Popout Director.

Priority order:

1. User action required: permission, ask-user, budget approval.
2. Error / recovery: `session_error`, retry, provider recovery.
3. Workflow live: running, paused, replaying, future draft/review states.
4. Managed task active worker / verifier.
5. Current tool activity, especially file mutation tools.
6. Context compaction / token pressure.
7. Thinking / writing / sending.
8. Plan / todo update / todo drift warning.
9. Idle environment summary.

### UX Rules

- The summary should never steal focus.
- The summary should not auto-open full popouts.
- Running status may open the right sidebar only when existing sidebar auto-open behavior already applies; it should not add a new interruption rule.
- Completed state can linger briefly, then fall back to idle environment.
- Error and attention states should persist until the next user action or dismissal.
- Partner surface does not get this in v0.1.21 unless Partner gains its own right-side information model.

### Non-Goals

- Do not build a dashboard card wall.
- Do not duplicate `TasksPanel`, `PlanPanel`, `WorkflowPanel`, Diff, Artifact, or Terminal.
- Do not add a settings page for summary customization in MVP.
- Do not run new git IPC from the summary until git snapshot logic is shared.
- Do not expose API key env names, raw env values, or private runtime internals.

### Implementation Plan

1. Extract shared live projection from existing live/status logic.
   - Reuse the semantics currently spread through `ActivitySpinner`, `WorkflowWorkStrip`, `AmaWorkStrip`, and `BackgroundTaskBar`.
   - Add pure-function tests for terminal, running, waiting, error, and idle states.

2. Add `PinnedRuntimeSummary`.
   - Render compact headline, detail, and metrics.
   - Add stable dimensions so frequent event updates do not resize the layout.
   - Use icons and color sparingly; severity should help scanning without shouting.

3. Integrate into `RightSidebar`.
   - Add sticky summary at the top of overview mode.
   - Clicking opens the sidebar if closed and uses existing width controls for "fuller" view.
   - Keep Artifacts tab behavior unchanged.

4. Consolidate bottom live strips.
   - Prefer the new summary as the main live readout.
   - Leave bottom strip content only where it materially helps while typing, or remove duplicated text once parity is verified.

5. Add regression tests.
   - Pure projection tests.
   - Renderer/component tests for idle and running summary states.
   - Existing sidebar resize and popout behavior must keep passing.

### Acceptance

- With no project open, the summary shows an actionable no-project state.
- With a project open and no task running, the summary shows environment readiness without noisy runtime internals.
- During a send, the summary moves through sending -> running/tool/workflow/worker states using one shared projection.
- If permission, ask-user, budget approval, or session error occurs, the summary prioritizes it.
- Clicking the summary opens or expands the right-side information surface.
- The right sidebar still exposes Plan, Workflow, Workers, Changes, Working Folder, and Context in full.
- No new IPC or SDK dependency is required for MVP.
