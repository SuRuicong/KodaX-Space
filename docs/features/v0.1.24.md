# v0.1.24 - Customer Timebox Entitlement MVP

> Design batch: F105
> Status: Planned
> Source of truth: this file + [../FEATURE_LIST.md](../FEATURE_LIST.md)

---

## Intent

`v0.1.24` exists to give customer-facing builds a practical time limit before the broader enterprise licensing and distribution strategy is complete.

This is deliberately smaller than the full legal/commercial program:

- ship an offline-verifiable entitlement mechanism,
- allow customer trial builds to expire after 30 days,
- avoid any license prompt for normal community, education, research, and personal use,
- keep the codebase single-track,
- document that old Apache-2.0 releases remain under Apache-2.0 and cannot be retroactively restricted.

## FEATURE_105: KodaX-AI Fair Core License and Offline Enterprise Entitlement

### Problem

KodaX Space needs to be shared with early enterprise or institutional customers before the full v0.2.x distribution track. Some of those customers are intranet or air-gapped, so online activation is not acceptable. At the same time, community and education users should not be interrupted by a license wizard or forced trial countdown.

Apache-2.0 remains valid for versions already released under Apache-2.0. For future customer-facing packages, KodaX-AI needs a clear path toward commercial rights while the immediate implementation gives customer builds a 30-day timebox.

### Scope Split

`v0.1.24` ships the **customer timebox entitlement MVP**:

- offline signed `.kodax-license` import,
- 30-day trial entitlement for customer builds,
- managed-required mode for customer packages or enterprise policy files,
- license status in Settings/About/Diagnostics,
- expiry handling that blocks customer/enterprise-only execution without deleting user data,
- no startup prompt and no countdown for community/permitted free use.

Full legal and distribution follow-ups remain outside the MVP:

- final legal text for **KodaX-AI Fair Core License** (`KAI-FCL`),
- public license switch announcement,
- full source-available license file replacement,
- enterprise purchasing docs and contract templates,
- future SSO/SAML/SCIM or hosted admin surfaces.

### License Policy Direction

Use the working public name **KodaX-AI Fair Core License** (`KAI-FCL`) for planning. Legal review may rename the final artifact, but the license family should start with `KodaX-AI` so it can apply across KodaX-AI products rather than only KodaX Space.

The intended policy shape is:

- personal use: free and no-nag,
- education/research/open-source/community use: free and no-nag,
- enterprise evaluation: signed 30-day trial entitlement,
- enterprise production, intranet deployment, managed deployment, commercial redistribution, or paid-service use: commercial license required,
- bypassing or disabling license enforcement for commercial/customer builds: prohibited by the commercial terms.


### Issuer Tooling and Repository Boundary

Yes, F105 needs tooling to produce customer tokens. In product language these should be called **license entitlements** or **license tokens**, not generic access tokens, because they grant offline product rights rather than API access.

Split the tooling into public and private parts:

- **Public in this repo**: entitlement schema, verifier, import/export request code, public-key bundle, fixtures, and tests with non-production test keys.
- **Optional public/dev tool**: a local test issuer that uses a checked-in throwaway dev key and can only create `edition: "dev"` or `edition: "test"` entitlements that production builds reject.
- **Private KodaX-AI repo or secure operator machine**: production issuer CLI/service, private signing keys, customer registry, license id generator, renewal workflow, and audit log.
- **CI/secrets only**: any automated customer-package signing path must read private keys from secret storage; private keys must never be committed, packed, logged, or shipped.

Suggested private issuer command shape:

```powershell
kodax-ai-license issue `
  --product kodax-space `
  --customer "Example Corp" `
  --edition trial `
  --days 30 `
  --features customer-trial,offline `
  --binding-request .\example-corp.request.json `
  --out .\ExampleCorp.kodax-license
```

The open-source/package side only needs to verify the resulting `.kodax-license` using embedded public keys. Production token issuance is a business system and should stay outside the public KodaX-Space repository.
### Entitlement Model

Use asymmetric signing:

- KodaX-AI keeps private signing keys outside the client and outside the public repository,
- KodaX Space embeds only public verification keys,
- entitlement payloads carry a `kid` for key rotation,
- signatures should use Ed25519 or another modern asymmetric signature supported by the packaged runtime,
- verification must work fully offline.

Example entitlement payload:

```json
{
  "product": "kodax-space",
  "licenseFamily": "KodaX-AI Fair Core License",
  "licenseId": "lic_xxx",
  "customer": "Example Corp",
  "edition": "trial",
  "issuedAt": "2026-06-22T00:00:00Z",
  "expiresAt": "2026-07-22T00:00:00Z",
  "features": ["customer-trial", "offline"],
  "binding": {
    "mode": "site-or-machine",
    "siteId": "example-corp-intranet",
    "deviceIds": ["sha256:..."]
  }
}
```

Supported MVP modes:

- **Community mode**: no entitlement required, no startup prompt, no trial countdown.
- **Customer trial mode**: signed 30-day entitlement, fully offline.
- **Customer licensed mode**: signed entitlement with expiry and feature claims.
- **Managed required mode**: customer installer, launch flag, or policy file sets `license.required=true`; the app opens only the license import/status screen until a valid entitlement is present.

### Runtime UX

Community use should feel unchanged. The only visible surface is a quiet Settings/About line such as `Community use` with a link to terms.

Customer/managed UX:

- Settings > License shows status, customer, edition, expiry, and enabled features.
- Users can import `.kodax-license` by file picker or drag/drop.
- Users can export a request file containing app version, platform, machine id, and requested entitlement mode.
- Expiry blocks new customer-gated runs/actions but keeps project data readable.
- Invalid signatures, wrong product, expired entitlement, and binding mismatch show clear non-secret errors.

### Gate API

F105 should expose a small internal primitive:

- `license.getStatus()` returns `community`, `trial`, `licensed`, `expired`, or `invalid`,
- `license.importEntitlement(file)` stores and verifies a signed entitlement,
- `license.exportRequest()` produces an offline request file,
- `license.requireEntitlement(reason)` enforces managed-required/customer-gated behavior,
- `license.hasFeature(featureId)` supports later F098/F101 enterprise gates.

The first implementation should gate only customer-package behavior and explicitly customer/enterprise-labeled surfaces. Personal provider settings, local sessions, and community workflows remain available outside managed-required mode.

### Storage and Tamper Boundary

Store the imported entitlement in app data and mirror it to OS keychain when available. The entitlement is not a secret; authenticity comes from the signature. Do not store private signing keys in the repo, package, or user machine.

Offline enforcement is not perfect DRM. It should catch invalid signatures, expiry, product mismatch, binding mismatch, and obvious clock rollback signals. The business goal is enforceable commercial terms plus reasonable friction for customer builds, not an impossible-to-patch local client.

### Acceptance

- Clean community install starts without license prompt or countdown.
- Managed-required/customer build blocks normal app use until a valid entitlement is imported.
- A signed 30-day customer trial entitlement works offline.
- Expired entitlement prevents customer-gated execution while preserving data access.
- Invalid signature, wrong product, expired entitlement, and binding mismatch are rejected.
- Settings/About/Diagnostics show enough license status for customer support.
- Signing private keys are never packaged into the app or committed to the public repository.
- Existing Apache-2.0 releases are documented as not retroactively revoked.

### Non-goals

- No online activation server dependency.
- No full public license switch in this patch.
- No separate Community/Enterprise source tree.
- No broad enterprise admin platform.
- No SSO/SAML/SCIM.
- No license nagging for permitted community/education/research/personal use.
