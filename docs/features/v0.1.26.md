## Feature_108 Composer Dropped File References

### 1. Goal

Let users drag files into the composer and see durable, sendable references directly in the input draft. This is the Space-owned UI and shell integration layer for file references; it does not try to define the full KodaX SDK file/video artifact contract.

This closes the immediate desktop gap after OC-31 image paste: users can now drop local files, directories, and supported image files onto the input surface instead of manually typing paths.

### 2. Scope

Completed Space-owned behavior:

- Resolve a user-dropped browser `File` to an OS path through Electron `webUtils.getPathForFile` exposed as a narrow preload bridge.
- Insert a safe project-relative reference as `@relative/path` when the file is inside the current project and the path can be represented safely.
- Insert an external markdown file link as `[name](<file://...>)` when the file is outside the current project or needs quoting/escaping.
- Show dropped file chips above the composer with remove actions that also remove the inserted reference from the draft.
- Cap pending file references to 32 per draft.
- For dropped PNG/JPEG/WEBP files, reuse the existing OC-31 image path: persist the image through `clipboard.saveImage`, show an image chip, and send it as a `kind: 'image'` input artifact.

### 3. Non-Goals and Boundaries

This feature deliberately does not claim the whole multimodal roadmap:

- Native clipboard image fallback is implemented for sandboxed persistence and renderer attachment, but still intentionally avoids broad non-image `Files` paste interception.
- GIF animation preservation is not guaranteed. KodaX 0.7.56 includes `image/gif` in its media contract, but Space's current `clipboard.saveImage` sandbox persists only PNG/JPEG/WEBP.
- Video is not a structured Space artifact yet. KodaX 0.7.56 can report provider video capability state, but dropped videos are still referenced by path/link only until Space defines capture, persistence, preview, and provider fallback behavior.
- Ordinary non-image files are not sent as structured `kind: 'file'` artifacts. KodaX 0.7.56 still reports file artifacts as unsupported for this contract; they remain prompt-visible references that the agent can reason about or open through existing tools when appropriate.
- This feature does not add a file picker or attach menu redesign.

### 4. Design Notes

The preload bridge is intentionally narrow:

```ts
getPathForFile(file: File): string | null
```

Renderer code never receives Electron primitives. The only supported path source is a real user-provided drag/drop `File` handle, which keeps this distinct from arbitrary renderer-originated path injection.

Reference formatting follows two rules:

1. Project-local safe paths prefer `@relative/path`, matching existing KodaX/Space path reference semantics.
2. External or hard-to-tokenize paths become encoded `file://` markdown links so spaces, Windows drive letters, UNC paths, and special characters survive insertion.

Images use the stricter existing artifact path:

- accepted types: `image/png`, `image/jpeg`, `image/webp`,
- max size: existing OC-31 image paste cap,
- persisted in the Space clipboard sandbox,
- sent through `session.send({ artifacts: [{ kind: 'image', path, mediaType }] })`.

### 5. Acceptance

- Dropping a project file inserts an `@relative/path` reference into the composer.
- Dropping an external file inserts a clickable markdown `file://` reference into the composer.
- Dropped references appear as removable chips and removing a chip removes the corresponding inserted reference.
- Dropping more than 32 files caps the draft and reports a local composer warning.
- Dropping PNG/JPEG/WEBP files both inserts file references and attaches image artifacts for the next send.
- Sending a draft with only dropped file references is allowed.
- Unsupported media types do not crash the composer; they remain path references unless a later structured artifact contract supports them.

## 9. Runtime Dependency and Linux Compatibility Decision (Electron 42.5.0)

During release prep the working tree briefly downgraded Electron `^42.5.0` → `33.2.0` (with the esbuild target lowered to `node20`). The release review reverted this to **`electron@42.5.0` (exact)** and restored the `node24` esbuild target. Rationale:

- **Security:** `npm audit` flags Electron `<=39.8.4` as 18 HIGH-severity Chromium/Electron advisories (per-advisory fix floor is `39.8.5`). Shipping `33.2.0` would have been a security regression versus the already-public `v0.1.25` (which shipped on `42.5.0`). `42.5.0` reports `0 vulnerabilities`.
- **Linux / Kylin V10 compatibility is unaffected by the upgrade.** Some customers run **麒麟/Kylin V10**, whose base/server editions ship **glibc 2.28** (Debian 10-class; some Desktop SP1 builds are Debian 11 / glibc 2.31). Electron's official platform-support matrix lists the **same** Linux runtime floor — *"verified on Ubuntu 18.04+, Fedora 32+, Debian 10+"* — for **33, 40, and 42** alike. Chromium builds against a pinned older sysroot, so moving the *build host* from Ubuntu 20.04 → 22.04 (33 → 42) does **not** raise the runtime glibc floor. (The "Chromium 114 needs newer glibc" reports concern *source rebuilds* such as ungoogled-chromium, not the official prebuilt binaries Electron ships.) Net: `42.5.0` keeps the glibc 2.28 / Debian 10 floor, so Kylin V10 stays supported.
  - Operational note (independent of the 33↔42 choice): Electron on 国产化 / hardened Linux sometimes needs `--no-sandbox` or hits seccomp quirks; behavior is the same on 33 and 42.
- **Version range, for reference:** any Electron `>= 39.8.5` clears all 18 advisories; `40.10.5` is the lowest *currently-supported* major that is fully patched; `42.5.0` is chosen because it is the supported `latest` and the version `v0.1.25` already shipped and validated end-to-end through the release pipeline.
- **Native modules:** `@napi-rs/keyring` is N-API (ABI-stable across Node/Electron); `better-sqlite3@12.11.1` is rebuilt for the Electron 42 ABI at pack time via `scripts/ensure-sqlite-native.mjs` / `electron-builder install-app-deps`. No native-ABI regression results from the upgrade.
